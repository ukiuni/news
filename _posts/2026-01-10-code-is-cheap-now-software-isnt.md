# Code Is Cheap Now. Software Isn’t. - コードは安くなった。ソフトウェアはまだ高い
個人化された「使い捨てソフト」の時代が来た — エンジニアはまだ必要だが役割が変わる

## 要約
生成系AIで「コードを作るコスト」は劇的に下がったが、本当に価値あるソフトウェアを長く動かすコスト（保守、堅牢性、配布）は依然として高い。重要なのは「何を作るか／誰に届けるか／どう維持するか」。

## この記事を読むべき理由
AIツールで簡単にプロダクトを作れる今、誤った期待でリソースを浪費しがちです。とくに日本の開発現場では、社内業務改善や地方・業界特化のニッチサービスで「作る」価値が大きく変わります。本質を押さえれば短時間で効果的なツールを作りつつ、破綻を防げます。

## 詳細解説
- バリアの崩壊：Claude Codeや類似のLLMツールにより、個人や非開発者が短時間でプロトタイプや実用ツールを作れるようになりました。CLIやローカル中心のワークフローが注目され、複雑なUIやオンボーディングを不要にすることで「その場限り」のツールが生まれやすくなっています。
- 「SaaSからスクラッチパッドへ」：従来のSaaSは長期的な継続・拡張を前提に最適化されますが、今は一時的な用途を解くための“スクラッチパッド的ソフト”が増えています。スプレッドシートで一時解析する感覚に近い使い捨てユーティリティです。
- コードは安いが、ソフトは高い：LLMはコード生成を得意とする一方で、実運用で起きるエッジケース、データ互換、同期・オフライン要件、セキュリティ、運用コストなどの理解・実装は依然として人間の工学力が要ります。デモは良くても現場で壊れるケースが多いのはそのためです。
- 配布と差別化の重要性：今や「コードを書く力」は差別化要因でなくなりつつあります。成功するプロダクトは、配布（ユーザーへの届き方）、タイミング、ユーザー理解、そして味付け（UX・業務理解）で勝ちます。日本市場では業界慣習や規制、言語対応が重要なモート（参入障壁）になります。
- エンジニアの役割変化：低レイヤの実装よりも、アーキテクチャ設計、データ所有権、耐障害設計、運用設計、そしてビジネス理解に重きが移ります。LLM出力は「味見」に使い、必ず人のレビューと検証を入れるべきです。

## 実践ポイント
- 小さく作る：5分〜数時間でプロトを立て、まず現場の一人に使わせる。「動くこと」を優先して早めにフィードバックを得る。
- CLI／ローカルファーストを検討：オンボーディングの摩擦を減らせば使われやすい。社内ツールやパワーユーザー向けに有効。
- 維持コストを前提に設計：外部CSVの変更やDOM変化に弱い構造は避ける。テスト、監視、フォールバック戦略を最初に考える。
- LLMはアシスタント、レビュアーは人間：生成コードは必ず「PRレビュー」する運用をルール化する。誤作動や安全性の確認を怠らない。
- 配布戦略を先に決める：何人に届けば成立するか、既存顧客や社内ネットワークをどう活用するかを設計段階で決める。
- ドメイン専門家と組む：業務特化ツールはドメイン知識が勝敗を分ける。現場の知見を取り込め。
- ドキュメントとナレッジ生成にLLMを活用：コードのウォークスルーやオンボーディング資料はAIで高速生成→人が校正すると効率的。

短く言えば、「コードが安くなった今、勝負はアイデアの正しさと運用・配布・保守の設計に移った」。日本の現場では、業務知見と配布チャネルが依然として最も価値ある武器になります。
