# .plan files (2020) - .plan ファイル
毎日の「つぶやき」を仕事の資産に変える──今こそ復権させたいシンプルなエンジニア習慣

## 要約
プレーンテキストの個人ログ「.plan」を日常ワークフローに取り入れることで、タスク管理・ナレッジ蓄積・技術文章力が同時に向上する、という提案。

## この記事を読むべき理由
煩雑なツールに疲れた日本のエンジニア／チームに向け、導入が容易で永続的価値を生む軽量なドキュメント習慣を紹介するため。

## 詳細解説
- コンセプト：.plan はプレーンテキストで公開可能な日次ログ兼メモ帳。構文ルールは最小限で、Markdown 風の見出し（＝日付）と簡単なプレフィックスで読み書きする。  
- 書き方の核：日付ごとのセクション（例：# 2020-05-16）に、完了は `*`、未解決タスクは `?`、後で直したものに `+`、興味喪失は `~` といった目印を付ける。区切りに `---` を使って複数サブエントリを持てる。  
- 運用例：個人用、業務用、チームメンバー別（1:1 の議事録や成果管理）など複数ファイルで運用。Dropbox や Git で同期し、どこでも編集可能にする。公開しておくと外部からの発見やフィードバックも得られる。  
- ツール連携：Vim のシンタックス／フォールディングや autocmd マッピングで編集効率を上げる小技、cron で自動取得して RSS を作るなど自動公開の仕組み化も可能。  
- 効果：デバッグの経緯や小さなアイデアを都度残すことで「同じ失敗の再発防止」「技術文章力の継続的向上」「チーム内の知識共有」が期待できる。

例（書式イメージ）:
```text
# 2020-05-16
? Travis-CI deploy error when uploading to GitHub
* Fixed Windows stdout issue for CL executables (console vs GUI)
---
+ Resolved CI quoting bug after escaping backticks
```

## 実践ポイント
- 今日から1行でも書く：まずは日付ヘッダと1エントリから始める。  
- シンプルなルールを決める：完了、未解決、興味喪失のプレフィックスを統一する。  
- 同期と公開：Dropbox または GitHub で同期／公開してモバイルからも編集できるようにする。  
- チーム適用：1:1 用の .plan をチームメンバー毎に持たせると振り返りが楽になる。  
- 自動化：編集支援（テンプレート、エディタマッピング）と公開（cron → RSS/GitHub Pages）を少しずつ整備する。

まずは1週間続けてみて、日々の小さな記録がどれだけ後で役に立つかを体感してほしい。Happy .plan -ing
