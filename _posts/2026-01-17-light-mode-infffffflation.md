# Light Mode InFFFFFFlation - ライトモードの白化インフレ
白は“クリーン”なのか？MacのUIが年々どんどん明るくなっている「白化インフレ」を可視化してみた

## 要約
海外記事は、過去16年間のmacOSスクリーンショットを分析して「ライトモードの背景が年々明るく（白く）なっている」ことを示します。作者はPillowで平均輝度を測り、macOSの既定ウィンドウがついに100%明度に達したと報告しています。

## この記事を読むべき理由
日本でもiPhone/iPadやMacを長時間使う開発者やデザイナーは多く、ディスプレイの明度変化は視認性・目の疲れ・デザイン判断に直結します。HDR対応端末の普及も進む日本市場で、「何を白とするか」が今後のUI設計で重要になります。

## 詳細解説
- データと手法  
  作者は512 PixelsのmacOSスクリーンショットライブラリから同等のスクリーンショットを取得し、ウィンドウの代表領域を切り出して解析しています。画像をグレースケールに変換し、PillowのImageStatで平均明度を計算するシンプルな方法を採用しました。例：
  
  ```python
  from PIL import Image, ImageStat
  import os

  for file in sorted(os.listdir('.')):
      image = Image.open(file)
      greyscale = image.convert('L')
      stat = ImageStat.Stat(greyscale)
      avg_lightness = int(stat.mean[0])
      print(f"{file}\t{avg_lightness}")
  ```
- 主な発見  
  年ごとのグラフを見ると、デフォルト（ライト）モードのウィンドウ明度はおおむね上昇傾向にあり、Snow Leopard（2012頃）で約71%だったものが、Tahoeでは100%に到達しています。ダークモード側も明度は上昇傾向だが、ライトモードほどではない、という二本のラインが見られます。
- 制約と解釈の注意点  
  - 単純な平均輝度は「人間の知覚する明るさ」や色味（壁紙によるティント）を反映しません。より正確にはsRGBの線形化や相対輝度（luminance）を使うべきです。例えば相対輝度は次のように表されます： $$Y = 0.2126R + 0.7152G + 0.0722B$$ ただしR,G,Bは線形化された値です。  
  - サンプルはウィンドウクローム（枠周り）中心で、アプリ全体の表示明度とは差があります。非アクティブウィンドウの色や背景全体を見ると別の結果になる可能性があります。
- UIトレンドの影響  
  ブライトなデザインは「清潔・洗練された」印象を与えるため、デザイナーは明度を上げがちです。さらにHDR対応で一部ハイライトが白を超える表現が可能になると、インフレは加速する恐れがあります。作者はBig Surの明度上昇をきっかけにシステムをダークモードに移行したと述べています。

## 実践ポイント
- 50%グレーを検討する：純白（100%）は眩しく感じやすい。UI背景に中間灰（約50%グレー）を使うと目の負担を減らしつつ適切なコントラストを保てます。  
- HDRとSDRの差をテストする：HDR端末でUIがどのように“過度に明るく”見えるか、実機で比較検証してください。  
- 人間の視覚に近い指標で測る：平均輝度だけでなく相対輝度（Y）やWCAGのコントラスト比で評価しましょう。  
- ウィンドウ間の識別性を維持する：ダークモード採用時でもウィンドウごとの微妙なトーン差や境界を設け、どのウィンドウがアクティブか分かる工夫を。  
- 既存のスクリーンショットを使った簡易調査の方法：公開スクリーンショットコレクションを利用して、バージョン別のUI明度を比較することは少ない労力で有益なインサイトを生みます。

この記事を読んで、あなたのプロダクトや日常の環境で「本当に必要な明るさ」は何かを再検討してみてください。私たちの画面は、知らず知らずのうちに“白化”しているかもしれません。
