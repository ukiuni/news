# MCP is a fad - MCPは一過性の流行
魅力的なタイトル: 「MCPブームの正体──“簡単さ”が招いた落とし穴と日本の現場で取るべき選択」

## 要約
MCP（Machine-Connector-Protocol）はツール共有を簡単にする一方で、プロセス分離・運用不透明性・サプライチェーンや権限まわりの重大なリスクを伴い、実運用では代替手段のほうが有利なことが多い、という警鐘です。

## この記事を読むべき理由
日本でも企業のAI導入やOSS連携が増える中、手軽に「AI対応」を謳う技術が出てきています。MCPは導入の敷居が低い反面、運用・セキュリティで痛い目を見る可能性が高く、日本企業の現場判断に直接関係する話題です。

## 詳細解説
- MCPとは何か（初心者向け）
  - 「エージェント（AI）と外部ツールをつなぐ共通プロトコル」。異なるAIプラットフォームやツール群を「MCPサーバー」というデーモンとして公開し、エージェントはそれを叩いて機能を使うイメージ。
  - 背景にある課題は「NxM問題」──複数モデル（N）と複数ツール（M）の組み合わせで個別コネクタが増える点。MCPはその手間を減らすことを目指す。

- MCPが便利に見える理由
  - サーバーとして配置するだけでツールを公開でき、ツール作者にとっては「配布の簡便さ＝宣伝効果」がある。Anthropic含む一部エコシステムの後押しもあり普及が早かった。

- 問題点（運用・セキュリティ・品質）
  1. プロセス分離による管理不透明性：ツールが別プロセスで動くため、リソース消費やログ・エラー管理が分散し、障害対応が複雑化する。
  2. ツール群の非一貫性：各ツールが独立しているため、エージェントは「どのツールが適切か」を判断しづらく、性能低下やトークン浪費につながる。
  3. 環境依存と起動失敗：npm/vpip/環境変数など環境の違いで起動トラブルが発生しやすい。
  4. 供給側リスク（サプライチェーン）：誰でもサーバーを公開でき、署名や中央管理が無いままローカルでデーモンを走らせる危険がある。実際に複数の脆弱性やデータ流出事例が報告されている。
  5. セキュリティモデルの錯誤：MCPで「エージェントにAPIキーを渡さない」設計でも、エージェントが危険なツール呼び出しを行えば権限濫用は可能。信頼先が増えるだけになる場合がある。

- なぜ代替が強いのか
  - 多くのユースケースでは、単純なスクリプトやローカルコマンド、OpenAPI / SDK（LangChain 等）が十分で、プロセス分離のコストを払う価値がない。
  - 内製アプリならツールを同一コードベースに置けば監査/認可/ロギングを統一できる。

## 実践ポイント
- まずはシンプルな選択を：社内用途なら「1st partyツール（同一コードベース）」やOpenAPI/SDKで始める。
- ツール数は絞る：OpenAI推奨の目安（20未満）を参考に、ツールは整理して一貫性を保つ。
- セキュリティ基本を忘れずに：
  - スコープを狭めたトークン、最小権限、監査ログを必須にする。
  - 外部MCPサーバーを使う場合は署名・ソース検証・静的解析を必ず行う。
- ローカルスクリプト＋コマンドランナー（just/makeなど）を試す：Agentが呼べる短いスクリプトで済むなら運用が遥かに楽。
- 開発者向け：エージェントに渡すツールの説明は最小限にして、冗長な説明でトークンを浪費しない工夫を。
- 日本企業への注意点：パッケージ管理や社内ネットワークの制約（プロキシ、社内認証）でMCPの起動が難しいケースが多い。導入前にPoCで環境検証を行うこと。

MCPは「簡単に見える」分だけ落とし穴も大きい道具です。まずは既存の堅牢なやり方（OpenAPI / SDK /社内関数）で安全に価値を出すことをおすすめします。
