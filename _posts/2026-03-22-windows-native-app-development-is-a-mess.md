# Windows Native App Development Is a Mess - Windows ネイティブアプリ開発はカオスだ
魅せるはずが泥沼に：Windowsネイティブ開発の“今”をひと言で知る

## 要約
元記事は、作者が小さなユーティリティを作ろうとして遭遇した、断片化されたWindowsネイティブ開発の現実（APIの分岐、互換性問題、配布コストなど）を詳しく解説しています。

## この記事を読むべき理由
日本でも企業・個人問わずWindowsが主戦場の場面は多く、ネイティブで「ちょっと便利なツール」を作る際に直面する現実がそのまま当てはまります。時間・工数・配布・保守の判断が変わるため、知っておく価値があります。

## 詳細解説
- 歴史的経緯：Win32 → MFC → WinForms → WPF → WinRT → UWP → WinUI 3 と UI/API が何度も作り直され、層が増えた結果、互換性と一貫性が失われた。  
- フレームワークの選択難：C++（小さく速いがメモリ安全性の問題）、C#の「フレームワーク依存配布」（古い .NET がプリインストールされており最新版配布が面倒）、あるいは.NET AOT（ランタイム込みでバイナリ肥大化）のトレードオフ。  
- 必須機能のギャップ：ディスプレイ列挙や非アクティブウィンドウ、グローバルホットキー、トレイアイコンなど、WinUI/Windows App SDK だけでは足りず、古いWin32 APIへのP/Invokeが頻発。  
- 相互運用ツールの未成熟：CsWin32 等のラッパーは便利だが未完成で文字列/構造体周りなどで問題あり。言語（C#）自体に向上が必要な領域も残る。  
- 配布と署名の壁：MSIXやストアに頼ると署名コストやレビューポリシーで高いハードルが発生。サイドロードは管理者権限や複雑手順が必要で、インディー開発者には重い。  
- 結果として：多くの開発者がElectronなどWeb技術で回避し、Microsoft自身の第一線アプリにもWeb技術採用が見られる。

## 実践ポイント
- 小さなユーティリティなら：AutoHotkeyや既成のElectronテンプレを検討して工数を減らす。  
- ネイティブで行くなら：最初に「必要な低レベル機能」を明確化し、P/Invokeが避けられないかを確認する。  
- 配布戦略：署名コストやMSIX要件を事前に調査し、ターゲット環境（企業管理PCか個人PCか）に合わせてEXE/インストーラとMSIXを使い分ける。  
- ライブラリ選定：CsWin32などは“便利だがベータ”と割り切り、テストとフォールバック（手書きP/Invoke）を準備する。  
- 日本市場留意点：社内ポリシーで外部インストール/署名が厳しいケースが多いので、配布と運用面（IT管理者への説明）を早めに詰める。  

（参考）簡単な代替案：UIの凝った新規機能が不要なら既存のAutoHotkeyスクリプトやストアの軽量アプリを活用するのが現実的です。
