# What’s a bug you couldn’t fix for days that turned out to be embarrassingly simple? - 何日も直せなかったバグが実はとても単純だった話
思わず「あるある！」と声が出る、現場で学ぶデバッグのコツ

## 要約
長期間ハマるバグの多くは、環境差・設定ミス・単純なタイプミスなど“見落としやすい小さな原因”で起きる。再現手順の切り出しと「最小構成」での検証が早期解決の鍵。

## この記事を読むべき理由
日本の企業でもレガシー環境や複数プラットフォーム、言語混在チームが増え、単純ミスが大きな障害に発展しやすい。初心者も含め誰でも実践できるデバッグ習慣はコストと時間を劇的に節約するため必読。

## 詳細解説
よくある「数日ハマるが実は単純」パターン：
- 環境差：開発はWindows、CIはLinuxでケース感度（大文字小文字）やパス区切りが原因。
- 設定/環境変数：APIキー、DB接続文字列、タイムゾーンやロケールの違いで動作が変わる。
- キャッシュ/ビルドアーティファクト：古いバイナリやキャッシュが残っていて意図しない挙動に。
- タイポ/インデックスずれ：変数名のタイプミス、オフバイワンによる配列アクセス違反。
- 権限/ファイルモード：実行権限やファイル所有者が原因の読み書き失敗。
- 依存関係のバージョン差：マイナーバージョンでAPIが変わることがある。

デバッグ手法（実務で効く順）：
1. 最小再現ケースを作る — 複雑なプロジェクトから問題箇所だけ切り出す。
2. ログと差分確認 — 直近のコミットや設定変更を git diff / CI 設定で確認。
3. 環境の再現性確保 — DockerやVMでローカルと本番を一致させて検証。
4. バイナリ/キャッシュのクリア — clean build、node_modulesやvenvの再作成。
5. 二分探索（git bisect）で壊れたコミットを特定。
6. ペアレビュー／ラバーダッキング — 第三者に説明すると気づくことが多い。

## 実践ポイント
- まず「本当に同じ手順で再現するか」を確認する（手順を書き残す）。
- 環境を疑う：OS、シェル、エンコード、Locale、依存ライブラリのバージョンを列挙して比較する。
- CIが通っているか、本番環境での差分を最優先でチェックする。
- キャッシュ消去・クリーンビルドを試す（例：rm -rf node_modules && npm ci）。
- 小さく切り分けて試し、見つかったらチケットに「再現手順＋直し方」を必ず残す。

短い時間で試せるチェックリストを日常化すると、「数日ハマる」を防げます。
