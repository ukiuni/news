# A new worst coder has entered the chat: vibe coding without code knowledge - 新たな“最悪のコーダー”現る：コード知識ゼロの「バイブコーディング」
非エンジニアでも「アプリが作れる」魔法か、それともイナバ物置みたいな負債か？

## 要約
AIを使った「vibe coding」（自然文で指示してアプリを生成する手法）は、非開発者でも短時間で動くアプリを作れる一方で、セキュリティ・保守性・品質に深刻な穴が開きがちだという実例報告。

## この記事を読むべき理由
日本でもローコード／ノーコードやAI支援ツールの導入が進んでおり、非エンジニアの“即席開発”が増える可能性が高い。現場で何が起きるかを知っておかないと、後で大きな手直しコストや法規的リスク（個人情報保護等）を招く。

## 詳細解説
- vibe codingとは  
  自然言語で「こういうアプリを作って」と指示すると、ツール（例：Bolt）がコード・フォルダ構成・UIまで自動生成し、ライブプレビューで動作を確認できる流れ。開発経験がほとんどない人でも、短時間で「見た目上」動くものを作れるのが利点。

- 実際の体験で起きたこと（要点）  
  - 簡単にアプリの骨子が出来上がり、UI操作でテスト可能。ただし内部はブラックボックス化しがち。  
  - エラーが出たときはツールにエラーメッセージを投げて直してもらえるが、非エンジニアは「何が直ったか」を理解できない。  
  - 生成物のコードは整理されておらず、スタイルがコンポーネントにインラインで埋め込まれる、ファイル配置が不自然、READMEだけで文脈が足りないなど保守性に問題。  
  - セキュリティが無防備なまま公開される危険がある（APIエンドポイントの露出、認証・権限制御の欠如）。  
  - 生成後の改善は可能だが、結局は経験ある開発者の手を借りる必要があり、「生産性税（productivity tax）」として66%の開発者が問題を報告している。  

- 技術的な断面（具体例）  
  - 使用技術の断片：JSONデータ、Redis等のデータストア、npm run devでのホットリロードなどが自動構成される。  
  - 問題点：エンドポイントのルーティングや認証ミドルウェアが不完全、ユニットテストが未作成、依存関係のバージョン固定や脆弱性チェックが行われていない、リポジトリ構造が探索しづらい。

## 実践ポイント
- 本番に出す前提で使わない：趣味やプロトタイプ以外は必ず経験あるエンジニアのレビューを入れる。  
- セキュリティ最優先：パブリックに置くデータは最小限にし、認証／権限チェックを必須化。静的解析／依存脆弱性スキャン（例：Snyk、Dependabot）を導入する。  
- リポジトリの整備：トップレベルに主要ファイルを置き、READMEに環境構築・起動手順（npm run dev等）を明記する。  
- テストとCIを入れる：最低限のユニットテストとCIパイプラインで自動テストを走らせる。  
- 非エンジニア向け教育：ターミナルの基本操作、差分の考え方（リリース＝上書き）、ログやエラーメッセージの読み方を簡単に学ばせる。  
- ステージング環境の運用：直接プロダクションにデプロイせず、公開前にステージングでセキュリティ・動作確認を行う。  
- コミュニティへの公開は慎重に：外部レビューを求める場合、公開リポジトリに個人情報や鍵・シークレットがないかを必ず確認する。

短時間で「動くもの」を生む力は魅力的だが、それが「信頼できるソフトウェア」になるかは別問題である点を押さえておくことが重要だ。日本企業や個人がこの波に乗るなら、開発ガバナンスと最小限の技術的知識をセットで整備することが鍵になる。
