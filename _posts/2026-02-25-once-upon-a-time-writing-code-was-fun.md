# Once Upon a Time, Writing Code Was Fun - かつて、コードを書くことは楽しかった
あの「深夜の高揚感」はどこへ行った？AI時代に失われつつある“作る喜び”と取り戻し方

## 要約
AIでコーディングの効率は飛躍的に上がったが、手を動かして作り切ることで得られた「理解」「所有感」「フロー」が薄れている――そんな観察と対処の提案。

## この記事を読むべき理由
日本の現場では長寿命なシステム保守やオンコール対応が多く、コードを「深く理解する」ことが生産性と品質に直結する。AI導入で速度は出せるが、トラブル対応力や技術力の育成に影響が出る可能性があるため、今のうちに対策を知る価値がある。

## 詳細解説
- AI加速のパラドックス  
  - 自動生成で短時間に大量のコードが作れる一方、作られたコードが「自分の頭で鍛えられた産物」ではなくなる。達成感は減り、出力指標だけが良く見える現象が起きる。
- フローと身体性の喪失  
  - 手を動かしながら思考を整理する「フロー状態」は、バグや境界ケースを先回りする力を養う。AIに任せてしまうと、思考と手の同期が失われ記憶・直感に残りにくい。
- 所有権ギャップと運用での影響  
  - 生成コードに起因する障害発生時、昔のように「どこを見ればいいか」が直感的に分からず、デバッグに時間がかかる。レビューはできても、体に染み込んだ設計知識が不足する。
- 役割の変化（悪い話だけではない）  
  - タイプする役割から、アーキテクト・質問設定・トレードオフ理解へと職能がシフトしている。抽象設計やシステム思考の価値が相対的に上がる。

## 実践ポイント
- 意図的な「AIオフ」時間を作る：週に一度は手で実装する課題を設定する。  
- コアロジックは自分で実装、AIはボイラープレート・テスト生成に限定する。  
- マインドマップや状態遷移図でメンタルモデルを可視化してからAIを使う。  
- 生成コードに対しては必ずコードウォークスルーを実施し、設計意図をドキュメント化する。  
- 障害時の所有権を担保するため「誰がどの部分を作ったか」「想定した振る舞い」はチームで共有する。  
- 若手向けに手書きコーディング練習（Kata）やペアプロを導入し、技能継承を補強する。

短い結論：AIは強力な道具だが、「たまには自分の手で作る」ことを意図的に残さなければ、技術的な“所有”と深い理解を失うリスクがある。
