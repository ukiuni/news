# Code is cheap. Show me the talk - コードは安い。議論を見せてくれ
もうコードだけじゃ通用しない――AI時代に問われる「価値」と「責任」

## 要約
LLM（大規模言語モデル）によってコード作成は劇的に短時間化・自動化され、コードそのものの「希少価値」は下がった。これからはアイデア、説明責任、設計・運用の質が勝負になる。

## この記事を読むべき理由
日本のエンジニアやプロダクト担当は、開発プロセス、採用基準、OSS運用、品質保証の常識が変わる波に直面している。早めに実務での影響と対策を押さえておかないと、品質リスクやガバナンスの穴に悩まされる。

## 詳細解説
- LLMの能力：ドキュメント、テスト、UI、インフラ設定まで一気に生成できるため、従来の「手書きで時間をかけた証拠」がなくても見栄えの良い成果物が出せる。結果、READMEや美しいコメントだけでは信頼できなくなった。
- 生産性の逆転：以前は「良いコード＝時間と手間の賜物」だったが、LLMを適切に操れば数週間分の仕事が数時間で済むこともある。逆に大量の“機械生成”コード（いわゆるslop）が増え、検証コストは上がる。
- 重要性のシフト：機能するコードはあふれるため、価値は「誰が・なぜ・どう管理するか（プロヴェナンスとガバナンス）」に移る。責任を明確にできる組織やプロセスが優位になる。
- OSSとコミュニティ：外観が整ったPRやドキュメントが増えて、貢献者の信頼性を見分けにくくなる。メンテナンス性や長期的な運営方針を重視する必要がある。
- リスク面：安全性や法的責任、依存関係、ライセンス問題が表面化しやすく、特に金融・組み込み・医療など規制の強い分野では慎重な運用が求められる。
- 新しい技術運用：プロンプト設計、LLMエージェント、生成物の検証自動化、ビルドの再現性、署名・メタデータによる出所管理などが実務スキルになる。

## 実践ポイント
- LLMを試す：まず実業務で小さなプロトタイプを作り、生成物の得手不得手を把握する。  
- 厳格な自動テストとCI：生成コードは必ずテストで網羅し、CIで継続検証する。  
- プロヴェナンスを残す：PRに「誰が何をしたか」「どのプロンプトで生成したか」を記録するポリシーを作る。  
- 人間の設計レビューを重視：実装よりもアーキテクチャ、トレードオフ、運用方針をレビュー項目に入れる。  
- ガバナンス導入：OSS採用基準、ライセンス監査、セキュリティスキャンを組織ルールにする。  
- スキルアップ：エンジニアに対してプロンプト設計、LLM検証、メタデータ管理の教育を行う。  

短く言えば、コードは「安く」なっても、責任と設計の価値は下がらない。AI時代の仕事は「より良い問いを立て、説明できること」と「生成物を検証・運用する能力」にシフトする。
