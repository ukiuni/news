# //go:fix inline and the source-level inliner - //go:fix inline とソースレベル・インライナ
1行で置き換える未来へ：Goの //go:fix inline で安全にAPIを一斉移行する

## 要約
Go 1.26で導入されたソースレベルのインライナは、ライブラリ作者が簡単なコメントディレクティブで呼び出しを安全に置き換え、go fix や gopls を使って大規模なAPI移行を自動化できる仕組みです。

## この記事を読むべき理由
古いAPIの置換や設計ミスの修正を、手作業で多くのリポジトリに適用するのは手間とリスクが高いです。日本のOSSメンテナや企業の大規模コードベースでも、安全に自動移行できる手段は価値が高く、本機能はその有力な武器になります。

## 詳細解説
- 何が起きるか  
  「ソースレベルのインライナ」は呼び出しsiteを呼び出される関数の本体で置き換え、引数をパラメータに差し込んだ上でソースを恒久的に書き換えます。これはコンパイラ内部で行う一時的な最適化とは異なり、ソースを直接更新する点が特徴です。

- 使い方（概念）  
  ライブラリ側で古い関数・型・定数に対してコメントでディレクティブを付けます：
  ```go
  //go:fix inline
  func ReadFile(filename string) ([]byte, error) { return os.ReadFile(filename) }
  ```
  その後、ユーザー側で go fix を走らせると、呼び出しが自動で新しい呼び出しに置き換えられます。gopls を使うとエディタ上で即時に診断と提案が見えます。

- 実例（概要）  
  - ioutil.ReadFile → os.ReadFile の置換：インポートも自動調整され、元の呼び出しが新しい呼び出しに書き換わります。  
  - oldmath → newmath のように引数順が違う関数や不要なラッパーを、新API呼び出しに置き換えて設計改善を促せます。型や定数も forwarding (= newpkg.T) で移行可能です。

- 技術的課題（主要点）  
  1. パラメータ削除：引数がリテラルの場合でも、複数回参照されると不整合を避けるため束縛変数を挿入する必要がある。  
  2. 副作用と評価順序：引数の評価順やcallee内の評価との関係で振る舞いが変わらないことを保証する必要があり、安全性が証明できない場合は保守的に変数束縛を入れる。  
  3. コンパイル時評価の副作用：定数化されるとコンパイル時エラーになる表現が生成され得るため、生成前に制約解析を行い、不適合なら束縛を残す。  
  4. 影（shadowing）やスコープ変化、ループ内使用などの細かいケースも扱うため、実装はコンパイラに近い複雑さ（数千行）になる。  
  このためインライナはかなり保守的に振る舞い、場合によっては自動化後に人の目でスタイル調整が必要になることがあります。

- 運用実績  
  同様の手法は他言語で大規模リポジトリのAPI削除に使われており、Go版も既に大量の自動パッチ作成実績があります。

## 実践ポイント
- ライブラリ作者向け
  - 置換したい関数/型/定数に //go:fix inline を付けて実装を新APIへフォワードする。  
  - リリースノートでユーザーに go fix の実行を促す。CIで自動で go fix を試すのも有効。

- 利用者（利用する側）向け
  - gopls を有効にしてエディタ内の診断と提案を確認する（VS Code の Source Action）。  
  - まずローカルで go fix -diff ./... を実行して差分を確認し、テストを必ず全通させる。  
  - 副作用や評価順が重要な箇所は人手でレビューし、必要なら手動で束縛やリファクタを行う。

- 組織的運用
  - PRを自動生成する夜間バッチ＋CI（テスト・ビルド）で安全に大規模移行を進めるのが理想。  
  - 移行後は不要になった古いパッケージを段階的に削除して依存を軽くする。

この機能は「安全性を重視した自動リファクタリング」を目指すための強力な道具です。まずは小さなAPIで試し、テストとコードレビューのワークフローを整えてから本格導入してください。
