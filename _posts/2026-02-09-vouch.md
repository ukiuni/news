# Vouch - 明示的な「推薦(Vouch)」で作るコミュニティ信頼管理
AI時代のOSSに必要な“明示的な信頼”を導入する――荒れるPRを防ぐコミュニティ運営ツール

## 要約
Vouchは「誰かが誰かを推薦（vouch）／拒否（denounce）する」ことで貢献者の参加可否を制御する軽量なコミュニティ信頼管理システム。GitHub ActionsとNushellベースのCLIで手軽に導入でき、プロジェクト横断の“ウェブ・オブ・トラスト”構築を想定している。

## この記事を読むべき理由
AI生成コードの増加で「見た目はまともだが品質の低い」PRが増加する中、日本のOSS・社内オープンな開発でも信頼ベースの明文化が求められている。Vouchは既存ワークフローに組み込みやすく、スパムや低品質な貢献を自動検出・制御できるため、レビュー工数削減やコミュニティ健全化に直結する。

## 詳細解説
- 基本概念：プロジェクトが「VOUCHED.td」という単一フラットファイルで推奨/拒否リストを管理。1行1ハンドル、プラットフォーム接頭辞（例: github:mitchellh）、前に「-」でdenouncedを示す。コメントは#。
- なぜ必要か：従来は「門戸の高さ（実装負荷）」が品質フィルタだったが、AIで敷居が下がり悪意・低品質貢献が増加。明示的な推薦により既存の信頼者から段階的に信用を伝播させる。
- GitHub連携：公式のActions群で簡単導入。
  - check-pr (pull_request_target)：PR作成時に作者がvouchedかをチェックし自動クローズや許可の柔軟設定が可能。
  - manage-by-issue / manage-by-discussion：コラボレーターがIssue/Discussionコメントでvouch/denounceを実行しVOUCHED.tdを更新。
- CLIとライブラリ：Nushellモジュールとして提供。依存はNushellのみ。主要コマンドは vouch check/add/denounce、GitHub向けコマンドも含む（gh-check-pr 等）。
- ウェブ・オブ・トラスト：他プロジェクトのvouchリストを参照可能。相互に信頼を共有すると、多プロジェクトで信用を継承できる。
- 注意点：実験的（Ghosttyで運用例あり）。ポリシー設計や悪用防止（恣意的な排除）をプロジェクト側で慎重に決める必要がある。

## 実践ポイント
- 導入手順（簡易）
  1. リポジトリに .github/VOUCHED.td を作成し基本ユーザーを書き込む。
  2. GitHub Actionsに check-pr / manage-by-issue を追加して自動チェックと運用を両立。
  3. CIで GITHUB_TOKEN を設定（ghトークンがあれば自動利用）。
  4. ローカル運用にはNushellを入れてCLI操作を試す。
- すぐ使えるコマンド例（dry-run推奨）:
```bash
vouch check someuser
vouch add someuser --write
vouch denounce badactor --reason "Submitted AI slop" --write
vouch gh-check-pr 123 --repo owner/repo --auto-close
```
- 日本の現場での活用案：企業OSSやOSS参加の門戸が広い組織で、採用候補者や外部協業者の信頼管理、脆弱性対応チームのメンバー管理などに適用可能。導入前にポリシー（誰がvouchできるか、異議申し立て手順）を文書化すること。

以上を踏まえ、まずは小さなリポジトリで動作確認（dry-run）→運用ルールを定める流れを推奨。Vouchは「信頼を明示する」新しい運用モデルの試金石となる。
