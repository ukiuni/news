# CSS sucks because we don't bother learning it (2022) - CSSが嫌われるのは学ぼうとしないから
なぜCSSが“めんどくさい”と敬遠されるのか — 学び直しで見える本当の理由と実務で使える処方箋

## 要約
CSSが「使いにくい」と言われるのは、言語そのものの欠陥ではなく、設計や基礎を学ばずに都度その場しのぎのハックで乗り切っているからだ、という主張。

## この記事を読むべき理由
フロントエンド実装を外注したり、バックエンド寄りの開発者が後回しにしがちな日本のプロジェクトでは、CSSの作り方のまずさが保守負荷や品質低下につながる。基礎を押さえれば生産性とUXが劇的に改善するため、実務で即役立つ視点を得られる。

## 詳細解説
- 本質論：CSSは「ただの構文」ではなく、レイアウトや見た目を設計するためのツール群（ボックスモデル、カスケード、特異性、スタッキングコンテキストなど）であり、適切なメンタルモデルが必要。習得には時間と設計経験が要る。  
- よくある失敗パターン：プロジェクト終盤で慌ててCSSを書く→position:absoluteや!importantの乱用、冗長なクラス名、フレームワークを場当たり的に被せる。これらは一時的に動くが、レスポンシブ対応や改修で崩れやすい。  
- レイアウトの歴史：CSS2時代のセンタリング問題や回避策の蓄積があり、CSS3（Flexbox, Grid）で解決できる場面が増えた。ただし「新しい技術で古いHTML構造を無理やり整える」だけでは保守性は向上しない。  
- 方針の重要性：あるプロパティ（例：position:absolute）を“禁止”しておくルールは、使いどころを明確にするための戦術。設計とガイドラインがないと誰もが勝手に使って破壊的なスタイルを作る。  
- 学習の勧め：単発のスニペット探しではなく、体系的な学習（レイアウト、アクセシビリティ、レスポンシブ設計、カスケードの原理）を行えば、CSSに対する見方が変わる。

## 実践ポイント
- 最優先で学ぶべき基礎：ボックスモデル、display（block/inline/flex/grid）、position（static/relative/absolute/fixed/sticky）、カスケードと特異性、レスポンシブの基本。  
- レイアウトにはまずGrid/Flexboxを使う：古い絶対配置より可変性・保守性が高い。  
- HTMLを正しく書く：意味のあるマークアップ（セマンティクス）はCSSをシンプルにする。  
- ルールを作る：プロジェクト固有の禁止事項（!importantや無制限のposition:absoluteなど）と命名規約（BEMなど）を文書化する。  
- ツール導入：stylelint、プレプロセッサ（必要なら）、UIコンポーネントライブラリで再利用性を高める。  
- 小さく学ぶ：1ページ分のデザインを最初からCSSで作る練習を繰り返す。断片的なコピー&pasteより効果的。  
- チーム運用：コードレビューでCSS設計をチェックし、スタイルガイドをCIに組み込む。  
- 学習リソースを投資する：短いハウツー記事より体系的なコースや実践課題を選ぶと習得が早い。

短くまとめると、CSSが「嫌い」なのはCSS自体ではなく学び方と設計の欠如によるもの。日本の現場でも、基礎とルールづくりに投資すれば保守性とデリバリースピードは確実に改善する。
