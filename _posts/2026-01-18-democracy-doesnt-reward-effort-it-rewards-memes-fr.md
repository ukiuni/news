# Democracy doesn't reward effort. It rewards memes. (From an experiment letting GitHub reactions decide what ships) - 民主主義は努力を評価しない。ミームを評価する（GitHubリアクションでマージを決める実験より）
破壊か進化か？「毎日マージ」で一気に加速したオープンソース実験の顛末

## 要約
OpenChaosという公開リポジトリで、GitHubのリアクション投票で「どのPRをマージするか」を決める実験を行ったところ、可視性や「速さ」を訴える提案が支持され、週1回の大きな変化が週7回の積み重ねに変わる――という加速とガバナンスの形成が起きた。

## この記事を読むべき理由
日本でもOSS運営やコミュニティ主導のプロジェクトが増える中、投票・可視化・自動化がどう現実の決定に影響するかは実務的に重要。実験は「どの提案が支持されるか」だけでなく、運用上の落とし穴（APIの既定表示件数、CI不整合、マージ自動化の波及）も露呈しています。

## 詳細解説
- 実験の仕組み
  - 誰でもPRを出せるリポジトリで、GitHubのリアクション（👍など）数で順位付け。最多票のPRをマージするというルール。ルール自体もコミュニティで変えられる点が特徴。
- 視認性の問題（Invisible Ballot）
  - GitHub APIは既定で30件しか返さないため、40以上のオープンPRがあると古いPRが一覧に出ず投票できない状態に。これは「投票の前提（投票対象が見えること）」を損なうバグで、メンテナが介入して修正した。投票制度はまずインフラが正しく動くことが前提。
- ミーム vs. 努力
  - 「Rustに書き直せ」PR（#13）は458票を集めたが、当初はビルドすら通らない状態だった。その後別の寄稿者が実装を完成させるPR（#91）を出すと、きちんと作り込まれた方は18票に留まった。結果：「票はタイトルやミーム性に引っ張られやすい」。
- 速度が選好される
  - 単純な「毎日マージする」PR（#51）が急伸して788票で勝利。これにより7週かかるはずの変更が7日で反映されるシナリオが成立。さらに自動マージ（GitHub Actions）を提案するPR（#63）が現れ、人的トリガー不要の「指数的なカオス」が示唆された。
- 数値の変化（Week1 → Week2）
  - Stars: 400 → 690 (+73%)
  - Forks: 35 → 57 (+63%)
  - Open PRs: 30 → 45 (+50%)
- ガバナンスの萌芽
  - 投票でルール（マージ頻度やダウンボート機能）が変わり、コミュニティが「カオスを制御する仕組み」を作り始めた。投票は破壊衝動よりも制度設計に向かう傾向が見えた。

## 実践ポイント
- 可視性を保証する：APIの既定値やページネーションで投票対象が隠れないようにする。投票制度はまず「見える化」が必要。
- CIとビルドの必須化：票を集める前に最低限のCI通過を条件にすると「ミームだけで壊れる」リスクを下げられる。
- リリース頻度の設計：ユーザーやコントリビュータが「速さ」を好むことを踏まえ、デイリーロールアウトは魅力的だがコンフリクトや品質管理のコストも計上する。
- 自動化は段階的に：自動マージはスケールするが暴走も招く。ルール（テスト必須、レビュアー承認、フェイルセーフ）を先に決める。
- 貢献の敷居を下げる：Rustなど学習コストの高い言語への全面移行はコミュニティ参加率に影響する。翻訳ドキュメントや小さなタスクを残す設計を検討する。
- ロードマップを公開する手段としての「キュー」：日次マージで次週分の変更が見える化され、期待とリスク管理に使える。

短く言えば、民主的な投票で動くプロジェクトは「速さと見えやすさ」が勝ち、制度設計（投票ルール、CI、可視化）がその方向性を左右する。日本のOSS運営でも応用できる実践的な教訓が詰まっています。
