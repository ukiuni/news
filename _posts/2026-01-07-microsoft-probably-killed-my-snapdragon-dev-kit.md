# Microsoft probably killed my Snapdragon Dev Kit - Microsoftがおそらく私のSnapdragon開発キットを“殺した”
Windowsアップデートで“燦然と輝いていた32GBのARM開発マシン”が突然起動不能に——開発キット廃棄の危機から学ぶ現場の教訓

## 要約
Windows 11のセキュリティ更新（KB5068861）が失敗し、その後の再起動でプロファイル消失・アプリ不可・起動ループ・UEFI操作不能といった致命的な事象が発生。Qualcommが開発キットのサポートを打ち切っていたため、復旧手段がほぼ存在せず、実質“文鎮化”した事例。

## この記事を読むべき理由
日本でもARMベースのWindows端末（Snapdragon系）に注目が集まる中、開発キットや実機でのアップデート運用・復旧設計の重要性は増している。本件は「OSアップデート→ファーム書換り・復旧パス欠如→業務停止」に直結するリスクの実例で、日本の開発・導入現場が取るべき対策を示す生きたケーススタディになる。

## 詳細解説
- 発生状況: 2024年10月から日常運用されていたSnapdragon Dev Kit（Snapdragon X Elite, 32GB RAM, 512GB SSD）で、2024年12月にKB5068861のインストールが失敗してロールバック。後日、同更新を適用し再起動した際にロールバックが不完全になり、4回の再起動の末にWindowsへ入るもMicrosoftアカウントのプロファイルが消失（PINでログイン可能だが新規プロファイル）。Windows TerminalやEvent Viewerなど主要アプリが起動不可。
- その後の挙動: 再起動試行の途中でロゴ表示直後にランダムで再起動または電源断。UEFIのBoot Device Selection（BDS）メニューも選択不能やフリーズが発生。USBインストールを試み、Secure Boot無効化やUSBブート優先、外部ディスプレイ有効化など設定を行い、Windowsインストーラで再インストールを実行するも、ドライバ指定画面でフリーズ→電源断。それ以降はスナップドラゴンロゴから先に進まない。SSD自体は別PCで問題なし。
- 技術的な考察: 考えられる原因は複数で、更新がUEFIブートローダやファームを書き換え（あるいは一部破損）した可能性、Secure Boot／TPMの状態不整合、もしくは電源管理関連のファーム不具合によるランダム電源断。開発キットにはメーカー側の公式なファーム復旧ツールやサポート経路が無く、通常のOEM製品であれば使えるリカバリパスが存在しない点が致命的だった。
- プラットフォーム評価: 本体の性能やARM Windowsの挙動自体は良好で、問題は「開発キット特有のサポート欠如」と「OS更新が絡んだ低レイヤー障害の回復手段がない」ことに集約される。

## 実践ポイント
- 重要環境での更新はステージング環境で先行検証する：特に開発キットや非消費者向け機種は本番適用前に必須。
- 常時フルバックアップとイメージ保存：リカバリ用のSSDイメージ、WinPEブートUSB、ドライバ類のオフライン保存（ZIPで複数場所に保管）。
- ファーム／UEFIの復旧手段を確認：購入時点で製造元のリカバリ手順・ツールの有無を確認し、ない場合は作業計画を変更する。
- 自動更新は管理下で運用：重要端末はWindows Updateの自動適用を遅延させ、更新ログと既知の不具合情報をチェックしてから適用する。
- 代替機と冗長化：開発／検証用は代替機を準備し、文鎮化した場合の業務継続策を定める。
- ベンダーとサプライチェーンへの要求：特に日本企業や導入責任者は、ARMプラットフォーム採用時にファーム復旧経路とサポートライフサイクルを契約要件に含める。

短い結論：高性能なARM機は魅力的だが、開発キットやサポート切れデバイスでは「更新＝潜在的な死命のトリガー」になりうる。導入前に復旧計画を作ることが最も安い保険である。
