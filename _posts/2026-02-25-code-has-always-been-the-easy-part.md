# Code has always been the easy part - コードはいつだって簡単な部分だった
コードが「ほぼ無料」になる今、エンジニアの仕事はどう変わるのか？

## 要約
AIでコード生成コストが急速に下がる中、重要なのは「コード」ではなくシステム設計やチームの働き方、顧客価値をどう作るかだと主張する記事です。

## この記事を読むべき理由
日本の開発現場でも、コード中心の評価や面接手法、レビューワークフローが通用しなくなりつつあります。今のうちに「人とプロセス」を再設計する視点を持つことが競争優位になります。

## 詳細解説
- 核心：著者は「コードは常に容易な部分だった」と断言。最近の大規模言語モデル（例：Claude Codeなど）でコード生産のコストが急激に下がっており、コードを書く行為自体はさらに軽くなる。
- 結果：価値を生むのは「システム全体」と「人＋技術の組み合わせ」。要件理解、ドメイン知識、信頼性、運用設計、顧客価値の継続的改善といった領域が本丸になる。
- 組織への影響：過去にもウェブ、CI/CD、モバイル、MLなどがチームの働き方を刷新してきたが、今回も同様に「社会的契約（仕事の分担や評価）」の再定義が必要。コードレビューの疲弊、採用手法（ホワイトボードでのアルゴリズム問題等）の意味喪失も指摘。
- リーダー視点：小規模の「ダークソフトウェア工場（少人数で完結）」の成功例もある一方で、大勢の人をどう組み込むかを創造的に考える必要がある。倫理的・資本主義的な問題意識も併せて提示。

## 実践ポイント
- プロダクト価値を中心にKPIを再定義する（コードの量ではなく顧客への効果で評価）。
- 採用と評価基準を刷新する：実務シナリオやドメイン理解を重視し、ホワイトボード式のアルゴリズム問答に依存しない。
- 開発フローを自動化（テスト・CI/CD）し、コード生成ツールを安全に取り込むガイドラインを作る。
- コードレビューの負担軽減：レビュー対象の定義、ペアプログラミング、AIアシストの活用を検討。
- 人材育成は「ドメイン知識」と「システム設計能力」に投資する（機械で代替されにくい領域を強化）。

簡潔に言えば、コードを書けることは当たり前の土台になりつつあり、これからは「何を作るか」「誰とどう作るか」が差を生みます。日本の現場でも今のうちに組織と評価を見直すことが重要です。
