# Show HN: HyTags – HTML as a Programming Language - HyTags — HTMLをプログラミング言語として使う
HTMLに振る舞いを直接書く時代へ — hyTagsでサーバー駆動UIをもっとシンプルに

## 要約
hyTagsは「HTMLの中に振る舞いを宣言的に書く」小さな言語で、バックエンド主導のアプリや静的サイトでフロントエンドの複雑さを減らすことを狙います。イベントやDOM挿入、再利用可能なコンポーネントをHTMLタグとして記述できます。

## この記事を読むべき理由
フロントエンドとバックエンドを分離するモダンな設計は柔軟ですが、API設計やルーティング、データ同期の重複という税を払います。日本の中小チームや業務系アプリ、あるいはJAMstack的な静的サイト運用では、シンプルにUIの振る舞いをHTML側で完結させられる手法が生産性を劇的に高める可能性があります。まずは概念と使いどころを押さえておく価値があります。

## 詳細解説
- コアコンセプト：hyTagsはHTMLに埋め込む宣言的DSLです。HTML要素の中にイベント（例：<on click>）やDOM操作（例：<selection-insert-after>）を直接書いて、UIの振る舞いを定義します。結果として「マークアップ」と「振る舞い」が同一ファイルに近い形で保たれます。
- どんな問題を解くか：従来のバックエンド＋フロントエンド分割では、APIの設計、データ移送、クライアント側のステート管理など複数レイヤーが必要になります。hyTagsはそこを単純化し、サーバーで生成するHTMLに必要なインタラクションを直接付与して配信する使い方を想定しています。
- 技術的な特徴：
  - 宣言的イベントハンドリング：<on click>…</on>のような構造でイベント処理を記述。
  - DOM操作用のビルトインタグ：挿入・置換・削除などを宣言的に指定可能。
  - 再利用性：小さなUI断片をコンポーネント化して組み合わせられる設計。
  - リアクティブ風のプロパティ：map-valueやmap-propertyのような仕組みで、要素の状態に応じた反応を組める。
  - ユースケース：動的フォームフィールド、HTTPリクエスト結果の部分挿入、スタイル遷移アニメーション、再帰的なロジック（例：フィボナッチ）などがサンプルとして挙げられています。
- 実行環境と制約：hyTagsはブラウザ側で動く仕組みと、サーバー側でレンダリングされたHTMLに適用するパターンの両方を想定しているようです。利点は導入の敷居が低い点（既存のテンプレートに混ぜやすい）がある一方、複雑な状態管理や大規模アプリでは専用フレームワークの方が適している場合があります。
- セキュリティと保守性：HTMLにロジックを混在させることは可読性と責任の分離に影響します。日本の保守重視の現場では、ドキュメント化・テスト・コードレビュールールを厳しくすることで安全に利用できます。

例（簡略化）：
```html
<!-- html -->
<button>Say something</button>
<on click>
  <selection-insert-after>
    <div>Hello, World 🚀</div>
  </selection-insert-after>
</on>
```

## 実践ポイント
- 小さく試す：まずはフォームの動的追加や、非同期レスポンスの部分差し替えなど、限定的な機能で導入して効果を確認する。
- 既存テンプレートとの併用：バックエンドテンプレート（Rails, Django, PHP, Go等）にhyTagsの断片を埋めて段階的に移行すると学習コストが低い。
- 保守ルールを決める：HTML内にロジックが入るため、命名規則・コンポーネント分割・テスト方針をチームで決めておく。
- SEOとアクセシビリティを確認：サーバー側で初期レンダリングを行うならSEO面は比較的安全。動的操作がアクセシビリティを損なわないか確認する。
- 適材適所の判断：リアルタイム性や複雑なクライアント状態が必要な部分は従来のフロントフレームワークのままにし、管理画面やシンプルなインタラクションはhyTagsで効率化するのが現実的。

興味があるなら公式サイト（hytags.org）でサンプルを触って、小さなUIを1つ作ってみることをおすすめします。hyTagsは「分離のコスト」を下げたい現場で有望な選択肢になり得ます。
