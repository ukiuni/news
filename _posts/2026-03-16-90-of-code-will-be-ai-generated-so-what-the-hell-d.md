# 90% of Code Will Be AI-Generated — So What the Hell Do We Actually Do? - コードの90%はAIが書く──じゃあ僕らは一体何をするのか？
「コードの9割がAI生成」に備える——今すぐ身につけるべき“理解力”と実務習慣

## 要約
AI生成コードの比率は急速に増えており（現状で数十％、採用が進む現場ではさらに上昇が予想される）、重要なのは「どれだけ書けるか」ではなく「AIが書いたコードを理解・検証・設計できるか」である。

## この記事を読むべき理由
日本の開発現場でもAIツール導入は進んでおり、若手雇用や品質管理の課題が既に出始めている。単にAIを使うだけでなく、事故やセキュリティ問題を防ぐための具体的なスキルセットが今すぐ役立ちます。

## 詳細解説
- 「90%」の意味：AIがアシストする割合（オートコンプリート、ボイラープレート、テスト、ドキュメント、リファクタ提案など）が合算されて高値になる。AIがアプリ全体を丸ごと設計するわけではない。  
- 落とし穴：AIで生成したコードを表面的に受け入れると、設計意図や状態遷移を理解しないまま本番で障害対応に苦しむ（実話ベースの失敗談）。  
- 実証データ：複雑で成熟したコードベースでは、AIを使うことでコンテキスト提供や検証のオーバーヘッドが発生し、逆に遅くなるケースがある（研究での指摘）。若手ほどAI出力を無批判に受け入れがちで、品質・セキュリティ問題が残る傾向。  
- 価値のシフト：人の価値は「何を作るか」から「なぜそうするか／壊れた時に直せるか／長期的な設計判断ができるか」へ移る。AI運用そのもの（設定・制限・監査）を理解する人材も重宝される。

## 実践ポイント
- 週に1日はAIなしでコーディング：基礎力を維持してAI出力を正しく評価できるようにする。  
- AI生成コードは「動くか」ではなく「壊れるとどうなるか」を読む：null、競合、分散環境、認証・権限などの観点で攻める。  
- 体系設計（System Design）を学ぶ：コンポーネント単位ではなく、既存データモデル／運用制約を踏まえた設計力を磨く。  
- LLM／ツールの限界を理解する：モデルの誤りパターン、トレーニング由来の盲点、コストとレイテンシを含めた運用判断が必須。  
- 思考のログを残す（ドキュメント化／公開）：採用理由やトレードオフを記録し、将来の検証や引き継ぎを容易にする。  
- マージ前に自問する習慣：「これを本番で2時に直せるか？」答えがNOなら説明を求める・自分で書き直す。

以上を意識すれば、AIが「コードを書く割合」を増やしても、理解力や設計力で差をつけられます。どの10%を自分のものにするかが勝負です。
