# Create a semi-working desktop only based on runtime Windows 3.10 shipped with Windows 95 installer - Windows 95 インストーラに同梱された Windows 3.10 ランタイムだけで作った半分動くデスクトップ
Windows 95インストーラの“中身”だけで、Windows 3.10ランタイムを組み合わせて動くデスクトップを部分再現した試み

## 要約
筆者はWindows 95のインストーラに含まれていたWindows 3.10ランタイムのファイル群を抽出・組み合わせることで、完全ではないが実際に操作できる「半分動く」デスクトップ環境を再現しています。レトロOSの構成要素や互換性の限界を探る面白い実験です。

## この記事を読むべき理由
レガシー環境の解析やソフトウェア保存、互換性調査に関心のあるエンジニアや趣味者にとって、インストーラ内部に実行可能なランタイムがどれだけ詰まっているかを示す好例であり、Windowsの歴史的構造や実験的な手法を学べます。日本のレガシー系開発や組込み・エミュレーション分野でも応用可能な知見が得られます。

## 詳細解説
- アイデアの核  
  Windows 95のインストーラには、セットアップ中に使うためのWindows 3.1系（3.10相当）の実行ファイルやリソースが含まれています。筆者はそれらを単体で取り出し、最小限の構成ファイルと組み合わせて、プログラムマネージャやファイルマネージャなどのUIが動く状態を組み上げました。
- 何が動くか／何が動かないか  
  抽出したファイルで起動できるGUI要素やユーティリティはある一方で、ドライバやセットアップ用のサービス、ハードウェア依存部分は欠けるため完全動作はしません。結果として「見た目のデスクトップ」「一部アプリの起動」は可能でも、安定性・入出力周りで制約があります。
- 実験の意味合い  
  この手法は「インストーラ＝ただのパッケージではなく、実行可能なランタイムの集合体である」ことを示します。ソフト保存（software preservation）や古いアプリの再現、互換性検証の出発点として意義があります。また、どのDLL／EXEが最低限必要かを調べることで、OSのモジュール依存性の理解が深まります。
- 技術的な手法（概略）  
  インストーラのアーカイブ（CABや自己解凍EXE）からファイルを抽出し、必要な実行ファイル・設定ファイルを拾い上げる。次に16ビット実行環境（DOSベースの環境や互換エミュレータ）を用意して動作を確認します。依存するINIやDLLを逐次追加して動作範囲を広げるのが典型的な手順です。

## 実践ポイント
- 法的注意：必ず自分が正当に入手したインストーラや、利用が許諾されているアーカイブを用いること。
- 環境の用意：実機ではなく、DOSBox-X、PCem、あるいは専用の仮想環境で実験する。ホスト環境を汚さず安全に試せます。
- 抽出ツール：7-Zipやcabextractなどでインストーラ中のアーカイブを展開して必要なEXE/DLLを取り出す。
- 最小構成の見つけ方：まずProgram ManagerやFile ManagerなどUIプログラムを起動し、足りないDLL/INIをログで確認して順次追加することで動作領域を広げる。
- 学びのポイント：依存関係の解析、古いリソース形式の扱い、レガシー互換性の限界を体験的に学べる。業務で古いシステムを扱う場合のトラブルシュート能力向上にも役立ちます。

興味があれば、まずは合法的に入手したインストーラを用意して、エミュレータ上で小さく試してみてください。
