# I let Claude Code configure my Arch install - Claude CodeにArchの設定を任せてみた
LLMに任せて“ほぼ自動”で作ったArch環境――時間は激減、でも技術はどうなる？

## 要約
著者は中古ThinkPadにArchを入れ、Claude Code（LLM）に端末やdotfilesの調査・設定を丸ごと任せた。手作業より約95%時間を短縮し、構成はCLAUDE.mdでドキュメント化・バージョン管理されたが、手作業で培った技能の衰えを懸念している。

## この記事を読むべき理由
- 日本でも中古ノート＋軽量Linuxで自分専用環境を作る需要は高い。  
- LLMを導入すれば短時間で「使える」開発環境が作れる一方、運用とスキル維持のバランスが重要になるため、その実用性と落とし穴を理解しておくべき。

## 詳細解説
- 背景：Archはカスタマイズ自在だがドキュメント読解と手作業が必要になる。著者はその学習コストをLLMに委ねた。  
- 何を任せたか：端末エミュレータ（foot）などツール選定、設定ファイル（dotfiles）の生成、インストール済みソフトの一覧化と相互関係のドキュメント化（CLAUDE.md）。  
- ワークフロー：Claude Codeに「自分の好み」を伝え、オンラインで最適ツールを調査させ、指定どおりに設定を書かせる。結果は短く読みやすいdotfilesとリポジトリ（Codeberg）で管理。  
- 利点：時間短縮、設定の一貫性、再現性、マシン固有の「コンテキスト」をLLM側で保持して改善が継続できる点。  
- 懸念点：設定を自動生成すると、ウィンドウマネージャやTUIの細かなノウハウが衰える可能性。変更履歴は残るが「設計する楽しみ」は失われる場合がある。セキュリティや自動化の誤設定には注意。

## 実践ポイント
- 小さく試す：まずはサブノートや仮想マシンでLLMに設定を任せ、生成物をよくレビューする。  
- ドキュメント化：CLAUDE.mdのように「何を」「なぜ」入れたかを書き残し、将来の自分／チーム向けに説明可能にする。  
- バージョン管理：dotfilesはGitで管理し、差分が一目で分かるようにする（Codeberg/GitHub等）。  
- 技能維持：週に1回は手で設定変更を行うなど、手作業で学ぶ時間を意図的に確保する。  
- セキュリティ確認：自動生成された設定は権限や外部アクセスの設定を必ず検証する。  

短時間で「動く環境」を手に入れつつ、習熟と安全性をどう両立するかがキモ。日本の個人開発者や社内環境構築にも即効性のあるアプローチだが、使い方に配慮が必要である。
