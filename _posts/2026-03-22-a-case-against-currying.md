# A Case Against Currying - カリー化に反対する一件
カリー化は確かに“かっこいい”けど、実務では思わぬ落とし穴がある — 部分適用の真実と代替案

## 要約
カリー化（引数を段階的に適用する関数定義）は部分適用が自然にできる利点がある一方、型の形状や合成、パフォーマンス、実務上の扱いづらさといったデメリットも存在する、という議論。

## この記事を読むべき理由
Haskell等の関数型言語や、Rust/JavaScriptで関数設計に悩む日本のエンジニアが「どの引数スタイルを採るべきか」を判断するための実務的視点が得られます。

## 詳細解説
- 3つの引数スタイル
  - パラメーターリスト（命令型言語で一般的）
    ```rust
    fn f(p1: P1, p2: P2, p3: P3) -> R { ... }
    ```
  - カリー化（関数型でよく見る）
    ```haskell
    f p1 p2 p3 = ...
    -- 型: P1 -> P2 -> P3 -> R
    ```
  - タプル（単一引数に複数値を束ねる）
    ```haskell
    f (p1, p2, p3) = ...
    -- 型: (P1, P2, P3) -> R
    ```

- 部分適用
  - カリー化は左からの部分適用が自然で便利。例: add 1 が新しい関数を返す。
  - しかしタプル/パラメータリストでもラムダや「穴（$）」のような構文糖で同様の部分適用は表現可能で、必ずしもカリー化だけの専売特許ではない。

- 実務での問題点
  - 中間関数の生成によるオーバーヘッド（最適化で解消可能だが実装次第）。
  - 型の「形」の違い：関数の入力が複数ある場合にカリー化は出力側が関数になるため、汎用API（例: map）が期待する In -> Out と合わないことがある。map にタプルのリストを渡して sayHi を直接適用できない等の不便さ（uncurry が必要）。
    ```haskell
    sayHi name age = "Hi " ++ name ++ " " ++ show age
    people = [("Alice",70), ("Bob",30)]
    -- map sayHi people -- 型が合わずエラー。uncurry sayHi が必要
    ```
  - 依存型（Dependent Types）では、引数ごとに戻り型が変わる場合にカリー化が自然に働くため、カリー化が有利なケースもある（ただし依存型自体は上級者向け）。

## 実践ポイント
- 普段のアプリ/ライブラリ設計では、タプル引数（あるいは明示的なレコード）を検討すると合成性と型の一貫性が高まる。
- 部分適用が欲しい場面だけカリー化／トップレベルのラッパーを用意する（乱用を避ける）。
- 汎用関数（map, fold 等）は In -> Out の形を好むので、カリー化関数を頻繁に渡すなら uncurry やアダプタ関数を用意する。
- 言語やコンパイラの最適化特性を把握して、パフォーマンス問題が出るかどうかを測る（ベンチを取りましょう）。
- 言語仕様や標準ライブラリを設計する立場なら、タプルスタイル＋部分適用構文の採用を検討すると実務での扱いやすさが上がることがある。
