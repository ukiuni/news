# DOS Memory Management - DOSメモリ管理
知られざるDOSメモリ管理の「仕組み」と落とし穴：16バイト単位の世界が教える低レイヤの教訓

## 要約
DOSのメモリ管理は見た目は単純だが、段落（16バイト）単位の管理、MCB（メモリ制御ブロック）、ALLOC/DEALLOC/SETBLOCKの挙動、そして断片化／結合の手順に由来する興味深い落とし穴がある。

## この記事を読むべき理由
レガシー環境や組み込み系、レトロPC趣味、あるいはメモリ割当の基礎を学びたい日本のエンジニアにとって、DOSの設計は現代のヒープ管理や断片化対策、互換性の問題を理解する良い教材になるからです。

## 詳細解説
- メモリ単位とMCB  
  DOSはバイトではなく段落（16バイト）単位で管理する。各ブロック先頭にMCB（arena header）があり、署名（'M'/'Z'）、所有者（PSPアドレスまたは0＝空き）、サイズ（段落単位）を保持する。MCBチェーンは連続領域を示し、ポインタではなく位置とサイズで次を暗黙的に示す。

- 主要APIの振る舞い  
  ・ALLOC（48h）: 要求段落数を受け、成功時はAXに割当てたセグメントを返す。失敗時はAXにエラー、BXに最大確保可能サイズを返す。よく使われたテクニックはまずFFFFhで呼び、返り値で最大を得て再度割当てする方法。  
  ・DEALLOC（49h）: 所有者フィールドを0にするだけで、即時の結合は行わない。  
  ・SETBLOCK（4Ah）: サイズ変更（realloc的）。移動はしない。縮小は常に成功し、拡張は失敗することがある。成功時は呼び出しプロセスを所有者にするため、他プロセスのブロックを奪ってしまう可能性がある。

- 断片化と結合（coalescing）の仕組み  
  DEALLOCでは結合しない。SETBLOCKは後続の空きMCBをまとめる（呼び出し側が所有者になることと合わせて注意）。ALLOCはチェーン全体を走査して可能な限り空きブロックを結合してから最適ブロックを選ぶため、長いチェーンではコストが高いが最大ブロックを見つけられる。

- 典型的な「罠」・バグっぽい挙動  
  ・ゼロ長ブロック（MCBだけ）が存在しうる。  
  ・SETBLOCKで空きブロックを操作すると実質割当になる。  
  ・SETBLOCKが要求サイズより小さい最大値を返すと同時に、そのサイズまで既にブロックを拡張してしまう（ほぼバグ）。  
  これらは互換性維持のため放置された歴史的な振る舞い。

- 進化（DOS 2.11 / DOS 5.0）  
  DOS 2.11でAllocOper（INT 21h/58h）が追加され、first/best/last fitを選べるように。DOS 5.0ではUMB（Upper Memory Blocks）対応が入り、従来の連続領域に加えてUMB側のアリーナを使う戦略（UMB優先／UMBのみなど）が選べる。これらの設定はグローバル状態なので、変更したら元に戻すべき。

## 実践ポイント
- レトロ環境やエミュレータ（DOSBox, PCem）で試すと理解が深まる。  
- 自分で割当処理を書くなら「段落単位」「ヘッダ管理」「結合タイミング」を明確に設計する。  
- 他プロセスのメモリを弄らない（SETBLOCKは所有チェックを強制しない）。ツールやTSRを書いているならAllocOperの状態を変更したら必ず復元する。  
- 大きな連続領域が欲しいときは、ALLOCを2回呼ぶテクニック（まずFFFFhで最大を得る）を使う。  
- 歴史的設計は現代のmalloc/realloc/フラグメンテーション対策の教科書になる：ヘッダ肥大、結合戦略、グローバル設定の落とし穴を学ぼう。
