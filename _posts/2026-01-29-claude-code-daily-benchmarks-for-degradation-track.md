# Claude Code Daily Benchmarks for Degradation Tracking - Claude Code: 劣化検出のための日次ベンチマーク
魅力的タイトル: 「急落を見逃すな：Claude Code（Opus 4.5）を毎日監視して“静かな劣化”を早期検出する方法」

## 要約
Marginlabが公開する日次トラッカーは、Claude Code（現行SOTAモデル Opus 4.5）をCLIで実行し、SWE-Bench-Proの厳選サブセットでの性能低下を統計的に検出します。短期ノイズを平滑化するために日次・週次・月次で集計し、$p<0.05$ を基準に有意な劣化をアラートします。

## この記事を読むべき理由
AIモデルの小さな性能低下がプロダクション品質や開発ワークフローに即影響する日本のプロダクト開発現場では、外部の独立トラッカーで「実ユーザーが体感する性能」を継続監視する価値が高いからです。Anthropicの過去の劣化事例も踏まえ、事前検知体制の参考になります。

## 詳細解説
- 監視対象と実行方法：MarginlabはClaude Code CLIをそのまま使い、カスタムハーネスを噛ませずに検証を実施。これにより「ユーザーが実際に触る挙動」を反映します。対象モデルは Opus 4.5（記事時点）。
- ベンチ仕様：SWE-Bench-Proから汚染（トレーニングデータ被り）を避けたサブセットを厳選。日次評価はN=50、週次は直近7日で合算（約250評価）、月次は30日合算（約655評価）とし、合算するほど推定の不確実性は下がります。
- 可視化と統計判断：各期間のパス率と95%信頼区間（CI）を算出し、基準となる過去平均（Baseline 58%）の周りに「有意差なし」と判定される閾値帯を表示。例：日次は±14.0%、週次は±5.6%、月次は±3.4%（試行数に依存）。変化が閾値を越えると $p<0.05$ として「統計的に有意な劣化」と判定され、購読者へ通知。
- モデル化：各テストをBernoulli試行として扱い、成功確率$p$の推定に基づく。標準誤差は $SE=\sqrt{p(1-p)/N}$、95%CIは概ね $1.96\cdot SE$ で計算できます。
- 実データの所見（抜粋）：Baseline 58% に対し、最新日次は50%（50 eval）、7日は53%（250 eval）、30日は54%（655 eval）。30日ベースの変化は統計的に検出された例がある（記事内で「検出あり」と表示）。

## 実践ポイント
- すぐできる監視：プロダクトに組み込む評価を「実ユーザー実行に近いCLI実行」で行うと、実感と一致する指標が得られます。
- サンプル数の設計：検出感度はNに依存。変化量δを検出したければ必要試行数は概ね $$N\approx\frac{1.96^2\;p(1-p)}{\delta^2}$$ を目安に。例えば$p\approx0.58$で±3%を検出したければNは数百〜千規模が必要です。
- アラート設計：日次はノイズ大、週次・月次で信頼性アップ。短期アラートは「注意」、月次有意差は「実対応」のトリガーに使うのが現実的です。
- 日本市場への示唆：Anthropicや他ベンダーのモデル更新で日本語や国内向けコード生成性能が変わる可能性があるため、国内プロダクトは独自の品質ゲート（CIに組み込むベンチ、SLO設定、通知フロー）を持つことを推奨します。
- Marginlabの活用法：外部トラッカーを参照しつつ、自社で同様の「汚染耐性のある」「ユーザー相当の実行環境」で再現検証を行い、外部検出を確認・補強してください。
