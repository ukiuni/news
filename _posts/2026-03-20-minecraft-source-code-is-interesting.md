# Minecraft Source Code is Interesting - Minecraftのソースコードは面白い
PS3版ソース流出で見えた「実戦的で痒い所に効く」低メモリ最適化テクニック

## 要約
PS3向けにC++へ移植されたMinecraftの実装は、限られたハード資源で快適に動かすための実践的な工夫の宝庫だった――ポインタ詰め、ロックフリー設計、平面圧縮、空間ローカリティ最適化、そしてJava互換性の厳密再現まで、多彩なテクニックが詰まっている。

## この記事を読むべき理由
低メモリ・低スペック環境やクロスプラットフォームの再現性（例：サーバとクライアントで同じワールドを作る）を考える日本のゲーム／組込み系エンジニアやパフォーマンス改善に関心ある開発者にとって、実務に直結するヒントが満載だから。

## 詳細解説
- ポインタとカウンタを1つに詰める  
  仮想アドレスが実質 $48$ ビットしか使わない点を利用し、下位 $48$ ビットをポインタ、上位 $16$ ビットをカウンタに割り当てて1つの64ビット値で管理。実際には atomic な64ビット比較交換で一発更新でき、読み取り側はロック不要となる。概念的には：  
  $$\text{dataAndCount} = (\text{count} \ll 48) \;|\; (\text{pointer} \& (2^{48}-1))$$

- 軽量な世代的メモリ解放（自前GC）  
  更新時に古いポインタを即解放できない問題を、回転する3つの削除キューで解決。ポインタは少なくとも2フレーム待たせてから解放することで、簡易的なエポック方式を実装している。

- 平面単位のライト圧縮（80%程度の削減）  
  ライティングは多くの水平面が全0か全15になる特性を利用し、各Yレイヤーに1バイトのインデックス（例：128＝全0、129＝全15）を持たせ、混在レイヤーのみ実データを保持する。結果、1チャンクのライトメモリが数KBに削減される。

- Morton（Z-order）によるビット散在座標とメモリ局所性  
  4x4x4ブロック単位で X/Y/Z のビットをインターリーブ（モートンコード）してメモリ配置。3次元上の近傍がメモリ上でも近くなるためキャッシュヒット率が上がり、ライト伝播や衝突判定のコストが大幅に下がる。

- 自己記述的圧縮タグ  
  各小ブロックのインデックスに圧縮方式情報を埋め込み、完全に均一ならデータをインデックス内に直接埋める（0ビットモード）など、複数モードで効率を最大化。結果的にチャンクを数十KB→数KBに圧縮。

- ヒープを避ける物理ページ割当  
  標準ヒープの断片化が致命的になるため、ページ単位で生の物理割当（4KB境界）を使い、解放時に確実にページを戻す設計を採用。断片化によるヒッチを予防するトレードオフ。

- Javaとのビット単位互換性再現  
  Java版と同じ乱数生成器（LCGの係数やビット幅）、HashMapの補助ハッシュ、さらにはInputStream系やコレクションの振る舞いまで再現。これによりワールドシードやデータの決定的一致を保証している。

- 爆発は物理演算ではなく「放射線」モデル  
  16³格子の外周セルから約1,352本のレイを飛ばしてブロック破壊を判定。計算を簡素化することでリアルタイム性を確保している。

## 実践ポイント
- 小メモリ環境では「データ表現」を見直す（全一色プレーンのインデックス化、自己記述圧縮など）。  
- 競合回避は「読み取りゼロロック、書き換えのみ同期」の設計が有効（64bit CAS の活用など）。  
- 空間データはモートン順で配置するとキャッシュ効率が劇的に上がる。  
- クロスプラットフォームで同じ結果を出すなら、乱数・ハッシュ・標準ライブラリの振る舞いまで再現する覚悟を。  
- 長時間稼働するシステムではヒープ断片化を評価し、必要ならページ単位／オブジェクトプール等で対策する。

短時間で学べる実践演習案：小さな3Dグリッドでモートン順読み出しを実装し、キャッシュミス率を比較すること。あるいは簡易エポックGC（三つの削除キュー）を試してロックフリー読取の感触を掴むと良いです。
