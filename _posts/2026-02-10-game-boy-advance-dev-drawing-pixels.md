# Game Boy Advance Dev: Drawing Pixels - Game Boy Advance開発：ピクセル描画

Game Boy Advanceでドットが自在に動く！レトロハードで「線・図形・ドット」を描く超入門

## 要約
GBA上で個別ピクセルを描く方法を、最もシンプルな「ビットマップ背景（Mode 3）」から、既存ライブラリの活用、そして複雑な「スプライト（タイル）への直接描画」まで段階的に解説します。

## この記事を読むべき理由
レトロゲーム開発の入門にも最適なトピックで、GBAの限られたハード資源を理解すると描画アルゴリズムやメモリ操作の基礎力が身につきます。日本のインディー/レトロ好きエンジニアにも応用できる知識です。

## 詳細解説
- Mode 3（ビットマップ背景）
  - 画面全体が240×160の直接色ビットマップになるため、各ピクセルに直接アクセス可能。VRAMの先頭アドレスを基準にオフセット指定で任意のピクセルを書き換えられる。
  - 色は15ビット（R5:G5:B5）の格納形式で、各チャネル5ビット。例えば純赤は下位5ビットを立てた値（0x1F相当）になる。  
  - 実装上はループでピクセルを書くだけで線や矩形が描けるため、試作やデバッグに最適。

- Toncライブラリ活用
  - libtoncはm3_fill / m3_rect / m3_line / m3_plotといった高水準関数を提供。モード3での基本描画を簡潔に実装でき、学習コストを下げる。

- スプライト（タイル）に描く難しさ
  - GBAのスプライトは「タイル（8×8など）」と「パレット」で構成。1ピクセルはパレットインデックス（0-15）を参照する。
  - タイル内部表現は1バイトに2ピクセル（ニブル2個）を詰める形式。さらにVRAMは16ビット単位での書き込みが必須のため、実際には「4ピクセル単位（16ビット）」で操作する必要がある。  
  - 結果として「1ピクセルだけ変えたい」でも、対象の半バイト位置を算出して、その属する16ビットワードを読み／マスク／書き換える処理が必要になる。
  - 一般的な実装は、タイル先頭ポインタに対して行(y)ごとにオフセット（y*2）を足し、x座標に応じてどのワード（x/4）を選ぶ、さらにその中で何ビットシフトするか((x%4)*4)を計算してマスク・挿入する、という流れになる。
  - こうした低レベルな手続きは関数化（setPixel相当）して隠蔽すると扱いやすい。

## 実践ポイント
- まずはMode 3で動かしてみる：画面全体が直接書き換え可能で学習用に最速。
- libtoncを使えば矩形・線・点は簡単に実装できる。プロトタイプはライブラリ頼みでOK。
- スプライトへ直接描く場合は「2ピクセル/バイト」「16ビット単位での書き込み」という制約を常に意識する。必ずマスクとシフトで既存データを壊さないようにすること。
- 実機でテストする前にエミュレータ＋devkitPro/toncの環境で検証。カラー変換ツール（RGB→GBA 15bit）を用意しておくと便利。
- 日本のレトロゲーム制作コミュニティや、GBA用のアセット変換ツール（png→タイル+パレット）を活用すると作業効率が上がる。
