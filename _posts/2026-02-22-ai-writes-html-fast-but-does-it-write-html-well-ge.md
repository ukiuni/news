# AI writes HTML fast but does it write HTML well, genuinely not sure anymore - AIはHTMLを速く書くが、本当に良いHTMLを書いているのか、もう正直わからない
AIが生成するHTML、本当にそのまま使って大丈夫？現場視点で検証する短評

## 要約
AIはHTMLを高速に生成できるが、セマンティクス、アクセシビリティ、パフォーマンス、保守性など現場で重要な要素が満たされているとは限らない、という問題提起。

## この記事を読むべき理由
AI活用が進む今、生成結果を“そのまま受け入れる”と後で手戻りやリスクになる。特に日本の現場（モバイル中心、アクセシビリティ規格やSEOの重視）では評価基準が異なるため、判断軸を持つことが重要。

## 詳細解説
- スピードと品質は別物：AIはパターンに基づき素早くマークアップを書くが、意味的に適切なタグ選択（header/article/sectionなど）や文脈に応じた構造化が欠ける場合がある。
- アクセシビリティ（a11y）：ARIAの適用やキーボード操作、ラベル付けが漏れやすい。日本ではJIS X 8341-3準拠が求められる場面も多い。
- SEOと国際化：lang属性、メタ情報、構造化データの最適化、UTF-8や日本語コンテンツ特有の文字エンコーディング配慮が必要。
- レスポンシブとパフォーマンス：モバイル優先設計や画像の最適化、不要なDOMやインラインスタイルでパフォーマンスが悪化することがある。
- セキュリティ・信頼性：ユーザー生成コンテンツや外部データをそのまま挿入するとXSSなどの脆弱性源になり得る。
- 保守性・チーム運用：クラス命名、コンポーネント化、スタイル分離がされていないと、長期運用で技術負債に。

## 実践ポイント
- まずは「人の目でレビュー」：セマンティクス、ラベル、alt、langを必ず確認する。
- 自動検査を組み込む：W3C Validator、Lighthouse、axe-core、ESLint/Prettier をCIに追加。
- モバイルとJIS対応をチェック：モバイル表示とJIS X 8341-3準拠テストを行う。
- テンプレート化してAIは部品生成に限定：ページ一式ではなく、コンポーネントやスニペットを出力させ人が組み立てる。
- サニタイズとサンドボックス：外部データを埋め込む際は必ずサニタイズ・Content Security Policyを設定する。
- 学習材料にする：AI出力の癖を把握し、プロンプトや社内ガイドを改善する。

短い結論：AIは高速なアシスタントだが「検査と人の判断」を前提に使えば生産性が上がる。放置すると後でコストになる点に注意。
