# Linux kernel security work - Linuxカーネルのセキュリティ対応の舞台裏
Linuxカーネルの“公表しない”セキュリティ運用が意味するもの — 速攻パッチと独立したプロセスの実態

## 要約
Linuxのセキュリティチームは、脆弱性を公表するよりも「迅速に修正して公開する」ことを優先する。CVEの割当てや外部発表は別チームが担当し、ハードウェア関連問題だけは例外的に封印（エンボargo）を伴う運用になる。

## この記事を読むべき理由
日本のOSベンダー、組み込み機器メーカー、ディストリビューション運営者は、カーネルの修正フローと連携ルールを正しく理解しておかないと、脆弱性対応や法規制（例：EUの新規法）への対応で遅れを取る。報告者やセキュリティ担当者が実務で取るべき行動指針も明確になる。

## 詳細解説
- 役割の分離: カーネルの「セキュリティチーム」は報告の triage と修正の取りまとめを担い、公表やCVE割当ては別の「CVEチーム」が行う。つまり修正の流通と脆弱性識別は別々のプロセスで動いている。
- 反応重視の運用: チームはプロアクティブ（防御的な仕組み作り）ではなく、報告に対するリアクティブな対応を主とする。一報を受けると該当サブシステムのメンテナを巻き込み、可能なら報告者が提供した修正を取り込む。
- 連絡のルール: 報告はカーネルのドキュメントに記載されたセキュリティ用エイリアスに「プレーンテキストのメール」で送る。HTMLやバイナリ添付、暗号メールは避ける（エイリアスは多数の個人に分配されるため暗号が機能しない）。
- 「告知しない」方針: 歴史的にカーネル側は「バグはバグ、特別扱いしない」を信条としており、修正は公開リポジトリにマージしておしまい。外部へのアナウンスは原則行わないため、事前に“プレ情報”を求める企業要請は却下されることが多い。
- 例外 — ハードウェア脆弱性: Spectre/Meltdownの教訓で、CPU／ファームウェアを跨ぐ問題は限定公開リストと暗号化ワークフローで協調し、短期的なエンバルゴを許容する運用がある。ただしこのワークフローは面倒で遅く、将来の法規や供給側の対応時間制約で変化が予想される。
- 独立性と守秘: セキュリティチームメンバーは個人として関与し、チーム内のやり取りを雇用主に事前に開示できない仕組みになっている。これが国境を越えた対応を可能にしている。

## 実践ポイント
- 脆弱性を報告するなら: カーネルのSecurityBugsドキュメントを確認し、セキュリティ用エイリアスにプレーンテキストで送る。再現手順と可能ならpatchを添える。バイナリやHTMLは送らない。
- CVEを期待しない: セキュリティチームに報告しても自動でCVEが割り当てられるわけではない。CVEは別プロセスで扱われ、修正が公開された後に進むことが多い。
- ベンダー／運用者向け: カーネル修正は素早く取り込む運用を設計する（stableブランチの追跡、自動パッチ適用、Livepatchの導入など）。「告知がない」前提で、ソースからの修正追跡体制を持つこと。
- 組み込み・ハードウェアメーカーへ: ハードウェアに関連する脆弱性はファームウェア・マイクロコードの連携が必要。サプライチェーンでの調整を早めに行い、EUのような新規法規に備えて連絡窓口と対応時間を明確にしておく。
- 研究者／報告者の心構え: セキュリティ修正を公開したい場合は、修正がマージされてから発表するか、CVEチームと調整する。公開前に脆弱性情報をばら撒かないこと。
