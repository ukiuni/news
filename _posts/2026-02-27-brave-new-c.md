# Brave new C# - 新しいC#の潮流とその光と影
C#の「増える文法」とあなたの開発現場が直面する選択

## 要約
近年のC#は機能追加が加速し、キーワードや構文が急増。便利な一方で言語の複雑化や互換性、API設計のトレードオフを招いている――という問題提起を整理した記事です。

## この記事を読むべき理由
日本の企業でも.NET／C#は業務系アプリ、Unity、IoTやゲーム開発で広く使われています。言語の進化は生産性向上につながる一方で、チームの学習負荷・ライブラリ設計・後方互換性に直結します。新機能の意義とリスクを把握しておけば、現場での導入判断やコード方針がブレません。

## 詳細解説
- 進化の速度: C#はここ数年でリリースサイクルが.NETに紐づき毎年新機能が追加。特にC#7→C#8以降、キーワード数と構文ノードが急増しています（公式ドキュメント／コンパイラ仕様やRoslynのSyntax.xmlの変更履歴で観察可）。
- 仕様と実装の乖離: ECMA-334などの公式仕様が最新版に追いつかないケースがあり、実際のコンパイラ実装（Roslyn）側で先行して機能が追加されることがある。
- 代表的な追加機能と問題点
  - ローカル関数／ラムダの機能拡張（属性・オプション引数など）やトップレベルステートメント（Main省略）は便利だが、導入でコードスタイルが多様化し初心者混乱の原因にも。
  - ref 系（refローカル・ref戻り値・ref readonly・ref struct・scoped ref など）はパフォーマンスや安全性向上に貢献するが、型システムやジェネリクスとの相互作用で複雑化。
  - record / record struct / record class / readonly record struct：等価性やボイラープレート削減に有効だが、価値型／参照型の違いや主コンストラクター（primary constructors）など学習項目が増える。
  - nint / nuint（native sized integers）と IntPtr/UIntPtr の取り扱い変化は、歴史的経緯でキーワードが冗長に見えるケースを生む。文脈キーワードの追加で識別子ルールが微妙になることも。
  - パターンマッチングの拡張（not/and/or 等）は表現力を高めるが、新演算子追加による冗長感も指摘される。
- ガバナンス: GitHubでの提案・投票プロセスによりユーザー要求が取り込まれる一方、賛否が分かれる提案も多く、言語設計のバランスは常に議論中。
- 実務的影響: ライブラリ作者はAPISurfaceの肥大化を意識する必要がある。新構文により既存ユーザーの理解コストが上がる可能性あり。

## 実践ポイント
- 新機能は「使える」「使うべき」の二段階で評価する（実行時利益 vs 学習・メンテコスト）。
- プロジェクトの言語バージョンを明示し、チーム内コーディングルールで採用可否を決める（csprojのLangVersion設定）。
- Visual Studio の新規作成オプション（例：トップレベルステートメントの無効化）や分析ツールを活用して一貫性を保つ。
- ライブラリ設計者はAPIのシンプルさを優先し、互換性と将来の拡張を見越した設計を行う。
- 新機能を試すときはターゲットフレームワークでのコンパイル互換性を必ず確認する（特にレガシー環境や外部公開APIの場合）。
- C#言語提案（Roslyn / csharplang）や公式ドキュメントの変更履歴を定期的に追うことで導入判断をしやすくする。

以上を踏まえ、C#の「便利さ」と「複雑さ」のバランスを現場でどう取るかが、今後ますます重要になります。
