# A Single Reason To Not Vibe Code - コードで「ゆるくやる」ことをためらうたった一つの理由
LLM時代に“プログラミング脳”が萎えるリスク — 失わないための実践法

## 要約
AIやツールに頼って「ゆるくコードを書く（vibe code）」と、使わなくなった認知能力が萎えて将来の学習や別作業に悪影響を及ぼす可能性がある、という警鐘。

## この記事を読むべき理由
日本でもAI補助の普及で手元のコード作業が楽になる一方、基礎的な思考力や時間変化を扱う「プログラミング的思考」が衰えると、新しい技術や設計課題に対応しづらくなる恐れがあるためです。

## 詳細解説
- 「萎縮（atrophy）」は使われない器官や能力が衰える現象。筋肉と同様、脳の特定能力も使わなければ弱まる可能性がある。  
- 著者はプログラマが持つ「時間的論理（temporal logic）」的な思考を例に挙げる。これはシステムが時間とともにどう振る舞うかを思考する力で、アルゴリズム設計や状態遷移、非同期処理の理解に直結する。  
- ロンドンの“Black Cab”ドライバーは膨大な道順を暗記する訓練で海馬が大きくなる研究があり、特定能力の鍛錬が他領域での認知にも効くことを示唆する。逆に言えば、ある技能をAIに任せ常に使わないと、その能力が弱まるリスクがある。  
- 大言語モデル（LLM）や自動生成ツールは短期的生産性を上げるが、同時に「考える回数」を減らし、深い設計判断や直観的なコードの読み書きを鈍らせる可能性がある。長期的に見て、新技術習得や複雑問題への対処能力が低下する危険がある。

## 実践ポイント
- 毎週「手で設計する」時間を確保する：ホワイトボードや紙でアーキテクチャや状態遷移を描く。  
- コードカタ（短い練習問題）を継続：アルゴリズムや非同期処理の練習を意図的に行う。  
- 教える／説明する機会を作る：人に説明すると理解が深まる。社内勉強会やブログが有効。  
- ツールは補助に留める：LLMはドラフト作成やアイデア出しに使い、最終設計や重要判断は自分で検証する習慣をつける。  
- 新しい課題に意図的に挑む：既知の枠を超えるプロジェクトを選び、脳の“筋肉”を鍛え続ける。

短期的な効率と長期的な技能維持のバランスを意識して、AI時代の「使う／使わない」を自分で設計しましょう。
