# Spec-driven development doesn't work if you're too confused to write the spec - 仕様駆動開発は、仕様を書けるほど混乱していなければ機能しない
驚くほど正直な指摘：仕様を書く前にチームが混乱しているなら「仕様駆動開発」は逆効果になる

## 要約
仕様駆動（spec-driven）開発は理想ではあるが、そもそも仕様を書けないくらい要件や設計が不明瞭なら、仕様作成に時間を費やしても誤った合意が生まれ、結果的に手戻りや技術的負債が増える。

## この記事を読むべき理由
多くの日本のチームもドキュメント重視や「合意済みの仕様で実装する」流儀を採るが、仕様を書く前の混乱を放置するとプロジェクトが停滞する。特に要件不確実性が高い新機能やドメイン未経験の領域では実務的な対処法が必要になる。

## 詳細解説
- 問題の本質  
  仕様駆動開発は「仕様＝合意された真実」を前提に動く手法。仕様が明確であれば実装・テスト・レビューが効率化する。しかし、仕様を書く段階で要件が曖昧、UXや非機能要件が未定、利害関係者の期待が揺れる場合、仕様作成そのものが推測や仮定に依存してしまう。結果、仕様が正しい保証はなく、実装後に仕様を後追いで修正する羽目になる。

- なぜ発生するか  
  1) ドメイン知識不足：チームがユーザーや業務フローを十分理解していない。  
  2) ステークホルダー間のズレ：期待値が揃っていない。  
  3) ドキュメント文化の誤用：完璧な仕様を求めすぎて探索（探索的プロトタイプ）を避ける。  
  4) 言語・文化の壁：グローバル設計資料をそのまま受け取ると日本市場の要件が抜けることがある。

- 技術的観点  
  仕様が曖昧だとAPI契約（OpenAPI/Swagger）、型定義（TypeScript、protobuf）、受け入れテスト（BDD）などが誤った前提で作られる。これにより契約テストや自動生成コードが信頼できなくなる。仕様は「静的な文書」ではなく「実験と合意のループ」で更新されるべきだ。

## 実践ポイント
- 仕様を書く前に小さく試す：短いスパイク実装やプロトタイプで不明点を洗い出す。  
- 具体的なユースケースと例を最初に作る：典型的な入力→期待出力を3つ程度用意して合意する。  
- インクリメンタル仕様化：全面仕様を一度に作らず、MVP範囲の契約（API/型）から始める。  
- ステークホルダーと共に「読みやすいドキュメント」を作る：図、シーケンス図、受け入れシナリオを重視。  
- 自動化で仕様の信頼性を保つ：契約テスト（consumer-driven contract）、そして失敗しないCIパイプライン。  
- 言語／市場差分を明示する：海外記事や仕様をそのまま使わず、日本特有の要件（法規、決済、業務フロー）をチェックリスト化。  
- 書く人を定め、時間を区切る：仕様作成は合意形成の場であり、書く担当とレビュー会を短期間で回す。

以上を実行すれば、仕様駆動の利点（明確な契約、テスト自動化、開発効率）を失わずに、混乱による無駄な手戻りを減らせます。
