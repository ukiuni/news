# java sucks (2000) - 「Javaはクソだ」（2000）
「2000年の毒舌レビューが告げる、今でも刺さるJavaの本当の問題点」

## 要約
1997–2000年にJamie Zawinskiが発表した辛辣な批評を日本語で整理。言語設計、ランタイム、標準ライブラリ、セキュリティを分けて問題点を指摘しつつ、当時の制約が現場に与える実務的影響を解説する。

## この記事を読むべき理由
古い記事だが、言語設計とプラットフォームの違いを見分ける視点、性能・移植性・API設計への根本的な疑問は今の日本の大規模システムや組み込み・レガシー運用でも役立つから。

## 詳細解説
- Javaという名前に四つの別物が混在している点（言語／クラスライブラリ／VM／セキュリティモデル）をまず区別すべきと主張。混同が採用判断や改善の障害になった。
- 言語面の不満：
  - クロージャや関数ポインタ、マクロ、ローカル関数の欠如による表現力不足（当時の匿名クラスは代替に不十分）。
  - プリミティブとオブジェクトの乖離（数値がオブジェクトでないこと）や配列型ごとの扱いの違いでアルゴリズムの汎用化が難しい。
  - staticメソッドが「本当のクラスメソッド」でない（オーバーライドできない）ことで継承設計が制約される。
  - finalization（ファイナライザ）や弱参照の欠如が、キャッシュや分散オブジェクト管理を難しくしていた。
  - コンパイル時最適化や低レイヤのメモリ管理へアクセスできないことで、特定用途で性能が出しづらい。
- ライブラリ／実装面の指摘：
  - Stringや配列の設計（不変性、内部表現の選択肢がない）によるメモリ／性能の無駄。
  - インタフェースは多重継承回避のパッチに見える、言語としての一貫性に疑問。
  - 「Write once, run anywhere」は理念だが当時は実情と乖離。ネイティブ最適化やプラットフォーム差分が残る。
- 著者自身も認めるが、多くの指摘は当時（1997–2000）のJDK実装や仕様に依存したもので、後のJava（enum、autoboxing、ジェネリクス、ラムダ式や改善されたGC/リファレンスAPIなど）で解消・改善された点がある。

## 実践ポイント
- 「Java＝言語／VM／ライブラリ／セキュリティ」を分けて評価する習慣を持つ（採用・移行判断で有効）。
- レガシーや組み込みで性能が必要なら、JVMの実装差・JITプロファイラを測定して選定する。
- 古い批評の指摘は設計の教訓になる：APIの不変性・オブジェクト/プリミティブ境界・キャッシュ設計は今でも重要。
- 現行のJDK機能を活用する（例：java.lang.refの弱参照、enum、自動ボクシング、ラムダ、Stringの内部最適化）ことで多くの問題は解消可能。
- 移植性を重視するなら実機テストを必須化し、ネイティブコードやビルドの選択肢（GraalVM等）も検討する。

（参考：原文は1997–2000年の視点による強烈な批評。現代のJavaは多くの改善を受けているが、設計上のトレードオフや歴史的課題の理解は設計判断に役立つ。）
