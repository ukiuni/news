# [Any application that can be written in a system language, eventually will be] - [システム言語で書けるアプリは最終的にシステム言語で書かれる]

サーバで10倍速く、電気代も激減？LLMが後押しする「Rust/Go時代」到来の現実味

## 要約
LLM（大規模言語モデル）とコスト最適化の流れで、性能・効率が重要な新規サービスはRustやGoなどのシステム言語で書かれる傾向が強まっている。特にサーバレス環境では実行時間とメモリが直接コストに響くため、言語選択の経済性が見直されている。

## この記事を読むべき理由
日本でもクラウド利用増加・省エネ・SaaSの薄利化が進む中、言語選定の「感覚的な好み」ではなく「コストとAIツール活用」で技術戦略を変える必要が出てきたため。新規プロダクトやマイクロサービス設計に直結する判断材料を得られます。

## 詳細解説
- 背景：Atwoodの法則「JavaScriptが普及した理由」は“どこでも動く＝アクセシビリティ”だった。だが今は「実行コスト」が同等かそれ以上に重要になっている。  
- 経済的要因：研究ではPythonがRustより数十倍エネルギー非効率になるケースが報告されており、サーバレス課金（実行時間×メモリ）ではその差がそのまま「ランタイム税」になる。結果として同じハードでGo/Rustなら数倍〜10倍近いスループットを実現できる。  
- AIの影響：従来の障壁は「学習コスト」や「借用チェッカー」などの難易度だったが、LLMがコード生成と反復修正（コンパイラエラー対応）を得意とするため、システム言語の採用障壁が大きく下がっている。コンパイラが事前に多くの不具合を潰すため、厳格言語で「ゆるいコーディング（vibe coding）」が安全になる逆説も発生。  
- 実務的分離：AIや実験的モデルの「頭脳」はPython等のエコシステムに残しつつ、API サーバやデータ取り込み、サイドカーなど「筋肉」部分をRust/Goで実装するハイブリッド設計が現実的。  
- 注意点：既存システム全量の書き換えはコストとリスクが高く「Rewrite Trap」に陥りやすい。あくまでグリーンフィールドや新サービス・高負荷パスに優先度を置くべき。

日本での意味合い（簡潔に）：
- AWS/GCP活用が進む国内スタートアップやエンタープライズではクラウド請求とカーボンフットプリントの圧縮が経営課題。効率化は技術的優位性に直結する。  
- 人材面ではRust/Goの採用が難しい局面でも、LLMを使った実装支援で採用ハードルが下がる可能性がある。

## 実践ポイント
- 新規マイクロサービスは初期からRust/Goを候補に入れる（特に高スループット・低レイテンシが必要な箇所）。  
- まずは「筋肉」コンポーネント（API、ETL、サイドカー）から段階的に移行し、既存の「頭脳」はPythonを併用するハイブリッド構成を検討。  
- LLMを使ってコード生成→ローカルでコンパイルとCIの厳格チェックを回すワークフローを整備する（コンパイラのエラーは安全フィルタになる）。  
- コスト試算とベンチマークを必ず実施する（実行時間×メモリ＝直接コスト）。AWS Lambda等の課金モデルで差が出やすい。  
- 大規模書き換えは避け、サイドカーや新エンドポイントでの採用から始める。  
- 省エネ・コスト削減の効果を数値で経営に示せるようメトリクス（スループット、レスポンスタイム、電力/請求）を記録する。
