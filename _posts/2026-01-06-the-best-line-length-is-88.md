# The Best Line Length - 最適な行長
88文字が“ちょうどいい”理由：人間の目・歴史・ツールが導く行幅の最適解

## 要約
ブラック（Black）が採用する「88文字」が示すのは偶然のラッキー数字ではなく、視覚生理、印刷史、エディタ習慣、差分ツールの実用性を勘案した現実的な折衷案である。

## この記事を読むべき理由
コードレビューやCI、プレゼン資料、複数人開発で「行長」が揃っていないと無駄な差分や読みづらさが生じる。日本の開発現場でもガイドライン化しやすい妥協点として、なぜ88文字という数値が意味を持つのかを理解しておくと実務で速く合意できる。

## 詳細解説
- 歴史的背景：80文字幅はパンチカードやタイプライタ、終には端末幅に由来する。新聞や印刷物では更に狭い列幅が使われ、可読性向上のための「列」の概念が古くからあった。  
- 生理学（saccades）：人間の眼は単語ごとに滑らかに追従するのではなく、短い跳躍（saccades）で読み進める。次行の開始位置が視界の周辺に入っていることがターゲティングの負担を下げ、結果として行長に上限が生まれる。  
- CPL（Characters Per Line）：研究では最適なCPLはおよそ45〜95の範囲とされ、コードはインデントや空白が多いため実効幅が短くなる。たとえばインデント8〜12文字を考慮すると、88文字は実質的に約76〜80文字相当の可読領域になる。  
- 実務的制約：ソースはdiffやツールで扱うため「ソフトラップだけで済ます」は現実的でない。差分が大きくなるとレビュー効率が落ちる。これらを踏まえ、Black が既定で 88 を選んだのは合理的な妥協と言える。  
- 画面とレイアウト：ウルトラワイド画面は有効だが、読みやすさのためにエディタ側で列幅制限や分割ペインを活用するのが現実的。多列表示や横分割を意識すると、空白を活かした作業空間が得られる。

## 実践ポイント
- チーム規約として明記する（例：Black の既定 88 を採用）。CI で整形チェックを回すとブレが減る。  
- Black を pyproject.toml で設定する例：

```toml
# toml
[tool.black]
line-length = 88
```

- VS Code の視覚補助設定（ルーラーとワードラップ）：

```json
// json
"editor.rulers": [88],
"editor.wordWrap": "wordWrapColumn",
"editor.wordWrapColumn": 88
```

- レビュー運用：差分を見やすくするためにハードラップ（行長制限）を維持する。大型表示器を活かすならエディタの分割ペインや隣接バッファで関連コードを並べて配置する。  
- プレゼン／Web表示：ドキュメントやスライドでも行幅を意識しておくと可読性が向上する。CSS の multi-column を検討する余地あり。

結論：厳密な「正解」はないが、視覚的制約とツール運用の折衷として 88文字は実務的で説得力のある選択肢。チームで統一してしまえばレビュー効率と可読性が同時に上がる。
