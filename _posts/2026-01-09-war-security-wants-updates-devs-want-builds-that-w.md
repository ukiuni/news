# War: Security Wants Updates, Devs Want Builds That Work - セキュリティは更新を求め、開発は動くビルドを求める
「毎週パッチを当ててほしい」と「今ある機能を壊さないでほしい」。チーム間の“戦争”を止める実務ガイド

## 要約
セキュリティチームは脆弱性対応の迅速さを最優先にする一方、開発チームは安定したビルドとリリースの継続を重視する。両者の対立は多くの現場で再現されるが、プロセスとツールで折り合いをつけられる。

## この記事を読むべき理由
日本の企業でも、コンプライアンスや顧客データ保護の観点から迅速なセキュリティ対応が求められる一方で、サービス停止や回帰バグは売上／信頼に直結する。本稿は、現場で両方を満たす実践的な方法を初心者向けに整理する。

## 詳細解説
背景
- セキュリティ側：CVEs（脆弱性情報）や外部監査に基づき、依存ライブラリやライフサイクル中のコンポーネントを早く更新したい。放置は法的リスクや侵害リスクを高める。
- 開発側：頻繁な依存更新や差し替えはビルド崩壊や非互換を招き、デプロイの信頼を下げる。特にテストが不十分だと回帰が発生する。

技術的ポイント（簡潔に）
- 依存管理の自動化：DependabotやRenovateのようなツールでPRを自動作成し、更新を小さな単位でステージングする。
- CI/CDとテストの強化：ユニット・統合・エンドツーエンドテストをCIで自動実行し、更新PRで即検証する。フレークテストを減らすことが重要。
- セマンティックバージョニングと互換性ポリシー：メジャー更新は慎重に扱い、マイナー／パッチは自動適用を優先するルールを決める。
- カナリア／ローリングデプロイ：最小限のユーザにまず配信し、問題なければ段階展開することで、迅速なパッチを安全に適用できる。
- フィーチャーフラグとフォールバック：新しい挙動はフラグで制御し、問題があればすぐに切り戻せるようにする。
- セキュリティテストの組み込み：SCA（ソフトウェア構成解析）、SAST（静的解析）、DAST（動的解析）をCIに組み込む。
- セキュリティ／開発の優先順位付け：CVSSスコアや公開エクスプロイトの有無で対応優先度を決定するSLAを設定する。

組織運用上のポイント
- 共通のSLAを定める：例）Criticalは72時間以内、Highは7日以内に対応検討を開始。
- セキュリティチャンピオン制度：各開発チームに1名の窓口を置き、調整をスムーズにする。
- クロスファンクショナルなデイリースタンドアップや週次レビューで情報共有。
- 変更の可視化とドキュメント化：変更ログとロールバック手順を必ず記録する。

## 実践ポイント
- まずルールを決める：Critical/High/Medium/Lowの定義と対応SLAをチームで合意する。
- 自動化を段階導入：Dependabot→CIテスト→ステージング→カナリア。小さく回して信頼を築く。
- テストカバレッジより「信頼できるテスト」を目指す：フレークを減らし、失敗の原因を素早く追えるようにする。
- フィーチャーフラグを導入して、更新リスクを限定的にする（LaunchDarklyやオープンソースの代替も選択肢）。
- 日本向けの配慮：個人情報保護（APPI）や金融・医療系の規制に対応する優先リストを作る。
- メトリクスを追う：ビルド成功率、平均パッチ適用時間（time-to-patch）、ロールバック率などを公開して改善サイクルを回す。

短い結論
対立は必ずしも悪ではない。異なる視点があるからこそリスクと品質のバランスを取れる。ルール化と自動化、そして小さなデプロイ戦略で「セキュリティの迅速さ」と「開発の安定性」は両立できる。
