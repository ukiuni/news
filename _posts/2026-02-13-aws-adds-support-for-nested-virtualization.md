# AWS Adds support for nested virtualization - AWSがネステッド仮想化に対応
AWSで「仮想の中の仮想」を実行できる！非ベアメタルEC2でネステッド仮想化が利用可能に

## 要約
AWS SDK for Go v2 の EC2モジュール（v1.288.0）で、仮想（非ベアメタル）EC2インスタンス上でネステッド仮想化（nested virtualization）を起動できる機能がリリースされました。これにより、EC2上でさらに仮想マシンを実行するユースケースが広がります。

## この記事を読むべき理由
開発・テスト、CI、仮想化ソフトの検証や教育用途で、従来はベアメタルに限定されがちだったネステッド仮想化がより低コストかつ柔軟に使えるようになります。日本の企業でもオンプレ寄せのワークロードからクラウド移行や、VMイメージの継続的テストが容易になります。

## 詳細解説
- 何が変わったか：公式CHANGELOGにて service/ec2 の v1.288.0 に「Launching nested virtualization」の記載。要するに、非ベアメタル（仮想化されたEC2インスタンス）上でさらにゲストVMを起動できるようになった、という機能追加です。
- ネステッド仮想化とは：ホスト仮想化（EC2上のハイパーバイザ）内で、さらに別のハイパーバイザを走らせ、その上でゲストVMを動かすこと。これによりハイパーバイザ開発、仮想マシンのイメージ検証、複雑な仮想ネットワーク構成のテストが可能になります。
- なぜ「非ベアメタル」で重要か：従来は高コストなベアメタルや専用インスタンスが必要なことが多かったため、開発環境や小規模な検証では導入障壁が高かった。本対応で一般的な仮想EC2インスタンスでも試せる可能性が高まり、コストと運用の柔軟性が向上します。
- 注意点：実際に利用するには対象のインスタンスタイプやAMI、ハイパーバイザ設定（CPUの仮想化フラグ有効化など）を確認する必要があります。公式ドキュメントやリリースノートで対応状況をチェックしてください。

## 実践ポイント
- SDKアップデート（Goの場合）:
```bash
go get github.com/aws/aws-sdk-go-v2/service/ec2@v1.288.0
```
- まずは小さなインスタンスで動作検証：対応インスタンスタイプ、カーネル／AMI、仮想化フラグ（VT-x/AMD-V相当）の有無を確認してからネステッドVMを起動する。
- CI/CDやテスト環境での活用例：VMイメージの自動テスト、マルチレイヤ仮想環境の検証、仮想化ソフトの回帰テストなどでコスト削減と環境再現性向上に役立ちます。
- 次のステップ：AWSの公式ドキュメントと対応インスタンスタイプを確認し、社内のセキュリティ要件やライセンス要件を満たすか検討してから導入してください。
