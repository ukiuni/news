# LLM=True - LLM=true（LLM用の環境変数で「騒音」を消す提案）
LLM=trueでAIエージェントが見る「コンテキスト汚染」を減らし、トークンコスト・出力品質・環境負荷を同時に改善しよう、という話。

## 要約
AIコーディングエージェント（例：Claude Code）が扱うコンテキストはSTDOUTなどの「ノイズ」で急速に埋まる。ビルド出力や更新通知など無関係なログを環境変数やツール設定で抑えることで、トークン消費と誤動作を減らせるという提案。

## この記事を読むべき理由
日本でもモノレポ＋CI、生成AIを使った開発が増加中。トークン課金やLLMのコンテキスト制限はコストと開発効率に直結するため、実務レベルで有益な節約策と運用方針を知る価値がある。

## 詳細解説
- 問題：turbo等のビルドツールやライブラリが大量の冗長メッセージ（UPDATE AVAILABLE、各パッケージのビルドログ、カラーエスケープなど）をSTDOUTに吐き、LLMのコンテキストを汚染する。失敗時にtailで切って見る運用は、情報欠損→再実行のループを生む（犬が自分の尻尾を追うような現象）。
- 対策の考え方：
  - ツール側でログ出力を抑える（例：turboの設定で出力をerrors-onlyに）。
  - 環境変数で全体挙動を変える（NO_COLOR、CI=trueなど既存の慣習）。記事は「LLM=true」環境変数の導入を提案し、ライブラリ側が「LLMモード」を実装すればトークン削減に寄与すると主張する。
- 実例（抜粋）：
  - turbo.json（ビルド出力を抑制）
```json
{
  "tasks": {
    "build": {
      "outputLogs": "errors-only"
    }
  }
}
```
  - .claude/settings.json（エージェントセッションにスコープした環境変数）
```json
{
  "env": {
    "TURBO_NO_UPDATE_NOTIFIER": "1",
    "NO_COLOR": "1",
    "CI": "true"
  }
}
```
- 留意点：環境変数だけではすべてのツールを制御できない。各パッケージの実装次第で挙動は異なり、運用ルールとライブラリ側の対応（LLM=trueを解釈する実装）が必要。

## 実践ポイント
1. まずはビルドツール設定で冗長ログを抑える（例：turboのoutputLogsをerrors-only）。
2. エージェント専用の環境設定にNO_COLORやTURBO_NO_UPDATE_NOTIFIER、CI=trueを追加（.claude/settings.json等でスコープ管理）。
3. 失敗時の情報欠損を避けるため、ログをファイル出力して要約だけをエージェントに渡す運用を検討する（例：npm run build > build.log 2>&1; tail -n 100 build.log）。
4. チームで「LLM=true」等の運用フラグ導入を検討し、ライブラリや社内ツールに対応を働きかける。小さなトークン削減がスケールすると意味を持つ。
5. 将来的議論：エージェント主導の開発が進むなら、HUMAN=true等の逆フラグ運用も検討課題。

短く言えば：ツール設定＋環境変数で「見せる情報」を制御し、AIエージェントのコンテキストを軽く保つだけで、コスト・品質・環境の三方に良い影響が期待できる、という提案です。
