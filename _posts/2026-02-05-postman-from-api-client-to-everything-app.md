# Postman: From API Client to “Everything App” - Postman：APIクライアントから“オールインワン”アプリへ
Postmanが“全部入りアプリ”へ変貌 — 使い勝手は便利？それとも危険な鍵か？

## 要約
Postmanは2026年3月のアップデートで、従来のAPIテスト／ドキュメントツールからAIエージェントやホスティング、内部ターミナルまで含む「オールインワン」プラットフォームへ大きく舵を切った。便利さの裏にベンダーロックインやパフォーマンス問題などのリスクがある。

## この記事を読むべき理由
日本の企業はシステムの安定性やデータ管理を重視するため、外部ツールへの依存が長期的コストや法令対応（データ所在など）に直結する。Postmanの変化は小〜中規模チームから大手SIまで影響を与える可能性が高い。

## 詳細解説
- 新機能要点
  - AI Agent Builders：コードを書かずに視覚ブロックでAIワークフローを作成。素早いプロトタイプ向けだが、ロジックの「所有権」は薄くなる。
  - API Portfolio：API・データソース・AIモデルの一元管理を目指し、会社の“ソース・オブ・トゥルース”にしようとする。
  - MCP Servers：Postmanが管理するクラウド上でAPIをホスティング。移行コストと依存リスクが発生。
  - Internal Terminal：ローカル環境ではなくアプリ内でコード実行を促進し、環境の分散を減らす一方でコントロールを手放す可能性がある。
- リスクと影響
  - 機能肥大（Feature Bloat）：エディタやモック、CI連携などを詰め込み、クライアントとしての軽快さが損なわれる。
  - 性能問題：メモリ使用量増加やUI遅延の報告がある。大規模コレクションで顕著。
  - UIの複雑化：単純なAPI検証が余計なクラウド機能に埋もれる。
  - 価格・ポリシー依存：無料枠縮小や料金変更でビジネスロジックが影響を受ける恐れ。
  - クラウドロックイン：MCPやAgentに依存すると、サービス終了や条件変更で大規模な移行が必要になり得る。
  - 拡張性の限界：ビジュアルエージェントは迅速だが、細かなカスタムロジックや最適化にはコードが有利。

## 実践ポイント
- まず社内でPostmanの利用状況（Agent・ホスティング・エクスポート可否）を棚卸しする。
- コアビジネスロジックや機密データをAgent／MCPに直接置かない。
- テストやAPI検証はcurl + jqや軽量スクリプトで代替し、依存度を下げる。
- エクスポート／バックアップ手順を確立し、万一の移行に備える。
- 新機能導入前にTCOとデータ所在、セキュリティを評価する（特に法令順守が必要なケース）。
- 既に依存度が高い場合は「エグジット計画」を作り、段階的に代替手段を整備する。

短期的な生産性向上と長期的なコントロール放棄の天秤を意識して、導入判断を行ってください。
