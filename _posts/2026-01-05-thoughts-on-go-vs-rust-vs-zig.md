# Thoughts on Go vs. Rust vs. Zig - Go vs Rust vs Zig に関する考察
Go、Rust、Zig――現場で「どれを採るべきか」を価値観で見抜く

## 要約
Goは「最小限かつ実用的」でチーム開発に強く、Rustは「安全と高速さ」を追求するために概念が濃密、Zigは「手綱を渡す」ことでデータ指向・明示的制御を好む設計。各言語は機能の差ではなく、採る価値観が違う。

## この記事を読むべき理由
日本のプロダクト現場では保守性、ビルド時間、採用しやすさ、組込みやレガシーとの親和性が重要。言語の「哲学」を理解すれば、技術選定で失敗しにくくなる。

## 詳細解説
- Go（ミニマリズム）
  - 設計方針: 言語機能を絞り、学習コストと読みやすさを優先。ガベージコレクションありでランタイムが手厚い。
  - 実務的利点: 安定した標準ライブラリ、簡潔な並行処理（goroutine/channel）、読みやすさによるチーム間の生産性向上。
  - 特徴例: スライスが可変長且つ実用的に振る舞うため、メモリ位置の扱いを言語が多く担う。
  - 向き先: Webサービス、社内基盤、運用ツール。組織での導入コストが低い。

- Rust（概念の密度）
  - 設計方針: コンパイル時に安全性（メモリ安全＋未定義動作回避）と高性能を保証するため、型システムやトレイト群が豊富。
  - 実務的利点: UB（未定義動作）を減らしつつゼロコスト抽象を実現。ライブラリ互換性と依存管理が成熟しつつある。
  - トレードオフ: 学習曲線が急で、設計表現（ライフタイム、所有権、多数の抽象概念）を理解する必要がある。
  - 向き先: セキュリティ敏感なシステム、パフォーマンスクリティカルなネットワーク/DB/エンジン、ライブラリ開発。

- Zig（明示的な制御とデータ指向）
  - 設計方針: 手動メモリ管理と明示的なアロケータ設計で、開発者に低レベルの制御を委ねる。OOP的機能を絞りデータ指向を奨励。
  - 実務的利点: 柔軟に割り当て戦略を選べるため、イベントループ単位で大塊を割り当てるような設計に向く。実行時チェック（illegal behavior）とビルド時の複数のリリースモードでバランス可能。
  - トレードオフ: 標準ライブラリの未熟さやまだ0.x台の安定度。学習は容易でも、手動管理の責任が増す。
  - 向き先: ツールング、ゲーム、組込み、特殊なメモリ戦略を求めるケース。

日本市場との関連（抜粋）
- 多くの日本企業はレガシー資産や短い開発サイクルを重視するため、Goの採用効果が出やすい。運用チームが多い組織では「読みやすさ＝保守性」。
- 金融や通信、ミッションクリティカル分野ではRustの「コンパイル時保証」が魅力。安全性要求が高いプロジェクトで採用が増加中。
- ゲーム会社や組込み系の一部スタートアップでは、Zigの明示的メモリ制御やデータ指向が刺さる場面がある。ただしエコシステム成熟度は要確認。

## 実践ポイント
- 採用判断は「機能の比較」ではなく「チームとプロダクトが重視する価値観」で行う。
  - 安価で早く回せるWeb APIや社内ツール → Goを第一候補に。
  - セキュリティ・パフォーマンス重視のコアライブラリ → Rustを検討。
  - 独自のメモリ戦略やデータ指向で低レイヤの制御が要る → Zigを試す（小さめのPoCから）。
- まずは小さな実案件で試す：
  1. Goで簡単なCRUD APIを1週間で実装して運用性を評価する。  
  2. Rustで既存Cライブラリの一部を安全に書き換えるPoCを作る。  
  3. Zigでアロケータを明示して短いイベントループ実装を作る。
- 採用時チェックリスト：ビルド/CIのコスト、チームの学習時間、既存資産との相性、標準ライブラリとエコシステムの成熟度。
- 参考学習順: Go Tour、The Rust Book、Zigのリポジトリと簡単なサンプル（Zigはソース読みが学習の近道）。

短く言えば、言語はツール以上に「価値観」を表す。チームの文化と目標に合う言語を、まずは小さく試してから本格導入するのが安全な選び方だ。
