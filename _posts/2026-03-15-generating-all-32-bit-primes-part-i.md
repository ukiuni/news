# Generating All 32-Bit Primes (Part I) - 32ビット素数を全列挙する（前編）
1.3GBで32ビット全素数を「秒単位」で作るには？C実装から学ぶ高速化のツボ

## 要約
Cで32ビット非負整数の全素数をバイナリ（小エンディアンの uint32_t）で出力する実装を比較し、逐次試し割・ホイール法・エラトステネスのふるいの性能差と実装上の工夫を解説する。

## この記事を読むべき理由
日本の開発者にとって、巨大範囲の素数列挙はアルゴリズム設計・メモリ対速度のトレードオフ・I/O処理の実務的教訓が凝縮された良い教材だから。実運用で使える高速手法や実装上の注意点を短時間で掴める。

## 詳細解説
- 目的と成果物  
  - 目標はすべての 32-bit unsigned 整数（$0 \ldots 2^{32}-1$）の素数を、4バイトごとに小エンディアンで PRIMES ファイルへ並べること。記事中の正しいSHA-256も提示されている。

- 試し割（Trial Division）  
  - 各候補 $n$ に対して $\sqrt{n}$ までの既知の素数で割り切れるかを調べる単純法。素数列を増やしながら順に検査する実装で約24分20秒（著者環境）。計算量は候補ごとに $\pi(\sqrt{n})$ 回の割り算（$\pi(x)$ は素数計数関数）。素数定理により $$\pi(x)\sim\frac{x}{\ln x}$$ で増えるため実用には遅い。

- ホイール法（Wheel Factorization）  
  - 最初の $k$ 個の素数の積（主に $2\times3\times5=30$ のような primorial）を周期として、そのうち小さな素因子で割り切れる剰余クラスを省く手法。候補数は減るが、除外されるのは比較的「検査が速い」数なので試し割の改良効果は限定的。著者の5素数ホイール実装で約23分30秒とわずかな改善。

- エラトステネスのふるい（Sieve of Eratosthenes）  
  - 配列（ここではビット配列）上で合成数を消していく方法。$p$ の倍数を順に消す「$p$-sifting」をすべての $p\le\sqrt{N}$ に対して適用する。理論的時間はおおむね $$O(N\log\log N)$$。ビット格納でメモリを削る工夫が重要（1ビット/数で格納すると $2^{32}$ 個だと約537MB）。著者実装は約32秒で実行（ただし出力ファイル作成を含む）。ビット配列(+全素数格納配列)で総メモリ約1.3GBを使用。

- 実装上の注意点  
  - 整数オーバーフロー対策（例：$p*p$ 比較の前に上限チェック）。  
  - ビット配列操作の効率（バイト単位・ワード単位でのマスク操作）。  
  - 結果を書き出す際のバイナリ表現とI/Oバッファリング。  
  - さらに高速化する既存ライブラリ（例：Kim Walisch の primesieve）は、メモリ/実装最適化で著者の純C実装より桁違いに速い（記事中で 0.061s と報告）。

## 実践ポイント
- まず学ぶなら：小さめの上限でビット配列版エラトステネスを実装して動作を理解する。  
- 32ビット全列挙なら：単体実装より segmented sieve や primesieve を使う（高速・メモリ効率良）。  
- ファイル出力：uint32_t を小エンディアンで fwrite し、SHA-256 で検証する習慣を付ける。  
- メモリ節約：全数ビット配列＋出力一時バッファで 1GB 前後に収める設計を検討する。  
- 注意点：$p*p$ の比較や乗算でのオーバーフロー回避、I/Oのバッファ化を忘れないこと。

参考実装（出力部分の一例）:
```c
#include <stdint.h>
#include <stdio.h>

void write_primes_bin(const uint32_t *primes, size_t count) {
    FILE *f = fopen("PRIMES","wb");
    fwrite(primes, sizeof(uint32_t), count, f); // 小エンディアン環境を想定
    fclose(f);
}
```

短くまとめると、学習目的なら試し割→ホイールでアルゴリズムの振る舞いを理解し、実運用ではビット配列ベースのエラトステネス（あるいは既成の最適化ライブラリ）を選ぶのが正攻法。
