# Claude Code is your customer - Claude Codeはあなたの顧客だ

AIエージェントが“顧客”になる時代――あなたのAPIはその選ばれる基準を満たしていますか？

## 要約
最新の潮流は「AIエージェント（例：Claude Code）がサービスを評価・導入する」こと。これに対応するにはAPIとドキュメントを“エージェント向け”に設計する必要がある、という主張です。

## この記事を読むべき理由
AI導入が進む日本の企業・開発者も、従来の「人間が使いやすければ良い」発想では競争に負けます。エージェントが自動で使えることが、採用の可否を決める新たな基準になりつつあるため、今から対策を取る価値が高いです。

## 詳細解説
- 背景：過去の「APIファースト」は人間開発者向けの合理化を目指していましたが、今は“エージェントが直接読むドキュメント”が第一のUIになりつつあります。  
- どこが変わるか：エージェントはランディングページやダッシュボードではなくAPIドキュメントを基に判断します。分かりにくい認証、曖昧なエラーメッセージ、コピー＆ペーストできない例は即座に機会損失になります。  
- 経済的インパクト：人間の切り替えコストが高かった従来とは違い、エージェントは数分〜数時間でベンダーを入れ替えます。スイッチングコストが下がることで「ダッシュボードの美しさ」や「ブランド」よりも「API品質」「可用性」「料金の機械判読性」が重要になります。  
- インフラとボリューム：エージェントは人間よりはるかに大量のAPI呼び出しを行うため、インフラや課金モデルも再設計が必要です（エージェントのデバッグで短時間にリトライが多発する等）。  
- 歴史的参照：Bezosの「APIを外部化前提で作れ」という考えは、今や“外部の開発者＝AIエージェント”に向けた設計として再評価されています。

## 実践ポイント
- APIを最初に作る：ダッシュボードは同じAPI上に構築する。APIで自分のUIが再現できなければ不十分。  
- エージェントでテスト：Claude Code等にドキュメントを与え、実際に統合できるか試す。成功率は新しいKPIに。  
- ドキュメントは即実行可能に：コピー＆ペーストで動くサンプル、明確な認証手順、分かりやすいエラーメッセージを用意。  
- 料金は機械判読可能に：ドキュメント上でプログラム的に解釈できる価格表示を。席単位課金は見直す（ユースベース課金推奨）。  
- 再試行設計と冪等性：エージェントは再試行するので、APIは冪等であること。  
- エージェント分析を導入：ヒトとエージェントの利用を分離してモニタリング。短時間高頻度の挙動を正しく解釈する。  
- 「Get API Key」導線を明確に：エージェントは営業に連絡できない。セルフサービスでキー発行できることが重要。  
- 日本市場への配慮：エンタープライズ向けの契約文化やデータ保護に配慮しつつ、開発側はAPI公開の自動化とコンプライアンスを両立させる。

短期的には「APIとドキュメントの品質」が顧客獲得の最大要因になります。Claude Codeのようなエージェントを“顧客”として想定した設計に今すぐ着手してください。
