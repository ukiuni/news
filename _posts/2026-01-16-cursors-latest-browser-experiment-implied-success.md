# Cursor's latest "browser experiment" implied success without evidence - Cursorの最新「ブラウザ実験」は成功を示唆するが根拠がない
「『ブラウザを作った』という宣言の裏側：Cursorの“実験”は本当に動くのか？」

## 要約
Cursorは「長期間自律でコードを書くエージェントがウェブブラウザをほぼ一週間で生成し、100万行を超えた」と報告したが、元記事はその成果が再現可能で動作するという証拠を示していないと指摘している。

## この記事を読むべき理由
AIによる自律的コーディングは日本の開発現場でも話題です。実際に使える成果か、単なるデモ映像やスクリーンショットなのかを見抜く力は、採用や導入判断で重要になります。

## 詳細解説
元記事の主張ポイント（要点と問題点）
- Cursorのブログでは「エージェントが数週間にわたり協調し、ウェブブラウザをゼロから構築した」と記述。ソースコードはGitHubに公開（github.com/wilsonzlin/fastrender）されている。
- しかし検証結果として、リポジトリはコンパイルできず、ビルドエラーやワークフロー失敗が多数あると報告されている（例：'cargo build'で34件のエラー、複数のCI失敗、最近のPRは失敗するCIのままマージ）。
- スクリーンショットや短い動画はあるが、動作するコミットや再現手順、動作確認済みのタグ/リリースは提示されていない点が問題視されている。
- 元記事はこの状況を「AIが大量のコードを出力できるが、それが動くとは限らない」──すなわち「AIスロップ（AIが生成した低品質なコード）」と表現している。
- 技術的に重要なチェックポイントとして、著者は最低限「サポートされるツールチェーンでビルドできる」「簡単なHTMLをレンダリングできる」ことを期待しているが、それすら満たしていないとされる。

なぜ「コンパイルできること」が重要か
- ソフトウェアの最低限の健全性を示す指標で、CIパス／ビルド成功は再現性の担保。
- エージェントが出力したコードが実際に動かない場合、プロジェクトとしての価値や運用可能性は大きく下がる。

「数百エージェントが同時に作業した」主張について
- 並列作業や自動マージのスケールは興味深いが、コンフリクト解決やアーキテクチャの統一、テスト整備といった工程が伴わないと「意味ある進捗」にはならない。
- 元記事は並列性を示すが、その結果生じた成果物の品質や再現可能性に関する証拠を提示していない。

## 実践ポイント
すぐに試せる検証手順
- リポジトリをクローンして手元でビルド：`cargo build`／`cargo check` を実行してエラー数を確認する。
- GitHub Actionsの履歴をチェックして、成功したワークフローがあるか確認する。
- 明示的なタグ／リリース、もしくは「動くことを示す最小コミット」があるか探す。
- IssueやPRの履歴を見て、CIが失敗したままマージされていないかを確認する。

導入前にチェックすべきポイント（企業・開発チーム向け）
- デモは「再現可能であるか」：手順・環境・固定されたコミットが提示されているか。
- CIとテストの整備状況：テストが存在し、CIが安定して通っているか。
- ドキュメントとランタイム要件：実際に起動・動作させるための明確な手順があるか。
- 責任の所在：生成物の品質保証や保守方針がどうなっているか。

発表側に期待したい良い慣行
- 動作する最小限のリリース（タグ／バイナリ／コンテナ）を公開する。
- 再現手順とサンプルHTMLなどのデモ素材を添える。
- CIが通ることを公開基準にする（少なくともビルドが通ること）。

短い結論
AIエージェントによる大規模自動生成は将来性がある一方、技術的成果を主張するなら「ビルド可能で再現できる証拠」が不可欠。日本の開発現場でも、デモ映像だけで判断せず、再現性とCIの有無をまずチェックしましょう。
