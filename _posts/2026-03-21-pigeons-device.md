# Pigeon's Device - ピジョンズ・デバイス
思わず読んで試したくなる「スイッチ×if」のワザ：古典Cトリックでソートの振る舞いを自在に切り替える

## 要約
古いCコードに見られる「Pigeon's device」は、switchのcaseラベルとifを組み合わせて制御フローを巧妙に分岐させ、比較関数の振る舞い（前向き・逆向き・日付は逆で時間は順）を1つの関数で実現するテクニックです。

## この記事を読むべき理由
日本の組込み系やレガシーCコードを触るエンジニア、あるいは低レイヤでパフォーマンスやAPI制約と向き合う人は、このトリックの意図と代替手段を知っておけば、既存コードの解析・保守・改修が圧倒的に楽になります。

## 詳細解説
- 背景：Duff's deviceと似た発想の制御フローハック。Pigeon's deviceは比較関数（qsortなどに渡す）で「モード」を外部から設定し、関数内でモードに応じた比較を行うが、モード0では別条件（同じ日付かどうか）でさらに分岐します。
- 技術の本質：switchのcaseラベルがifブロック内部に現れることで、ラベルが落ちる位置によって実行パスが変わる。典型的にはこういう構造です（簡略化）：

```c
c
int pigeons_device(int a, int b, int mode) {
    int result;
    switch (mode) {
    case 0:
        if (extra_check(a,b)) {
            case 1:
            result = cmp_forward(a,b);
            break;
        } else {
            case 2:
            result = cmp_reverse(a,b);
            break;
        }
    }
    return result;
}
```

- 実装意図：ライブラリAPI（たとえばqsortの比較関数）に第3引数を渡せない制約下で「同じ関数で複数振る舞いを実現」するための工夫。オリジナル例では「日付は降順、同日内は時刻は昇順」という複合ソートを1個の比較関数で実現している。
- 問題点：可読性・保守性が低く、誤読やバグを生みやすい。現代の環境ではより明快な代替を使うべき。

## 実践ポイント
- レガシー解析時：まずコメントを残してトリックの意図を明示し、ユニットテスト（境界日・同日・異日）を追加して動作を固定化する。
- 改修案：可能なら比較関数にコンテキストを渡せるAPI（qsort_r 等）を使う、または関数ポインタや小さなラッパーでモードごとの比較を分離する。
- 日本市場での適用：組込み・金融・古い社内ツールなどCの既存資産が多い領域では遭遇確率が高い。安全に保守するためのテストと段階的リファクタが実用的。
- 覚え書き：面白い技術的トリックだが「面白さ」で採用するのはNG。ドキュメント化して保守性を優先すること。
