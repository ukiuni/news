# Lessons in Grafana - Part One: A Vision - Grafanaの教訓 — 第1回：ビジョン
Grafanaで「転職活動」を可視化して続ける――小さな入力で大きな洞察を得る方法

## 要約
セルフホストのデータをGrafanaで可視化し、日々の仕事応募やライフログを「見える化」して習慣化するための設計思想と実装上の課題を整理した記事です。

## この記事を読むべき理由
転職活動や個人の行動ログを単なる一覧で終わらせず、可視化して改善につなげたい日本のエンジニア／テック愛好者にとって、低摩擦なデータ収集とGrafanaでの表現手法が実践的に学べます。

## 詳細解説
- ビジョン：最小限の入力で、スマホ／PCどこからでも記録でき、編集や遡及が容易な仕組みを作る。最終的にはGrafanaでダッシュボード化して「やる気を保つ」。
- 可視化の主要要素：
  - 円グラフ（Pie Charts）：カテゴリごとの割合表示。データは「名前」と「値」の2列でOK。
  - 時系列（Time Series）：時間に沿った数値の推移。タイムスタンプ列＋各系列の数値列が必要。
  - ステートタイムライン（State Timelines）：時間中の状態変化を追う。例：稼働/停止、面接ラウンドの状態。
  - サンキー図（Sankey）：フロー／遷移を可視化。複数ステップの旅程（例：応募→一次→二次→内定）を表現可能。
- データ源としてのスプレッドシートの長所と短所：
  - 長所：手軽、クラウドで同期可能、編集しやすい（非エンジニア向け）。
  - 短所：2次元に制約、強力なクエリ言語がないためGrafana側での変換が増え遅くなる、プライバシー管理が難しい。
- 採用トラッキングの要件例：
  - 入力摩擦を最小化、複数デバイス対応、編集や過去データの修正が簡単、可能なら自動判定（例：一定期間応答がない応募を「ghosted」とする）など。
- 実装検討の比較：
  - ローカルスプレッドシート：PC限定だが管理は確実。
  - Joplin（ノート＋CSV）：モバイルはOKだがCSVは人間には編集しにくい。
  - IMボット経由：入力は楽だが一覧性が低い。
  - 結論：最初はクラウドスプレッドシートで速く始め、将来的にPostgresなどへスクレイピング移行を検討。
- 実用例：応募データ列（Company, Position, Applied, 1st, 2nd, 3rd, Offer, Rejected, Ghosted）。Ghostedは「最新の左側セルが一定期間（例：2週間）経過していれば自動判定」する計算式で自動化。

例：GoogleスプレッドのMAP/LAMBDAでの自動判定（抜粋）
```text
=MAP( D2:D, E2:E, F2:F, G2:G, H2:H, I2:I, LAMBDA( D, E, F, G, H, I, IF( AND(NOT(H), NOT(I), NOT(NOT(D)), MAX(D, E, F, G) + $M$1 < NOW()), MAX(D, E, F, G) + $M$1, "" ) ) )
```

## 実践ポイント
- まず最小限のカラムでクラウドスプレッド（例：応募日・ステータス）を用意して週次でダッシュボードを見る習慣を作る。
- Grafanaでは「円→時系列→ステートタイムライン→サンキー」の順で可視化を増やすと理解しやすい。
- モバイル入力が必須ならスプレッドシートか専用フォームを用意。後でDBへ移行する前提で列設計をしておくと楽。
- センシティブ情報はクラウドに生のまま置かない。必要ならローカル暗号化やオンプレDBへ移行する計画を立てる。
