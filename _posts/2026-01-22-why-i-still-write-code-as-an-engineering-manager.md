# Why I Still Write Code as an Engineering Manager - なぜエンジニアリングマネージャーでもコードを書くのか
マネージャーがいまも「手を動かす」べき3つの理由 — 信頼、判断力、そしてチームへの教え方

## 要約
エンジニアリングマネージャーが適度にコードを書くことは、技術感覚を維持し、チームに「良い仕事の見本」を示し、現場の痛みを共有して正しい意思決定を下すために有効である。重要な点は「重要路線（クリティカルパス）」を奪わないこと。

## この記事を読むべき理由
日本のプロダクト開発現場でも、マネージャーが技術から離れすぎると現場との乖離や説得力の低下を招く。特にエンジニア不足やレガシー改修が多い日本市場では、技術感覚の維持が意思決定とチーム信頼のカギになる。

## 詳細解説
- 何をやるべきか：クリティカルパスの高可視度な仕事は担当チームに残し、代わりに小さなバグ修正、ツール改善、ドキュメント更新、テスト補強など「影響はあるが四半期を左右しない」作業を選ぶ。  
- 見本を示す効果：実際にコードを出すことでPRの構成、コミットメッセージ、テストの丁寧さ、レビュー対応の仕方などのスタンダードが自然に伝わる。「言う」より「一緒にやる」方が学習効果が高い。  
- Skin in the game（当事者性）：自分でデプロイやテストを経験することで、フラストレーションや工数の実態が理解でき、無責任な要求や非現実的な締め切りに対する説得力が増す。  
- バランスの取り方：人数が多すぎる、ポリシー上の制約、戦略業務が主のポジションなど、コードを書く意味が薄い場合もある。重要なのは「できない理由」か「すべきでないと自分で納得しているか」を問い直すこと。  
- チームへの波及：手を動かすマネージャーは信頼を得やすく、技術負債の優先度やプロセス改善の提案が受け入れられやすい。

## 実践ポイント
- 週1〜数時間を「現場の手を動かす時間」として確保する。  
- 重要な機能には手を出さず、バグ修正・ドキュメント・CI改善などを担当する。  
- PRテンプレートやコミットルールを実例で示す（小さなPRを出して見せる）。  
- 定期的にペアプログラミング／コードレビューを行い、働き方を可視化する。  
- 締め切りや技術負債の主張には、自分の体験（実際に直した箇所）を根拠にする。  
- 自分の役割が変わったら「なぜコードを書かないか」を説明できるようにする。
