# Code isn’t what’s slowing projects down - コードがプロジェクトを遅らせているのではない、コミュニケーションだ
「コードは速いのに進まない」原因は“話さない”チームにあった — 今すぐ試せるコミュニケーション改善術

## 要約
機能実装が遅れるのはコード自体の難しさではなく、チーム間の情報共有・優先順位・プロセス不明瞭さといったコミュニケーションの欠如が主因だ、という指摘。

## この記事を読むべき理由
日本企業でも部門分断や属人化は日常課題です。技術的障壁より「誰に聞けばいいかわからない」ことが納期遅延や過剰設計を生みます。本記事は組織構造とシステム設計の関係（Conwayの法則）を踏まえ、現場で使える対処法を示します。

## 詳細解説
- 核心：単独や小チームだとアイデア→実装が速いが、複数チームやオーナー不明のコンポーネントに跨ると遅延が顕在化する。原因は「知らない人／プロセス／前提」が増えるため。  
- Conwayの法則：組織のコミュニケーション構造がソフトウェア構造に反映されるため、連携しにくいチームはシステムを分断しやすい。結果としてサイロ化や冗長なアーキテクチャになる。  
- 逆Conway（Inverse Conway Maneuver）：望むアーキテクチャを実現するためにチーム組成を設計する発想。ただし既存組織では難易度が高く、経営・エンジニア双方の合意と段階的な再編が必要。  
- 実例：A（キュー/スロットリング）とB（モデル変換）という別オーナーのコンポーネントを、自チームの知見だけで片付けようとすると過剰設計になる。情報を取りに行き、相手の意図を理解して協調することで無駄を省ける。  
- 人的側面が鍵：定期的なクロスチームセッション、ドキュメント化、所有者明示、共有テストケースなどが問題発見を早め、チーム間の信頼とスピードを回復する。

## 実践ポイント
- 所有者を明記する：コンポーネントごとに「誰が責任者か」をREADMEやサービスカタログに記載する。  
- 早めにノックする：不明点は雑でもいいから担当に直接確認。放置が遅延の元。  
- クロスファンクショナルなスプリントやスパイクを設ける：短期間で集中的に実装と知識共有を行う。  
- API契約・インターフェースを先に決める：合意した契約があると調整コストが減る。  
- 定期的な「合わせ読み」（integration session）：実際に集まって仕様やログを見せ合う場をつくる。  
- ドキュメント＋会話の両方：ドキュメントは出発点、会話で前提を揃える。  
- 小さな組織改編を試す：機能横断のチームを一時的に作ってインパクトを検証する。

短く言えば、コードは道具であって速度低下の主因ではない。人と情報の流れに投資すれば、設計も実装もずっとスムーズになります。
