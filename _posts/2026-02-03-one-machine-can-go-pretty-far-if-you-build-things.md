# One machine can go pretty far if you build things properly (2022) - うまく作れば一台でかなりいける
地下室のPC群はもう古い — 「一台で足りる」を目指すシンプルな設計術

## 要約
大規模クラスタを最初から前提にするより、効率的な設計と適切なツール選びで「1台（〜2台）」で十分運用できることが多い、という主張。

## この記事を読むべき理由
日本のスタートアップや小規模チームは人数・予算・運用リソースが限られがち。無駄なクラスタ化でコストと運用負荷を増やす前に、まず「少ない台数でちゃんと動く」設計を学ぶ価値があります。

## 詳細解説
- 背景：昔のBBSや初期ネットワーク運用では「複数台を並べる＝当たり前」だったが、実際にユーザが消費する計算量は小さい場合が多い。現代の仮想化・コンテナ・効率的なランタイムを使えば、多くのワークロードは1〜2台で賄える。
- なぜ1台で済むことが多いか：
  - 多くの処理はI/Oバウンド（待ち時間が主）で、非同期IOやイベント駆動で効率化できる。
  - 仮想化（VM/コンテナ）によりリソースを細かく分割・共有できる。
  - キャッシュ、キュー、リバースプロキシ等で高負荷を吸収できる。
- 問題点：ツール選びを誤ると「1プロセス＝1マシン」になりがち。慣習や求人要件で無駄なフットプリントを正当化しないこと。
- 実務的な指針：まず1台でプロトタイプ→負荷計測→ボトルネックに応じて水平/垂直に拡張。Paul Barhamの言葉「最初の1台を使いこなして初めて2台目が許される」は本質をついている。

## 実践ポイント
- 計測から始める：RPS、CPU、メモリ、I/O待ち時間を定量化する。
- 適切なスタックを選ぶ：非同期処理、軽量ランタイム、効率的な言語を検討。
- キャッシュとキューを活用：DBや外部APIへの直アクセスを減らす。
- コンテナ/VMでリソースを整理し、1〜2台で運用できる設計を試す。
- 可観測性を整える：ログ、メトリクス、アラートで負荷の兆候を早期検出。
- 段階的に拡張する：まず垂直強化（より強いマシン）、次に水平拡張を検討する。
- コスト意識を持つ：クラウド請求が急増する前に「本当に必要か」を検証する。

少ない台数で堅牢に動かせれば、運用負荷とコストが劇的に下がります。まずは最初の一台を使いこなすことを目標にしてみてください。
