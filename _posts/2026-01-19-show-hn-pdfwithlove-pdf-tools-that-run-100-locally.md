# Show HN: Pdfwithlove – PDF tools that run 100% locally (no uploads, no back end) - Pdfwithlove：100%ローカルで動くPDFツール（アップロード不要・バックエンドなし）
クリックせずにはいられないタイトル案：ブラウザだけで完結する「秘密厳守」PDFツール—クラウドに上げない安心感

## 要約
Pdfwithloveは、ファイルをクラウドにアップロードせずブラウザ内だけでPDF編集や結合・分割などを行えるウェブアプリとして紹介されています。プライバシー重視の代替手段を求めるユーザーにアピールします。

## この記事を読むべき理由
機密文書や個人情報を扱う場面が増えた日本の現場で、データを外部サーバに預けず作業できるツールは価値が高いです。簡単に試せて「本当にアップロードしていないか」を確認する方法も紹介します。

## 詳細解説
- 基本コンセプト  
  Pdfwithloveのキーメッセージは「no uploads, no back end（アップロードなし・サーバ不要）」。つまりファイルの読み込み・処理・出力がすべてユーザーのブラウザ内で完結する設計を目指しています。これにより、外部にデータを渡したくないドキュメント作業で安心して使えます。

- なぜブラウザだけで可能か（技術的な背景）  
  近年のブラウザはファイル読み込み、バイナリ処理、ファイル保存といった機能を豊富に持ち、WebAssembly（Wasm）やクライアントサイドのJavaScriptライブラリで重めの処理も実行できます。多くの「ローカルで動くPDFツール」は、これらのブラウザAPIとライブラリを組み合わせ、サーバを介さずにPDFの結合・分割・圧縮・回転・注釈付けなどを実現します。

- セキュリティとプライバシーの注意点  
  フロントエンドで処理するといっても、実際にファイルが外部へ送られていないかは確認した方が安全です。ブラウザの開発者ツールでネットワーク通信を監視し、ファイル送信（POST/PUT等）が発生していないかをチェックします。また、サイトが外部ライブラリやCDNを読み込む場合、ロード時の通信は発生しますが、アップロードとは別です。最も確実なのはソースが公開されているか（オープンソース）を確認し、ローカルでHTML/JSを動かすなどの手段です。

- 制限事項  
  ブラウザだけで動かす設計は便利ですが、処理速度やメモリ、ブラウザ互換性（モバイルでの制約など）に影響を受けます。大容量PDFや複雑なPDF（フォーム、デジタル署名、多数の画像）では機能差や失敗の可能性があります。

## 実践ポイント
- まずは小さなファイルで試す：個人情報を含まないサンプルPDFで機能（結合・分割・圧縮等）を確認する。  
- 「本当にローカルで処理されているか」を確認する方法：ブラウザの開発者ツール → ネットワークタブを開き、ファイルを読み込んだときに外部へのPOST/PUTが発生していないかを確認する。外部ライブラリの読み込みは発生してもファイル送信がなければ問題ない。  
- オフラインでの動作確認：可能ならネットワークを切断して同じ操作ができるか試す（純粋にクライアント処理なら動作するはず）。  
- 組織での利用検討：社内規程や情報システム部と相談のうえ、ファイル取り扱いルールに沿って導入する。重要書類は常にバックアップを残す。  
- 代替手段の比較：既存のクラウドサービス（iLovePDF / Smallpdf 等）と比較して、セキュリティ要件や使い勝手、処理品質の差を評価する。必要ならローカルで動く別のオープンソースツール（デスクトップアプリやCLIツール）も検討する。

以上を踏まえれば、Pdfwithloveのような「ブラウザだけで完結するPDFツール」は、日本のプライバシー意識や企業コンプライアンスの高まりにマッチする実用的な選択肢になります。興味があれば、まずは開発者ツールで「通信が発生していないこと」を確認し、身近な文書で試してみてください。
