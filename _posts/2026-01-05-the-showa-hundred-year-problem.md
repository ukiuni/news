# The Showa Hundred Year Problem - 「昭和百年問題」
魅力見出し: 昭和の「00年」問題は本当に来たのか？レガシーと和暦が絡む実務チェックリスト

## 要約
昭和元年（1926年）起算の和暦を二桁などで扱っている古いシステムは、2025〜2026年の節目で誤動作する可能性が懸念されたが、大規模な事故報告は見当たらない。とはいえ、国内の金融・医療・行政系のレガシーを点検しておく価値は高い。

## この記事を読むべき理由
和暦は日本の法律・医療・会計データで今も多用されており、和暦特有の起算ルール（元年＝1年、0年が存在しない等）がソフトウェアのバグ源になり得る。Y2Kのローカル版とも言える「昭和百年問題」の教訓は、日本のエンジニア/運用者にとって即役立つ実務知識を含む。

## 詳細解説
- 和暦の仕組み：グレゴリオ年とは別に「元号＋年」で年を数える。昭和は1926年に始まり1989年に終わる（昭和1＝1926）。昭和年を西暦に変換する公式は  
  $$
  \text{西暦} = \text{昭和年} + 1925
  $$
- なぜ問題になったか：昭和期はコンピュータ黎明期と重なり、当時のプログラムが「昭和○○」（2桁）で年を保持している例がある。さらに、簡易パッチで「63以上なら引く」といったハックが加えられると、内部表現が二桁のまま残り、99→00のラップで1925に巻き戻る可能性がある。
- もう一つの落とし穴：和暦は「元年＝1」で0年がないため、もし「元年を0で格納して＋1して表示する」実装なら、百周年の定義がズレて昭和101（＝2026）で問題が出ることがあり得る。
- 実際の影響：事前に指摘や準備はあったが、目立った大規模障害は報告されていない。多くの古いシステムは既に廃棄・置換済みだったためと考えられる。

## 実践ポイント
すぐに使える現場チェックリストと対処法。

- 探索（IDE／リポジトリ内検索）
  - VS Code等でまず文字列検索：
    - "昭和" を含むファイルを洗い出す
    - 正規表現で和暦数値を探す（例: 昭和\s*[0-9０-９]{1,3}）
  - 例（端末）：  
    ```bash
    rg "昭和\s*[0-9０-９]{1,3}" --hidden
    ```
- コード点検ポイント
  - 年の格納幅（2桁か4桁か）を確認。2桁格納は要注意。
  - 「もしn>63なら…」のようなハードコードされた変換ロジックを探す。
  - 元年を0で扱う実装（+1で表示）を探す。
- テスト
  - ユニットテストに境界ケースを追加：昭和99→00/昭和100/昭和101（1925/2025/2026相当）を通す。
  - 代表的な変換ルールは確認：西暦 = 昭和年 + 1925
  - サンプルPythonテストスニペット：
    ```python
    import re

    def showa_to_gregorian(s):
        m = re.search(r'昭和\s*([0-9]+)', s)
        if not m:
            return None
        sy = int(m.group(1))
        return sy + 1925

    assert showa_to_gregorian("昭和1年") == 1926
    assert showa_to_gregorian("昭和100年") == 2025
    ```
- 運用上の改善策
  - 内部ではISO 8601の西暦（4桁）で保持し、和暦は表示レイヤでのみ扱う。
  - 標準ライブラリや実績あるロケール対応ライブラリ（例：Javaの日本暦サポートなど）を使う。
  - レガシー機器（組込み、銀行端末、医療機器）には個別の棚卸とテストを実施。

短く言えば：大災害には至らなかったが、和暦特有の起算・表示の違いが残るシステムはまだ存在する。まずはリポジトリ検索→境界テスト→内部データの西暦化、という実務フローを回すことが最短のリスク軽減となる。
