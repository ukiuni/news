# How did somebody create a site like this? - こんなサイトはどう作ったの？

ラジオが地球をぐるっと飛び回るUIに惹かれる人へ — Radio Garden風の「世界中の生放送ラジオをワンクリックで聴ける」サイトは、どうやって作られているのかをやさしく解説します。

## 要約
地図（あるいはグローブ）表示＋ストリーミング音声の組合せで実現。フロントはWebGLや地図ライブラリ、ブラウザのAudio API、バックエンドはストリームのプロキシやメタデータ集約を担うのが一般的。

## この記事を読むべき理由
世界中のラジオを可視化して聴かせるUIは、位置情報・リアルタイム配信・メディア配信の良い教材。日本の地域局やコミュニティラジオをデジタルで広げたい開発者やサービス担当者にとって実装アイデアと注意点が得られます。

## 詳細解説
- 表示部分（UI）
  - グローブや地図はWebGLベース（three.js など）か、Mapbox / Leaflet にカスタムマーカーを載せる構成が多い。WebGLは滑らかな回転や3D表現、Mapbox/Leafletはタイル地図＋ズームが簡単。
  - マーカーは位置情報（緯度経度）と局のアイコンを持ち、クリックでプレイヤーを開く。

- オーディオ配信
  - ブラウザ側は HTML5 <audio> と Web Audio API を使う。HLS（m3u8）やICY/HTTPストリーム（MP3/AAC）に対応するため hls.js や shaka-player を組み合わせる。
  - ブラウザの自動再生制限（ユーザー操作が必要）やモバイルの制約に注意。

- ストリーム取得とプロキシ
  - 多くのラジオストリームはCORS制約やICYヘッダ対応が必要なため、サーバ側でプロキシ（リバースプロキシ）を置いてブラウザ互換に変換する。プロキシは接続監視やフェイルオーバーも担当。
  - ストリームのメタデータ（曲名、番組情報）はICYメタ、Shoutcast/ICEcast API、または別途メタデータAPIをスクレイピングして集約。

- スケーラビリティとパフォーマンス
  - 多数の局を扱う場合、マーカーのクラスタリング、遅延ロード（visible only）、CDNによる静的資産配信が必須。
  - ストリーミングは帯域が重いので、負荷分散・キャッシュ戦略・接続数上限を設計する。

- 法的・運用面
  - 放送配信許諾や著作権（楽曲配信）に関する確認が必要。特に商用や広域配信では各国の規制に従うこと。
  - 地域局と協業して公式ストリームを得る運用が安全。

## 実践ポイント
- まずは小さなPoCを作る手順
  1. 地図ライブラリ（Mapbox/Leaflet）で地点を表示する。
  2. サンプルのMP3/HLSストリームをHTML5 <audio> や hls.js で再生してみる。
  3. サーバで簡単なプロキシ（Node.js + express）を作り、CORS/ICY対応を確認。
  4. メタデータ取得用のスクレイパーかAPIを実装して曲情報を表示。
  5. UIを磨いて、地域別フィルタやお気に入り機能を追加。

- 日本向けの応用案
  - 地域コミュニティ局や祭りの臨時放送を集めた「日本版Radio Garden」を作る。
  - 観光地のライブ配信や多言語の案内放送と組み合わせると利便性が高まる。

短時間で試せる「地図＋サンプルストリーム」のPoCから始め、法的確認とスケール設計を進めるのが現実的な進め方です。
