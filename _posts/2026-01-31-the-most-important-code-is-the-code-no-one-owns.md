# The Most Important Code Is The Code No One Owns - 「誰のものでもないコード」が最も重要だ
見えない“誰も所有しない”コードが次の障害点になる — 今すぐ守るべき理由と現場でできる対策

## 要約
多くのサービスは外部パッケージに依存しているが、その中には組織的な責任や資金のない「誰のものでもないコード」――孤立した依存関係や放置されたライブラリ――が多数存在し、運用・セキュリティ上の重大リスクになっている。

## この記事を読むべき理由
日本のプロダクトやサービスも例外ではありません。受託開発、組み込み、金融系など長期間稼働させる現場では、目に見えないオープンソースの依存が障害やセキュリティ事故を起こす可能性が高く、事前の対策がコストと信用を守ります。

## 詳細解説
- ソフトウェアは多層の依存でできている：アプリが直接書いたコードは少数で、残りは外部パッケージ。これらの多くは「メンテナ」が個人の善意で維持している。  
- 「所有されていない」とは作者不在ではなく、長期の責任（運用、脆弱性対応、互換性維持）が組織的に担保されていない状態を指す。  
- 孤立依存（orphaned dependency）は見た目は正常でも、バグや脆弱性対応が滞り、徐々に互換性が崩れる。多数のプロジェクトが波及被害を受ける。  
- ライブラリが“偶然のインフラ”になる過程：小さなユーティリティが広く使われ、サポート体制は無いまま重要性だけ高まる。  
- トランジティブ依存（間接依存）は特に危険で、表層の信頼が下層の脆弱性を覆い隠す。  
- フォークは確保手段にはなるが持続可能性を生むものではない。資金・人手・ガバナンスが不可欠。  
- サステナビリティ＝セキュリティ：放置されたコードは将来の脆弱性温床になる。

## 実践ポイント
- まず依存の「現状把握」を：SBOM（ソフトウェア部品表）を作る。  
- 自動化で監視：Dependabot/GitHub Security、OSS Scorecard、Snykなどを導入。  
- 重要な依存は評価基準を設ける：メンテ状況（最終コミット、Issue放置率）、メンテナの所属、ライセンス。  
- トランジティブ依存も可視化してレビュー対象にする。  
- クリティカルなパッケージはベンダリング、内部管理、またはスポンサー／寄付で支援する。  
- 組織での責任を明確化：外部依存に対するオーナーをアサインし、切替・緊急対応手順を用意する。  
- 小さく始める：まずは「依存一覧の出力」「自動脆弱性通知の有効化」「重要ライブラリ3つの維持状況確認」を実施。

見えないコードは無視できないインフラです。放置せず「依存を資産として扱う」ことが次の事故を防ぐ一歩になります。
