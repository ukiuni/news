# Snake game but every frame is a C program compiled into a snake game where each frame is a C program... - フレームが全部「Cプログラム」になっているスネークゲーム

魅惑のプログラミングトリック：再帰的に「次のフレーム」をCソースとして吐き出し、それをコンパイルして次の実行ファイルにするスネークゲーム

## 要約
YouTubeのデモは、ゲームの各フレーム自体がCプログラムになっていて、そのプログラムをコンパイルすると次のフレーム（＝次のCプログラム）になる、という自己再帰的な仕組みでスネークを動かすアイデアを見せています。

## この記事を読むべき理由
メタプログラミング、コンパイラ呼び出し、自己複製的なソフトウェアというトピックは、プログラミングの深い理解を促します。日本の学生やハッカーカルチャー、プロトタイピングを好むエンジニアにとって、発想と実装の融合を学べる良い題材です。

## 詳細解説
動画の核心は「コードがデータであり、データがコードである」ことを実演する点です。具体的には概念的に以下の要素が関係します。

- フレームを表すソースコード生成：各フレームの状態（ヘビの座標、画面、方向など）をCのソースコードとして出力する。
- 動的コンパイル：生成したソースをgccなどのコンパイラでコンパイルし、生成された実行ファイルを実行して次のフレームを生成するループ。
- 状態の受け渡し：実行ファイル間でフレーム状態をファイルや標準入出力、引数で受け渡す実装が考えられる。
- 制約と工夫：コンパイル速度、実行サイズ、サンドボックス（安全性）やファイルI/Oのオーバーヘッドをどう抑えるかが鍵。生成コードを極小化するためのテクニック（短い表現、圧縮的なデータ表現、共通ランタイムの活用など）も重要です。

このアプローチはクイーン（quine）や自己複製プログラム、ライブコーディングの一種で、教育用途やデモ、チャレンジ問題として面白い題材です。

## 実践ポイント
- 小さく始める：まずは「次の文字列を出力するだけのCプログラム」を生成してコンパイル→実行するパイプラインを作る。
- コンパイル呼び出し例（概念）：
```c
// c
#include <stdlib.h>
int main(){
  system("gcc next.c -O2 -o next && ./next");
  return 0;
}
```
- 性能対策：毎フレーム毎にフルコンパイルすると遅いので、最初は静的データを埋め込むか「差分だけ生成」するなどの工夫を。
- セキュリティ：任意コードを生成・実行するため、ローカルで実験する際も実行環境を分離（仮想環境やコンテナ）する。
- 学習用途：大学のコンパイラ演習、プログラミングコンテストのネタ、クリエイティブコーディングに応用可能。
