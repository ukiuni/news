# MCP is dead. Long live the CLI - MCPは死んだ。CLI万歳
なぜ“モデル用プロトコル”よりも端末（CLI）がAI時代の現場で頼りになるのか

## 要約
著者はModel Context Protocol（MCP）が実務では過剰で非効率だと主張し、既存のCLIツールの方がデバッグ性・合成性・認証周りで優れていると結論づけています。

## この記事を読むべき理由
日本の開発現場でもAIエージェントを実運用に組み込む動きが進んでおり、どの接続インターフェースに投資すべきかは重要な判断です。MCPとCLIの違いを理解すると、コスト・運用負荷・セキュリティ設計で賢い選択ができます。

## 詳細解説
- 背景：AnthropicのMCPは「LLMがサービスを安全に呼べる標準」を目指して注目を集めましたが、実装コストや運用の手間が現場で問題になっています。OpenClawやPiなど主要ツールがMCPを採用していない点も示唆的です。  
- LLMはCLIが得意：LLMは大量のマニュアルやシェルスクリプトで学習しており、`gh pr view 123` のような実CLIコマンドをそのまま使えることが多い。わざわざ新しいプロトコルでラップする必要性が薄いと著者は言います。  
- デバッグ性：CLIは「同じ入力＝同じ出力」が保証されやすく、人間がすぐ再現・確認できる。一方MCPはツールの振る舞いが会話内部に埋もれ、トラブル時にJSONログを解析する羽目になります。  
- 合成性（Composable）：UNIX哲学の利点で、`terraform show -json plan.out | jq ...` のように既存ツールをパイプやフィルタで組み合わせられる点はMCPでは再現しにくい。大規模データをコンテキストに入れるコスト問題もあります。  
- 認証・権限：AWS CLI、gh、kubectlなどは既にSSOやプロファイル、kubeconfigといった堅牢な認証フローを持つ。MCPは認証設計で冗長になりがちで、複数ツール時の煩雑さが増します。さらに、権限の粒度（閲覧のみ／マージ可など）もCLIの運用で柔軟に実現できます。  
- 運用面の摩擦：ローカルMCPサーバはプロセス管理・起動失敗・再認証問題など「動かし続ける」ための運用コストが発生。CLIは単なるバイナリで状態管理が少ないため軽量です。  
- 例外：CLIが存在しない特殊ツールや、標準化されたインターフェースが本当に有用なユースケースではMCPが適切な場合もあると著者は認めています。

## 実践ポイント
- まずは良いAPIと良いCLIを出す：新機能をMCPで囲う前に、まずはAPIとCLIを整備する。エージェントはCLIを使えるようになる。  
- 運用重視の設計：デバッグ再現性が高いワークフロー（同じコマンドを人も実行できる形）を優先する。  
- 認証は既存フローを流用：SSO／プロファイル／kubeconfig等、既に実績のある認証方式を使わせる設計にする。  
- 権限の粒度を考える：ツール単位だけでなく操作単位（read/merge等）での制御を設計する。  
- MCP導入は慎重に：CLI代替が無い本当に必要なケース以外は後回しにし、まずは運用負荷を下げる選択を。

著者のメッセージはシンプルです。AI時代でも「人と機械の両方が使える道具」であるCLIの実用性を忘れないでください。
