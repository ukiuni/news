# Please stop using passkeys for encrypting user data - ユーザーデータの暗号化にパスキーを使うのはやめてください
パスキーで「思い出」を失わないために。認証と暗号化を混同すると取り返しがつかない。

## 要約
パスキーのPRF拡張を使ってユーザーデータ（バックアップやウォレットなど）を暗号化すると、ユーザーがパスキーを削除・紛失した際に復元不能になり得るため危険。認証用とデータ暗号化用の役割を分離すべき、という注意喚起。

## この記事を読むべき理由
日本でもメッセージアプリ、クラウドバックアップ、暗号資産ウォレットなどでパスキー採用が進む中、ユーザーの大切な写真・文書・資産が「知らぬ間に復元不能」になるリスクが現実的に高まりつつあります。開発者・プロダクト担当は設計方針を今すぐ見直す必要があります。

## 詳細解説
- 背景：WebAuthnのPRF（Pseudo‑Random Function）拡張は、パスキーから派生鍵を生成できるため、暗号化キーのソースとして使われ始めている。これ自体は技術的に可能だが副作用がある。  
- 問題点：パスキーは本来「フィッシング耐性のある認証手段」。その削除や移行はユーザーにとって単純な操作に見えるが、同じパスキーで暗号化したデータは復元不能になる。ユーザー行動（クレデンシャルの整理、新端末でのアカウント復旧など）と暗号化のライフサイクルが乖離するため「失うリスク」の範囲（blast radius）が非常に大きくなる。  
- 正当なPRFの用途：OSやクレデンシャルマネージャのローカルアンロック等、短期的・運用的に安全性と利便性を両立させる用途には有用。これらはリカバリ手段（マスター・パスワード、デバイス鍵、リカバリキー、ソーシャルリカバリ等）で補強されることが前提。  
- 典型的な被害例：メッセージアプリのバックアップ（写真や動画）、暗号資産ウォレットの鍵、重要書類の暗号化ファイル。日本の家族写真や契約書データも該当し得る。

## 実践ポイント
- 基本方針：認証（パスキー）とデータ暗号化の鍵は分離する。パスキーを暗号化キーの唯一の依存先にしない。  
- 代替策：専用のデータ暗号化キー（KDFでの安全な派生・サーバ側管理・ユーザーが保管するリカバリキー）や複数の復旧手段を設計する。  
- UI/運用：もしPRFを使うなら、明確な事前説明と復旧不能リスクの警告を表示し、削除時にRP情報を見せる。サポートページで用途を公開し、Well‑Known の prfUsageDetails にURLを登録する。  
- クレデンシャルマネージャ側：PRF付きパスキーを削除する際に重大な警告を出す（RP情報の表示を含む）。  
- 日本市場向け配慮：LINEや国内サービスで導入する場合は「写真やトーク履歴の永遠の消失」を想起させる表現でユーザー教育を強化する。法務・サポート体制で復旧要請に対応できる設計を備える。

短く言えば、「パスキーは認証のままで強力に使い、ユーザーデータの暗号化には別設計で復旧手段を必ず用意する」。
