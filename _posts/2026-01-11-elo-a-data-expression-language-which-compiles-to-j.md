# Elo – A data expression language which compiles to JavaScript, Ruby, and SQL - Elo — JavaScript / Ruby / SQL にコンパイルできるデータ表現言語
フロントもバックもDBも「同じ言語」で記述できる――非技術者にも安心なデータ向け言語、Eloの魅力をざっくり解説

## 要約
Eloは「値のみ」を扱うシンプルで安全なデータ表現言語で、1つの式をJavaScript・Ruby・SQLに意味的に等しいコードへコンパイルできる。日付・期間やスキーマ検証を組み込み、ノーコード／ローコードや業務ロジックの共通化に向く設計が特徴。

## この記事を読むべき理由
日本のプロダクト開発ではフロント、サーバー、DBの間でロジックが散らばりがちで、日付計算や入力検証の差分がバグや不整合を生む。Eloは「同じ式を全スタックで同じ意味にする」ことで、こうした実務上の痛みを減らす可能性があるため、プロダクト設計やノーコード基盤を検討する人は知っておく価値が高いです。

## 詳細解説
- 言語設計（シンプル＆安全）
  - Eloは「純粋なデータ言語」。参照や副作用を排し、すべてが値として扱われるため非技術者が扱う想定でも安全性が高い。
  - 比較演算子や等価性は一貫した振る舞いを目指している（実運用での型差異によるトラブルを減らす設計）。

- ポータビリティ（1式 → 3ターゲット）
  - 1つのElo式をそれぞれのターゲットで意味が同じになるようにコンパイルする。フロント（JavaScript）、バックエンド（Ruby）、データベース（SQL）で同じビジネスルールを共有可能。
  - 例: 2^10 の判定や日付の年始比較が各言語で適切な関数や日付操作にコンパイルされる。

- 日付・期間・スキーマの標準化
  - 日付（Date、Datetime）や期間（Duration）をリテラルとして持ち、たとえば「ある日付に30日を足す」といった操作が直感的に書ける。パースや外部ライブラリ不要。
  - データスキーマ機能で外部入力を検証・強制変換（coerce）できる。フォーム入力やAPI受信データの「信頼できる形」への整形が組み込みで可能。

- パイプラインと関数型操作
  - |> パイプライン演算子、ラムダ（~>）や標準ライブラリ（filter, map, reduce 等）でデータ変換を宣言的に記述。
  - ガード（labeled guards）で前提条件を明示的にチェックし、失敗時に分かりやすく止められる。これにより「前提の検証 → 安全な処理」という設計が簡潔に。

- 将来機能と用途
  - 関係代数（Tutorial D に着想を得た機能）を導入予定で、複雑な結合・集約・照会もポータブルに書けるようになる見込み。
  - 主なユースケース：フォーム検証、注文処理、サブスクリプションの料金計算、No‑Codeプラットフォームのビジネスロジック層。

簡単な例（Elo とコンパイル先イメージ）:
```elo
let signup = D2024-06-15 in TODAY > signup + P30D
```

JavaScript にコンパイルされた例（イメージ）:
```javascript
DateTime.now().startOf('day') > DateTime.fromISO('2024-06-15').plus({ days: 30 })
```

SQL にコンパイルされた例（イメージ）:
```sql
CURRENT_DATE > DATE '2024-06-15' + INTERVAL '30 days'
```

## 実践ポイント
- まずは Playground で「日付計算」「フォームのスキーマ検証」「簡単なパイプライン」を試す：概念の理解が早い。
- ノーコード／ローコード製品を開発・評価しているチームは、Eloをルール表現の共通層にできるか検討する（フロント→DBまで同じ式で保証できる利点）。
- SQL生成に注意：複雑な集約やパフォーマンス要件は、生成SQLの実行計画を必ず確認すること（意味は保たれても効率は変わる可能性あり）。
- 日本市場での利点：祝日・タイムゾーンや業務日数計算など日付周りの仕様が多い業務でバグを減らせる。入力バリデーション強化は法令対応や監査の観点でも有用。
- 採用判断時はOSSライセンス（MIT）、コミュニティとメンテナンス状況、ターゲット言語での出力の読みやすさをチェックする。

短評：Eloは「業務ロジックを言語横断で共有する」実務的なアプローチを提供する小さく実用的な言語。特に日付処理と入力検証が厳しい日本の業務システムで採用検討に値します。
