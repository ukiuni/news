# Everything you might have missed in Java in 2025 - 2025年のJavaで見落としたかもしれないことすべて
Java 30で変わる日常：レガシーもクラウドも一歩先に進む新機能ガイド

## 要約
2025年、Javaは言語・JVM・エコシステムの基盤に関わる大きな改善を複数導入した。仮想スレッドのピン止め解消やStreamの拡張、バイトコードAPIの標準化、ポスト量子暗号対応、オブジェクトヘッダの縮小などが、実運用・コスト・移行の観点で即効性のある影響を与える。

## この記事を読むべき理由
日本のエンタープライズやクラウド運用では「既存資産の維持」「コスト最適化」「長期保存データの安全性」が重要課題。2025年のアップデート群はこれらに直接効くため、今後の採用判断や移行計画にすぐ役立つ知見が得られる。

## 詳細解説
- Java 30とコミュニティの盛り上がり  
  Javaは30周年を迎え、イベントやプラグイン、教材が増加。コミュニティ主導で資料・ツールが広がり、学習や導入のハードルが下がっている。

- 仮想スレッドの「ピン止め」問題解消（JEP 491 / JDK 24）  
  これまで synchronized による「ピンning」が仮想スレッドのスケーラビリティを損ねる大きな理由だったが、JDK側のモニタ実装変更により多くの既存コードがそのまま仮想スレッドと互換になる。大規模レガシーコードベースの移行コストが劇的に下がる。

- Stream Gatherers（JEP 485）でStreamが拡張可能に  
  window, scan, fold などの中間操作をネイティブに追加できるようになり、遅延評価と合成性を保ちつつ複雑なストリーム処理が書きやすくなる。従来の collect() に詰め込んだ冗長な実装を削減できる。

- Class-File API（JEP 484）でASM依存からの脱却  
  バイトコード操作の公式APIがJDKに追加され、フレームワークやツールが外部ライブラリ（ASM）に頼らずにクラスファイルを安全に扱える。新JDK対応の待ち時間や互換性リスクが減り、保守が楽になる。

- ポスト量子暗号の初導入（JEP 496 / JEP 497）  
  ML-KEM と ML-DSA をネイティブでサポート。今日暗号化したデータを将来量子で解読されるリスクを考えると、長期保存データ（行政記録、金融記録、医療データ等）は早めの準備が推奨される。

- Compact Object Headers（JEP 519）でメモリ効率向上  
  オブジェクトヘッダを縮小し、小オブジェクト多数のワークロードでヒープ利用が平均約22%削減。クラウド料金、コンテナのサイズ、GC挙動にポジティブな影響が出る。

- Scoped Values が ThreadLocal の常識を変える  
  実行スコープに紐づく値を安全に伝播できる仕組み。スレッドプールや仮想スレッド環境での漏洩やメモリリークを避け、より明示的で安全なコンテキスト伝播が可能に。

- Java Flight Recorder（JFR）の拡張（JEP 509 / JEP 520）  
  CPU時間ベースのプロファイリングやメソッドタイミングの精度向上により、クラウド／コンテナ環境で実際にCPUを消費している箇所を正確に把握できるようになった。運用コスト削減とパフォーマンス最適化に直結する。

## 実践ポイント
- 優先度高：まず社内PoCで最新JDK（24/25相当のビルド）を試す。仮想スレッド（Loom）を既存サービスで安全に動かせるかを確認すると移行判断が早まる。
- メモリ検証：Compact Headers を有効にしてヒープ使用量とGC挙動をベンチマーク。クラウドコスト試算に直結するためROIが見えやすい。
- 依存整理：フレームワークやビルドツールがClass-File APIへ移行しているか確認。ASM依存の影響範囲を洗い出す。
- セキュリティ計画：長期秘密保持が必要なデータ（20〜30年スパン）はポスト量子対応のロードマップに組み込む。今から鍵管理や移行方針を検討するのが現実的。
- 観測性強化：JFRの新機能でCPUタイミングを常時収集し、コスト改善につながるホットスポットを洗い出す。SREチームに組み込むと効果が高い。
- コード実践：ThreadLocal を Scoped Values に置き換える設計を検討。新しいコンテキスト伝播モデルはクラウドネイティブ設計に合う。

以上のポイントは、今すぐにでも小規模な実験から始められるものばかり。特に日本の金融・公共・ヘルスケア系の組織は「長期保存データの安全」と「運用コスト削減」の両面で恩恵が大きい。まずは手元のサービスで小さな実験を行い、段階的に採用を進めることを推奨する。
