# Bit-fields - ビットフィールド
C/C++で「メモリ節約」を狙うときの落とし穴と、現場で使える実践テクニック

## 要約
ビットフィールドは構造体の整数メンバをビット単位で詰めてメモリ節約できるが、配置（レイアウト）は実装依存で、思わぬパフォーマンス低下やサイズ増になることがある。正しい選び方と回避策を知れば安全に使える。

## この記事を読むべき理由
日本の組込みや低レイヤ開発、ネットワーク/ファイルフォーマット実装ではメモリ配置が重要。ビットフィールドの「見た目の節約」が実際の性能・可搬性にどう影響するかを理解しておくと無駄なバグや非効率を防げる。

## 詳細解説
- ビットフィールドとは  
  struct/classの整数メンバに対し「幅」をビット単位で指定する機能。例:  
  ```c
  // C
  struct BitfieldedStruct {
      uint32_t value1 : 20;
      uint32_t value2 : 10;
      uint32_t value3 : 2;
  };
  ```
  この例は $20+10+2=32$ なので合計で $4$ バイトに詰められることが期待できる。

- メモリ節約とCPUコストのトレードオフ  
  ビット抽出・挿入にはシフト/AND/ORなど追加命令が入る（x86やARMでの命令列を生成）。しかしデータが小さくなりキャッシュに入りやすくなれば逆に高速化する場合もある。

- 実装依存（ABI）とストラドリング（跨ぎ）  
  C/C++標準はビットフィールドの割り当て・アライメントを実装定義にしているため、同じ宣言でもプラットフォームやコンパイラでサイズや並びが変わる。特に小さなベース型（uint8_t）を使うとビットが隣り合うアロケーション単位を跨いでしまい、期待通りに詰まらない（例：16ビット分でも3バイトになるケースがある）。

- 回避策：基底型を大きくする／パック指定の落とし穴  
  uint8_t を uint16_t に変えると同一単位に詰められる保証が働きやすいが、アライメント要件が変わる。__attribute__((packed)) や #pragma pack を使えば詰められるが、アンアラインアクセスや余分な命令、キャッシュ境界跨ぎの遅延を招く。

- 通常メンバとの混在と並べ替え  
  ビットフィールドと通常のメンバを混ぜると、通常メンバのアライメントでパディングが入る。サイズを最小化するにはメンバ順を工夫して同じワードに詰める（コンパイラは順序を変えない）。

- 可搬性確保の実務指針  
  コンパイル対象（GCC/MSVC/ARM/x86）で sizeof を必ず確認し、static_assert で期待サイズを検証するのが現場の常套手段。

## 実践ポイント
- まず sizeof を測る：宣言後に sizeof(struct) を確認する。  
- 小さな幅を並べるなら、意図的に大きめの基底型（例: uint16_t）を検討する。  
- サイズ保証が必要なら static_assert を追加する。例:
  ```c
  // C
  struct PortableBitfield { uint16_t a:7; uint16_t b:7; uint16_t c:2; };
  static_assert(sizeof(PortableBitfield) == 2, "unexpected size");
  ```
- packed を使うのは最後の手段。性能測定（プロファイラ）とターゲットでのテスト必須。  
- 可搬性重視ならビット演算（シフト/マスクで明示的に詰める）を選ぶことも検討する。  
- 実機（ターゲットABI）で必ず動作確認：コンパイラやアーキで挙動が変わる。

短いまとめ：ビットフィールドは強力だが罠も多い。節約効果とコストを測り、型選定・並び・テストで安全に使おう。
