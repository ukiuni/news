# Vibe coding needs git blame - Vibeコーディングにはgit blameが必要
AIが作ったコード、誰の“意図”かわかりますか？ プロンプトもコミットして「git blame」できる開発フローのススメ

## 要約
AIプロンプト（人間の指示）がコード生成の中心になる「vibe coding」時代、プロンプトを記録・開示してgitで追跡できるようにすることがトラブルシュートと品質管理に有効だ、という議論。

## この記事を読むべき理由
日本の現場でもAIが開発ワークフローに入り始めています。AI生成の変更が増えると「誰が何を意図したのか」「どのモデルで生成したのか」が不明瞭になり、バグ追跡やレビュー効率、コンプライアンス面で問題になります。早めにルール化しておくと現場の混乱を避けられます。

## 詳細解説
- プロンプトは「意図（spec）」だが、自然言語は曖昧で実行可能なソースコードとは性質が異なります。LLMの出力は確率的で、同じプロンプトでもモデルや実行条件で結果が変わるため、プロンプト単体を“ビルド入力”として信頼するのは危険です。
- 再現性の問題：温度（temperature）やAPIバージョン、モデルの更新により結果が変動します。モデルのスナップショットが永続的に使えるわけではないため、依存関係の固定（パッケージ版固定やコンテナ化）とは違った課題があります。
- コンテキスト依存：会話履歴や外部ツール出力、スクリーンショットなどが生成結果に影響するため、プロンプト単独では完全な再現が難しい場面が多いです。
- だからこそ「プロンプトを記録」しておく価値がある：意図の検証、学習材料としての共有、AI生成箇所のレビュー優先度決定など、実務上の利点が明確です。
- 懸念点：プロンプトは下書きや愚痴、秘密情報（APIキー等）を含みがち。プライバシーとメンテナンス性を考慮した“赤字化（redaction）”やコミットの精査が必要です。
- コミュニティ動向：OSSや企業の間でAI貢献の開示を求める流れがあり、運用ルールを持つことは信頼性向上に寄与します。

## 実践ポイント
- プロンプトをコミットに紐づける  
  - 例：コミット本文やフッターに「AI-Contrib: model=XXX, version=YYY, prompt-file=prompts/123.prompt」を入れる。簡単なメタデータで追跡しやすくする。
- プロンプトをファイルで保存する仕組み  
  - リポジトリ内に .prompts/ や prompts/<issue>-<hash>.prompt を置き、gitで差分・履歴が追えるようにする。
- 再現性のためのメタ情報を残す  
  - 使用したモデル名・バージョン、APIプロバイダ、temperatureやseed、実行時のツール（Cursorなど）を記録する。
- セキュリティとプライバシー対策  
  - APIキーや個人情報は決してプロンプトに含めない。公開前に自動で赤字化（scrub）するCIステップを設ける。
- レビュー運用を明確にする  
  - 「AI生成＝レビュー要重点箇所」というルールを作る。UIはAIで作っても、認証や決済周りは人間が厳格チェックするなど役割分担を定める。
- コミットの品質向上習慣  
  - AIで生成したならコミットメッセージもAIで整形し、"fixed it"のような怠慢なメッセージを避ける。公開前に不要な“汚れた”プロンプトはスクワッシュ／編集する。
- 環境の固定化  
  - 依存はしっかりピン留め、DockerやCIでビルド手順を明文化して「動く環境」を担保する。
- 小さく始める  
  - まずは社内リポジトリでプロンプトの記録ルールを試し、運用コストと効果を検証してから広げる。

短くまとめると：プロンプトは便利な“意図の記録”であり、消してよい一時的なメモではない。gitで追跡できる形にしておくと、トラブル対応・学習・レビュー効率の面で大きな助けになります。日本の現場でも早めに「プロンプトの扱い方」を決めておくことをおすすめします。
