# So I've Been Thinking About Static Site Generators - 静的サイトジェネレーターについて考えてみた
サブ100msを目指す、次世代の超高速SSG設計図

## 要約
著者は「とにかく速い」静的サイトジェネレーター（SSG）を目指し、ビルドを単一のコンパイル済みプロセスで行い、細粒度のインクリメンタル（Red–Green）キャッシュとスクリプト化されたロジック分離で瞬時の再構築を狙う設計を提案しています。

## この記事を読むべき理由
日本でもブログやドキュメント運用で「編集→即反映」の高速な開発体験が求められており、本記事の設計原則は実運用での体感速度を劇的に改善できます。

## 詳細解説
- SSGの二分類  
  - 個人向けで独自性の高い実装（作者専用）と、Jekyll/Hugo等の大衆向け実装がある。著者は前者的な「自分が満足する速さ」を狙う。

- 「速さ」の定義とプロファイル結果  
  - 著者の計測ではファイルI/Oよりもメモリ内でのデータ変換や大量のJSプロセス間シャッフルが遅延の主因。straceではsyscall時間は僅かで、残りはランタイム内の処理負荷。

- 設計方針（要点）  
  1. ビルドはコンパイル済み言語で行う：変換処理（Markdown→HTML、HTML最適化等）は Rust/C などの高速実装が有利（例：comrak、minify-html）。  
  2. 単一プロセスで完結させる：多数の子プロセス起動や頻繁なファイルアクセスを避け、サイト全体をメモリで変換して差分だけ書き出す。  
  3. 真のインクリメンタル（Red–Green）キャッシュを採用：ファイルやクエリを細粒度でキャッシュし、入力に変化があっても出力が変わらなければ「緑」として再計算を回避する。これにより「前後リンク用のメタデータが変わった」などで全ページ再ビルドになる過剰再計算を防ぐ。  
  4. ハーメティック（決定論的）ビルド：入力が同じなら出力も同じという前提を担保できる設計が、Red–Greenなどのアルゴリズムを成立させる。  
  5. ロジックとドライバーの分離：変換エンジン（コンパイル済み）とサイト固有ロジック（スクリプト）を分ける。ロジックはインタプリタで即時編集・反映できると開発体験が大幅に改善する。

- 言語選定と実装案  
  - 著者は Rust の高速コア＋組み込みJS（rquickjs）を選択。コアでインクリメンタルエンジン／Markdownパーサ／HTMLミニファイ等を実装し、テンプレートやサイト固有の振る舞いはJSで書く。これによりロジック修正時の反復が早い。

## 実践ポイント
- まず現状を可視化：strace/timeでビルドのボトルネックを把握する（プロセス起動数・syscall頻度・ランタイムCPU）。  
- 単一プロセス＆インメモリ変換を試す：多数の小プロセスを生成するMakefile式より高速化しやすい。  
- 細粒度キャッシュ設計を導入：クエリ単位でキャッシュし、出力比較で再計算を回避（Red–Greenの考え方）。  
- ロジックをスクリプト化して分離：頻繁に変える表示ロジックはインタプリタ側へ。コンパイル時間を挟まず反映できる。  
- 既存部品の活用：Markdown→HTMLはcomrak、HTML最適化はminify-html、JSインタプリタはrquickjs等を検討。  
- 目標目安：サブ300msを目標、可能ならサブ100ms（著者はサブ20msも視野）。

短く言えば、「コンパイル済みで単一プロセスの高速変換エンジン＋細粒度インクリメンタルキャッシュ＋スクリプト化されたロジック」で、編集体験を“別次元に速く”できる、という提案です。
