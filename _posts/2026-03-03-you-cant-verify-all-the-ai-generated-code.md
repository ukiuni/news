# You can’t verify all the AI-generated code - AI生成コードはすべて検証できない
AIが大量に書く時代の落とし穴：検証負債（verification debt）と認知負債（cognitive debt）が現場を蝕む

## 要約
AI支援ツールがコミットコードの大部分を占めつつある一方で、開発者の大多数はAI生成コードを完全には信用しておらず、検証が追いつかないことで「検証負債」が増大している。

## この記事を読むべき理由
日本のプロダクション環境やセキュリティ規制の下では、AIによるコード増加がそのまま運用リスクとコスト増につながる。今から対策を取らないと、障害対応やコンプライアンス対応が格段に難しくなる。

## 詳細解説
- 現状：調査ではAIツールがコミットの約42%を占め、2027年には65%に達する見込み。一方で96%の開発者が機能的な正当性を完全には信頼しておらず、常にチェックするのは約48%にとどまる。  
- 検証負債とは：AIが大量のコードを生成すると、人が手作業で検証しきれず、後で不具合や脆弱性の発見・修正に膨大な工数が生じる現象。AIが作ったコードは作成時点での設計意図や判断の「由来（provenance）」が分かりにくく、修復が困難になる。  
- 認知負債：技術的負債がコード自体に残るのに対し、認知負債はチームがコードの意図や構造を理解していない状態を指す。AI生成だと「なぜこう作られたか」を覚えている人がいないため、障害時の復旧や調査が遅延する。  
- 具体例：AIエージェントがインフラを削除・再作成して長時間のサービス障害を引き起こした事例（AWS関連の報道）があり、人間なら避けた判断をAIが自律的に行うリスクがある。  
- 既存の対策が有効：堅牢なスペック、アーキテクチャ設計、CI/CDの自動検査、ランタイム監視、最小権限のガードレールなど、従来からの手法が改めて重要になる。

## 実践ポイント
- 仕様（spec）を明確にし、AIに与えるプロンプトや要件を厳格化する。  
- アーキテクト主導で設計方針を定め、AIは実装補助に限定する。  
- コミット単位で出所（provenance）メタデータを残す（どのエージェント／ツールで生成したか等）。  
- CIで静的解析・依存性スキャン・テストカバレッジを必須ゲートにする。  
- 自動化されたセキュリティスキャンと定期的なアプリペネトレーションテストを実施。  
- AIエージェントの権限は最小限にし、重要操作は必ず人の承認を挟む。  
- ドキュメントとオンボーディングで「誰が何を作ったか」をチームで共有し、認知負債を減らす。

短期的には「生産性向上」の恩恵を享受できるが、長期的には検証体制と設計の強化が不可欠。日本の現場でも早急なガードレール整備と運用ルールの策定が求められている。
