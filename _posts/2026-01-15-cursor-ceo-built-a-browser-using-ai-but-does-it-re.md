# Cursor CEO Built a Browser using AI, but Does It Really Work? - Cursor CEOがAIでブラウザを作ったが、本当に動くのか？
AIが「一週間で3百万行」を自動生成──実験の光と影

## 要約
CursorのCEOがGPT-5.2ベースの数百のAIエージェントを使い、一週間で独自レンダリングエンジンを含むブラウザを自律的に生成した。動くが「実用レベル」ではなく、多くの課題が残るというのが結論。

## この記事を読むべき理由
AIによる大規模自動コーディングは単なる話題性を越え、今後のソフトウェア開発プロセスや職務分担を変える可能性がある。日本のプロダクト開発や品質重視の現場にとって、実務適用の現実性とリスクを早めに理解しておく価値が高い。

## 詳細解説
- 何が行われたか  
  Cursorのチーム（CEO Michael Truellら）はGPT-5.2を複数同時に動かす「マルチエージェント」方式でブラウザを作成。結果は数千ファイル・3M+行のコードに到達し、レンダリングやCSS適用、テキスト整形、描画、独自のJavaScript VMまで含む。実装言語はレンダリング部分がRust。

- 成功の要因  
  単純に「多数の同格エージェントを放り込む」だけでは失敗した。キーは役割分担の導入：Planners（設計・タスク分解）、Workers（実装）、Judges（評価・継続判断）という階層的ワークフローで、これが協調のボトルネックを解消した。

- 技術的な限界  
  表示できるページは限定的で、ChromiumやWebKitのような成熟度には遠く及ばない。大きな欠落は、拡張機能、パスワード管理、セキュリティ脆弱性対応、アクセシビリティ、クラッシュハンドリング、数千のエッジケース対応など周辺機能。生成コードの可読性・体系化・テストの充実度にも疑問が残る。そもそも「人が目標や設計、ワークフローを決めた」点を忘れてはならない。

- 倫理・法的な観点  
  大規模言語モデルは公開コードやドキュメントで学習しているため、出力に含まれる設計や実装が既存のコードベースに依存している可能性がある。ライセンスや知財、再現性の観点で注意が必要。

## 日本市場との関連性
- プロダクト堅牢性を重視する日本企業（例：金融、組み込み系、メガサービス）は、生成されたコードをそのまま本番投入することは現実的ではない。だが、プロトタイプ作成やPoC、リファクタ作業の「起点」としては効率化効果が期待できる。  
- LINE、楽天、メルカリなど大規模サービス運営企業は、マルチエージェントによる自動化を社内ツールやテスト自動化に取り入れる余地がある。だが運用・保守体制と厳格なQA文化が採用の鍵になる。  
- 人材面では「コーディングするだけの職務」から「AIを設計・監督し、品質を担保する役割」へシフトする兆し。システム設計、テスト、自動化、セキュリティのスキル需要が高まる見込み。

## 実践ポイント
- 今すぐできること（入門）  
  - ブラウザ基礎（HTMLパーサ、CSSカスケード、レイアウト、レンダリングパイプライン、JSエンジン）の学習を始める。小さなレンダラを作ることで内部理解が深まる。  
  - RustやWasmの基礎を押さえる。新しいブラウザ系実装はRustで書かれることが増えている。  
  - マルチエージェント／ワークフローの設計思想（役割分担、評価ループ）を学び、社内のR&Dで小さな実験を回す。

- 実務上の注意点  
  - 自動生成コードは必ずレビュー、テスト、セキュリティスキャンを通す。保守性の計画（ドキュメント、テストカバレッジ、責任者）を前提に採用すること。  
  - モデル出力の出自（学習データ）とライセンスリスクを評価する。  
  - 長期的には「AIで作る→人が保守する」フローを前提に工数見積りを行う。初期生成より修正・保守の方がコスト高になる可能性が高い。

結論：今回の実験は技術的な驚きと可能性を示したが、「使える」ソフトウェアにするにはまだ人間の設計力・品質管理が不可欠。日本の現場では「試す価値あり、即本番はダメ」が現実的な判断だ。
