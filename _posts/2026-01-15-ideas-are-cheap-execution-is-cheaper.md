# Ideas are cheap, execution is cheaper - アイデアは安い、実行はもっと安い
アイデアは無料化した今、勝ち残るのは「何を作るか」を見抜き、速く学習できるチームだ。

## 要約
AIコーディングエージェントによって「実装コスト」が急落した。重要なのは実装よりも問題選択、センス、反復速度、そして流通（ディストリビューション）である。

## この記事を読むべき理由
日本でもAIツールの普及が進み、誰でも短時間で機能を作れる時代が来ています。これからの差別化は「早く作る力」ではなく「正しい問題を選び、ユーザーに触れさせて素早く学ぶ力」です。特にスタートアップやプロダクト担当者、個人開発者に直接役立つ視点が得られます。

## 詳細解説
- コア原則  
  実装速度はコモディティになった。価値を生むのは次の4点：問題選択（Problem selection）、センス（Taste）、学習・反復速度（Iteration speed）、流通（Distribution）。

- 問題選択フレームワーク  
  アイデアを手放しで作る前に自問する4つ：
  ```markdown
  - [ ] 自分が実際に困っている問題か？
  - [ ] 直接リーチできる候補ユーザーが100人いるか？
  - [ ] 今日払ってくれる人がいるか？
  - [ ] コアバリューを一文で説明できるか？
  ```
  いずれかを満たさないなら棄却する勇気が重要。

- 迅速な構築プロトコル  
  エージェントに対しては「実装手順」ではなく「期待するユーザー体験」を示す。コード・テスト・ドキュメントを同一プロンプトで要求し、完璧を待たず即デプロイする。エージェントは見積もりを長めに言いがちなので「今やる」姿勢が肝。

  例：
  ```bash
  # 良い指示例（成果を説明）
  # "ユーザーがバグを報告 → AIがトリアージ → 自動でPRを作成"
  ```

- 反復と学習速度の最適化  
  Build → Ship → Learn → Adapt を「時間単位」で回す。目安：アイデア→デプロイは数時間、最初のユーザーフィードバックは48時間以内。定量的な指標で判断し、希望ではなくシグナルでピボット／停止を決める。

- ディストリビューション優先の開発  
  製品を作る前／同時に問題を投稿して反応を測る。オープンに進捗を出し、既存のオーディエンスへローンチする。アイデアは数時間でコピーされる前提で、スピードとネットワークが防御手段になる。

- 避けるべきアンチパターン  
  事前に磨き込みすぎる、ステルスで隠す、技術的満足で作る、ユーザー未確認で機能を増やす、保守的な見積りで始める、など。

## 実践ポイント
- まず問題の検証に30分使う：上の4チェックを試す。  
- エージェントへは「ユーザーの流れ」を説明して、テストとドキュメントも依頼する。  
- 24時間以内に最低限の機能をデプロイし、48時間以内にユーザーフィードバックを得る仕組みを作る。  
- 発信は日本語で早めに：Qiita、Zenn、Twitter/X、LinkedIn、コミュニティ（Slack/Discord）で反応を見てから作る。  
- 計測する主要指標：アイデア→デプロイ時間、デプロイ→最初のフィードバック時間、週あたりの反復回数、殺したアイデア比率。

日本市場では企業側の導入スピードや言語障壁が差別化要因になり得ます。AIで「何でも作れる」今こそ、正しい問題を見つけて迅速に学ぶ訓練が、最も価値あるスキルになります。
