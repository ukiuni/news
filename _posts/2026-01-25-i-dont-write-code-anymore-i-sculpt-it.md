# I don't write code anymore – I sculpt it - コードを書かない、私は彫る
AIが書いたコードを職人のように「彫る」時代 — エンジニアの新しい役割

## 要約
AIコーディングツールが最初の実装を自動生成する一方で、人間はその出力を精査・削ぎ落とし・再構築する「彫刻家」役に移行している、という主張。

## この記事を読むべき理由
日本でもAI支援開発の導入が進む中、単にコード生成を受け入れるだけでは品質や保守性が損なわれる。今後は「AIとどう協働して良いコードを作るか」が現場の差を生むから。

## 詳細解説
- 流れ：明確なプロンプト／仕様 → モデル（例：Claude-code）が初期実装を生成 → 人間が「彫刻」する。  
- モデルの得意/不得意：標準的で洗練された抽象化や既存パターンの再利用は得意だが、創造的な解法提示や批判的判断（不要な設計を拒否する）は苦手。多くの場合、動くが冗長なコードや未使用の枝が残る。  
- 彫刻の具体作業：diffを精査して意図を確認、冗長行や重複の削除、責務分離と再利用可能なパターンへの整理、テストとドキュメントの強化。これらは単なるコード修正ではなく「設計の整形」であり、主観的な良し悪し（読みやすさ、拡張性）を決める工程。  
- 開発プロセスへの影響：プロンプト設計・自動テスト・静的解析・CIルール・コードレビューがより重要になる。AIは初稿を早く作るが、人間が最終的な品質ガードを担う。

## 日本市場との関連性
- 多くの日本企業はレガシー資産と高い品質要件を抱えているため、AIに頼り切ると保守負荷が増大する恐れがある。一方で人手不足を補う効果は大きく、ガバナンスと職人としての「彫刻力」を育てることで生産性向上が期待できる。  
- セキュリティ/コンプライアンスや社内コーディング規約の適用は国内企業で特に重要。企業向けモデルやオンプレ運用を検討すべき。

## 実践ポイント
- 明確な仕様（期待する入出力、性能制約、設計方針）を最初に用意する。  
- まずテストを生成させて「動くこと」を確認する（テスト駆動の確認プロセス）。  
- AI出力は「初稿」と割り切り、必ずdiffを詳細にレビューする。  
- 不要なコード・重複は積極的に削除し、責務ごとにモジュール化する。  
- リンター、型チェック、セキュリティスキャンをCIに組み込む。  
- プロンプトや生成物の履歴を残し、再現性と説明責任を確保する。  
- 社内で「彫刻スキル（リファクタリング、設計判断）」を育成するトレーニングを導入する。

短い結論：AIは素早い原型を作るが、良いソフトウェアにするには人間が彫り上げる工程が不可欠。
