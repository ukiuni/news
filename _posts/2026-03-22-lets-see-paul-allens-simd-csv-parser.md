# Let's see Paul Allen's SIMD CSV parser - ポール・アレンのSIMD CSVパーサを見てみよう
CPUの幅でCSVを一気に掃く――SIMDとビット操作で「64バイトずつ」解析する魔法

## 要約
Paul Allen流のCSVパーサは、SIMD命令でバイト列を並列分類→ビットマスク化→累積XORで引用内外を判定し、本当に意味のある区切りだけを取り出すことで高速化している。

## この記事を読むべき理由
日本のデータ現場では巨大なCSVが頻出します（ログ、ECデータ、マーケ等）。単純な逐次処理では限界が来るので、SIMD的アプローチは実運用での劇的な効果が期待できるため必読です。

## 詳細解説
- 全体像：CSV解析は「バイトごとの分類 → 偽の区切りを除去 → 区切り位置の収集」の3段階。これをバッチ（例：$64$バイトや$16$バイト）で一度に処理するのが要点。
- ニブル（半バイト）でのベクトル化分類：8ビットを高ニブル/低ニブルに分け、16エントリのルックアップ表（pshufb / vqtbl1q_u8 相当）を2回使い、ANDで正しいクラスだけ残す。これにより16/32/64バイトを分岐なしで一括分類できる。注意点：高・低ニブルの組み合わせで不要な交差（偽陽性）を生まないようグリッドを設計する必要あり。
- ビットマスク化：分類結果を各クラスごとにビット列（u64単位）に圧縮。1ビット＝1バイト位置なので、64バイトが1ワード。
- 引用内/外判定（フィルタリング）：引用(")のビット列に対して“prefix XOR”を取り、各位置が引用内か外かを判定する。数学的には各位置の外部フラグは
$$
o_i = \neg\left(\bigoplus_{j=0}^i q_j\right)
$$
（$q_j$は位置jの引用ビット）。64ビットのプレフィックスXORはシフト＋XORチェーンで6ステップ、あるいはARMでのcarryless multiply（vmull_p64）で1命令に短縮可能。
- 境界抽出：クリーンになったカンマ/改行のビットマスクから clz/ctz 等で1ビットずつ位置を取り出す。チャンク間では「前チャンクの引用パリティ」を持ち越し、プレフィックスの反転が必要になることに注意。
- アーキテクチャ：元実装はARM NEON（vqtbl1q_u8, vmull_p64 等）だが、x86にも等価命令がある（pshufb 等）。実機での命令やメモリレイアウト差を意識して最適化する。

## 実践ポイント
- まず simdjson 論文を読み、概念を理解する（JSON→CSVへ応用できる）。
- 小さく試す：RustでNEON/x86のintrinsicsを使ったプロトタイプを作り、ベンチマーク（実データ）で効果を確認する。
- 実装注意点：RFC4180のエッジ（CRLF、エスケープ連続、空フィールド）を検証・テストすること。ニブルルックアップのグリッド設計で偽陽性が起きないかチェック。
- デプロイ先を考慮：Mac M1/M2、AWS Graviton、モバイル等ARM系が多いならNEON最適化が有効。x86サーバならpshufb等の最適化を検討。
- 既製ライブラリも検討：自前実装は学びが大きいが、まずは simdjson 系ライブラリや既存のSIMD対応CSVライブラリで運用性を比較する。

参考に、64ビットプレフィックスXORの典型実装（Rust風）を示します。

```rust
// rust
fn prefix_xor_shift(mut x: u64) -> u64 {
    x ^= x >> 1;
    x ^= x >> 2;
    x ^= x >> 4;
    x ^= x >> 8;
    x ^= x >> 16;
    x ^= x >> 32;
    x
}

// ARMで可能なら vmull_p64 を使うと1命令で済む（実装はアーキ依存）
```

この記事で紹介した手法は「分岐を減らし、データ幅で勝負する」考え方の典型です。まずは小さなCSVでプロトタイプ→ベンチ→本番導入をお勧めします。
