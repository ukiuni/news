# Above the API: What Developers Contribute When AI Can Code - APIの上で：AIがコードを書くとき、開発者が提供する価値
AIがコードを書く今、速さだけでは食えない――「上流」に立つエンジニアになるための条件

## 要約
AIは素早く「動くコード」を作るが、長期的に価値を生むのは設計・検証・維持を担える人間の判断力である。AIを使って速く作るだけでは、理解やマスターが落ちる可能性がある。

## この記事を読むべき理由
日本の現場ではレガシーや複雑な業務要件が多く、単に機能を作るスピードより「あとで直せる・拡張できる」設計能力が求められる。AI導入が進むほど、上流スキルの重要性が相対的に高まるため、今から身につける価値が大きい。

## 詳細解説
- 新しい分断：「Below the API（AIが実行する領域）」と「Above the API（人間が担う領域）」
  - Below（AI向き）: ボイラープレート生成、単発の実装、明確な仕様に従う作業。速いが文脈理解が浅い。
  - Above（人間向き）: アーキテクチャ判断、複数コンポーネント横断の整合性検証、技術的負債の評価、いつリファクタ／再構築するかの判断。
- v1/v2の現実：AIは新規（v1）を速く作れるが、v2以降の保守・拡張（既存コードの理解・整合性維持）が市場で高く評価される。
- 技能の振り分け
  - コモディティ化されるスキル：新規機能の実装、テンプレート適用、単一ファイルのコード生成。
  - 代替困難なスキル：複雑なデバッグ、設計の整合性保持、将来的な影響を予測する判断力。
- 検証の難しさ：Anthropicの実験では、AI支援グループは速かったが理解度が下がり（習得テストで約17%低い）、「AIに任せるだけ」だと習熟が進まない。一方で「理解のためにAIを使った」参加者は高得点を出した。
- ナレッジ伝播の問題：過去の経験を持つ先達が若手に「疑う・検証する」習慣を伝えてきたが、AIネイティブ世代が増えるとその伝承が薄れるリスクがある。
- ツールの本質：現行の大規模言語モデルは出力の一貫性（コヒーレンス）を最適化しており、「正しさ」を自動で担保する設計ではない。だから人間の「ジャッジ層」が不可欠。

## 実践ポイント
- AIに「実装して」もらうだけでなく、必ず「なぜこれで良いか」を質問して理解する。  
  例：「この設計が将来どんな課題を生む可能性がありますか？」
- レガシー／既存コードに触れる：グリーンフィールドばかりでなく、既存システムの修正やリファクタに取り組む。
- メンターを探す：AI以前の保守経験がある先輩から検証と判断の習慣を学ぶ。社内外のコードレビューを活用する。
- 検証ルールを作る：単にテストが通るかでなく「この選択は拡張性・保守性にどう影響するか」をチェックリスト化する。
- ドキュメント化と公開の習慣：設計理由を明文化してチームで共有すると、後で検証や学習に使える。
- AI活用法の切り替え：スピードが目的ならAIを活用、学びたい・判断力を養いたいならAIを「教える相手」や「説明してくれる助手」として使う。

（短期的にはAIで開発効率が上がるが、長期的な市場価値は「上流で判断できる能力」に移る。今のうちにその力を育てよう。）
