# Rust at Volvo Cars - ボルボ・カーにおける Rust
安全性と性能を両立する「自動車ソフトウェアの次の一手」──ボルボの現場から見える導入のリアル

## 要約
NDC TechTown 2025での「Rust at Volvo Cars」講演の主旨を整理すると、ボルボが安全性・信頼性を高めるためにRustを実務へ取り入れ始めており、その現場での利点と課題、導入手順が示されていました。

## この記事を読むべき理由
自動車業界はソフトウェア比率が急速に上がり、メモリ安全・低レイテンシ・保守性が最重要課題になっています。ボルボという安全重視のメーカーがRustに注目していることは、日本の自動車メーカーや電子部品サプライヤーにも直接関係します。これから組込み・車載ソフトを扱う技術者にとって、実践的な導入知見を得られる内容です。

## 詳細解説
- なぜRustか  
  Rustはコンパイル時に所有権と借用の仕組みでメモリ安全を保証し、C/C++で起きがちなヌルポインタ参照やダングリングポインタによるバグを減らせます。車載ソフトでは安全性（ISO 26262に関連する設計）と長期運用性が鍵で、Rustの言語特性はここに響きます。

- どの領域で使われるか  
  ボルボで想定されるユースケースは、センサー前処理、通信スタック、セキュリティ関連モジュール、テレマティクスやツールチェーン周りのユーティリティなど。「フル車載ECUの全面移行」ではなく、まずは局所的（安全境界が明確なモジュール）な採用から始めるのが現実的です。

- 技術的ハードルと対策  
  - no_std環境、リアルタイム性、メモリ割当て：組込み向けのcrate（embedded-hal、cortex-mなど）やリアルタイムフレームワークの活用が必須。  
  - 外部Cコードとの相互運用（FFI）：既存のC/C++資産と混在させるための境界設計と「unsafe」ブロックの厳格なレビュー運用。  
  - 認証・安全規格対応：言語選定だけで安全証明になるわけではなく、テスト、静的解析、形式手法、トレーサビリティを含めたプロセス整備が必要。  
  - ツールチェーンとCI：クロスコンパイル、ハードウェア上でのテスト自動化、fuzzingやメモリ監査（cargo-audit, MIRI, Clippyなど）の導入が効果的。

- 組織面の導入ノウハウ  
  小さなチームでのPoC → 成果を示して段階的に拡大、社内教育（Rust基礎＋unsafeの扱い）、既存資産とのインターフェースを明確にすることが推奨されます。

## 日本市場との関連性
日本の自動車メーカーや部品サプライヤーは、安全規格や長期サポート要件が厳しく、ソフトウェア品質の向上は喫緊の課題です。ボルボの事例は「安全志向の企業がRustを選択肢に入れている」というシグナルであり、国内メーカーやTier1サプライヤーにとっても参考になります。組込み系のSIerや車載ソフト開発チームは、今のうちに小さなプロジェクトでRustを試して知見を蓄積する価値があります。

## 実践ポイント
- 小さく始める：安全クリティカル度の低いユーティリティやツールから導入。  
- ツールチェーン整備：rustup, cargo, クロスコンパイル環境をCIに組み込む。  
- unsafeポリシー：unsafeブロックを最小化し、コードレビュー基準を定める。  
- テストと解析：ユニット＋ハードウェアインテグレーションテスト、fuzzing、cargo-audit、MIRIを導入。  
- 参考セットアップ例（最初の一歩）：

```bash
# rustup と stable toolchain のインストール
rustup default stable

# 組込みターゲットを追加（例: ARM Cortex-M）
rustup target add thumbv7em-none-eabihf

# プロジェクト作成とクロスビルド
cargo new my_ecu
cd my_ecu
cargo build --target thumbv7em-none-eabihf
```

学習リソース：The Rust Programming Language、The Embedded Rust Book、cargo-audit や Clippy の公式ドキュメント。
