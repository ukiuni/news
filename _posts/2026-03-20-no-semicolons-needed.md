# No Semicolons Needed - セミコロンは不要
セミコロンなしで「改行だけ」で文を区切ると何が起きる？言語設計者が11言語を比較して見つけた落とし穴と実務での対処法

## 要約
著者は自作言語Rotoのために「セミコロン不要」をどう実装するか検討し、Python/Go/Kotlin/Swiftなど11言語の設計とトレードオフを比較している。各言語は改行処理に異なる解を持ち、誤解やツール依存を生むことが分かった。

## この記事を読むべき理由
改行ルールはコードの見た目だけでなく、パーサーの挙動、エラーメッセージ、フォーマッタやリンタの挙動に直結する。日本の現場で複数言語を扱う／言語設計やツール開発に関わる人は、よくある落とし穴と対策を知っておくべきだ。

## 詳細解説
- 目的と問題提起  
  セミコロンを強制しない言語は可読性向上を狙うが、改行だけで文の終端を決めると曖昧さが生じる。特に「単項演算子が前後で意味が変わる」「関数呼び出しの ( が行頭に来るかどうか」で解釈が分かれる例が多い。

- Python：行＝文、ただし結合ルールあり  
  - 原則：1行1文（logical line）。物理行を結合するのはバックスラッシュ `\` または括弧類での暗黙結合。  
  - 長所：ルールが明快で位置情報（インデント）で不整合を検出しやすい。  
  - 短所：括弧でラップする癖が生まれ、ツール（black等）で丸められる傾向。

  例：
  ```python
  # Python
  y = (long_func(1, x)
       + long_func(2, x))
  ```

- Go：字句解析器が改行でセミコロンを自動挿入  
  - 実装：lexerが特定トークンの後で改行の直前にセミコロンを挿入する（識別子・リテラル・return 等）。  
  - 長所：ルールは単純。  
  - 短所：コメントや改行位置で意図せずセミコロンが入りパースエラーになり得る。標準のフォーマッタ・スタイルに厳格でない書き方はエラー化する。

  例：
  ```go
  // Go
  x := 4
  y := 2 * x - 3
  // lexer は行末に自動でセミコロンを挿入する
  ```

- Kotlin：改行を文法に埋め込む・多数の例外  
  - 実装：どの位置で改行が許されるかを文法に明示。演算子や参照演算子ごとに挙動が異なる。  
  - 長所：IDEと組み合わせると利用者にとって自然な挙動を目指せる。  
  - 短所：仕様が複雑でカスタムパーサーやツールで再現しづらい。

- Swift：可能な限り行継続してパース、ただし例外あり  
  - 実装：改行をあまり重要視せず「できるだけ先まで読む」方式。ただし関数名と `(` が別行だと継続しないなどの特別ルールがある。単項／二項演算子の空白ルールで解釈が変わる。  
  - 長所：直感的で単純。  
  - 短所：演算子の空白による解釈差で微妙なバグや警告を誘発することがある。

- その他（Gleam, Lua, Ruby, R, Julia, Odin 等）もそれぞれ異なる妥協をしており、共通のテーマは「単純さ vs 柔軟さ vs ツール依存」。

## 実践ポイント
- 明示的に安全にしたいなら括弧／カンマで区切る習慣を付ける（特に複雑な式）。  
- チームでは言語ごとの公式フォーマッタ（black/go fmt等）を必須化してスタイル起因のエラーを排除する。  
- 単項／二項で意味が変わる演算子（`-` など）は行分けや空白に注意し、意図が曖昧な書き方を避ける。  
- 言語設計・パーサ実装時は「字句解析で挿入するか」「文法に明記するか」「行継続で読むか」の三択を明確に評価し、ツール（フォーマッタ・IDE）連携の容易さを重視する。  
- 新しい言語を学ぶ際は「改行が文を終えるか」のルールを最初に把握しておくとデバッグ時間が減る。

（参考：元記事は Roto 言語設計の観点から Python, Go, Kotlin, Swift など11言語の改行／終端ルールを比較した調査）
