# An ode to houseplant programming (2025) - ハウスプラント・プログラミング賛歌
自分だけの「鉢植えコード」でいい。完璧さを手放して生産性と創造性を育てよう

## 要約
「houseplant programming」は、自分専用の小さなソフトを愛でる考え方。汚くても“自分の環境で動けばいい”という健全な割り切りを称賛するムーブメントです。

## この記事を読むべき理由
日本でも業務ツールの自作や社内ユーティリティ、ハックデイ成果物が増えている今、完璧主義に縛られず価値ある小さな成果を共有・再利用するマインドセットは実務効率と創造力の両方に直結します。

## 詳細解説
- 定義：houseplant programming = 「自分（ひとり）のために書いた小さなソフト」。大量ユーザーや厳格な運用を前提にしない。作者の生活やワークフローに寄り添う“鉢植え”のような存在。
- 「It works on my machine」は免罪符ではなく目的：本来の用途が作者本人なら、厳密なテストや冗長な堅牢化は必須ではない。
- prototype と bouquet の使い分け：プロトタイプは将来の発展可能性を示す一方、bouquet programming は一回限りの用途（解析スクリプト、贈り物用のワンショットツール等）で保守不要。
- 特性の類比（原文のメタファー）：
  - 長寿性と廃棄（うまくいけば育て、駄目なら捨てて再利用）
  - 伝播（コードの“切り取り”を共有して他人が再利用）
  - 汎用化しない自由（全環境対応を目指さない）
  - 楽しさと完璧主義の克服
- 実践的要素：作者はバッジ（Houseplant / Bouquet）でプロジェクトの性質を明示する案を提示。必要に応じてプロダクト化するかどうか判断する文化が重要。

## 実践ポイント
- リポジトリに一行で「Houseplant」/「Bouquet」タグを付けて期待値を明示する（README冒頭でOK）。
- 最低限のセットアップ手順（依存・実行コマンド）をREADMEに書く。5分で試せることが大事。
- ワンショット解析やスクレイピングは「bouquet」と明記して保守責任を切る。
- 共有は躊躇しない：Qiita/GitHubで“使い方メモ”を残すだけで誰かの参考になる。
- プロダクト化判断基準：複数人が必要とする／継続運用コストが発生するなら、リファクタやテスト、CI導入を検討する。

日本の開発文化でも「まず作って試す」ことを肯定するこの考え方は、個人の学習・業務効率化・社内ツール文化をより健全にします。
