# A program that outputs a zip, containing a program that outputs a zip, containing a program... - プログラムがzipを出力し、その中に再帰的に同じ振る舞いをするプログラムを入れる
無限ループに見せかけた“自己複製zip”の仕組みと実務的な注意点

## 要約
あるプログラムが、実行するとzipファイルを生成し、その中に「同じことをするプログラム」を入れておく――という自己複製／ネストを実現するアイデアを解説します。概念実証はフォーマットの理解と入出力操作の工夫で成り立ちます。

## この記事を読むべき理由
自己複製やブートストラップ的な作品は、ファイルフォーマットの深い理解・デバッグ技術・サプライチェーンや圧縮に関するセキュリティ認識を高めます。日本の開発現場でもビルド再現性、アーティファクト管理、マルウェア検出対策などに直結する知見です。

## 詳細解説
- アイデアの本質  
  プログラムAは実行時にzipを作り、その中にプログラムB（通常はAと同種のスクリプト）を格納する。Bを実行すると同様にzipを出力し中にBを入れる――という循環を作れる。深さは制御可能で、実際に「無限」にはできない（ディスクや時間の制約があるため）。

- 実装の要素  
  - 入出力（I/O）：自プロセスのソースを読み取り、zipエントリとして書き込む。  
  - zipの構造：ローカルファイルヘッダとセントラルディレクトリを正しく作ること。多くの言語は標準ライブラリでzip作成を提供するので、それを使えば楽。  
  - ステータス（再現性）：タイムスタンプや圧縮レベルを固定するとバイト単位の再現が容易になる。  
  - 実行形式：スクリプト（Python/Bash）なら中のファイルをそのまま起動できるが、実行権限やシェルの互換性を考慮する必要がある。Windowsでは自己解凍実行形式やバッチ処理が絡む。  
  - サイズ制御：ネストするとファイルは指数的に増えやすい。中身を定数化（小さなスタブ）する、深さを数に制限するなどが必要。

- 応用とリスク  
  教育的な実験、コードゴルフ、パッケージングの検証などに有効。ただし、意図せず「zip bomb（圧縮爆弾）」的な挙動を作るとアンチウイルスやCI/CDパイプラインでブロックされる。サプライチェーンの攻撃面も増えるため運用ルールが重要。

## 日本市場との関連
日本ではIoTや組み込み、企業の連続的デリバリ環境でアーティファクト管理が重要視されています。こうした自己複製的な手法を理解すると、意図しない再帰的出力によるCI停止やアーティファクト肥大化、セキュリティアラートを未然に防げます。大学や社内ハッカソンの教材にもなり得ます。

## 実践ポイント
- まずは小さな深さ（例：1〜3）で試す。  
- zip作成は標準ライブラリを使い、タイムスタンプと圧縮率を固定する。  
- 深くネストする前にディスク使用量と実行時間の見積もりを取る。  
- CIや組織のセキュリティポリシーに従い、アンチウイルス誤検知や配布規則に注意する。  
- 学習用途なら、zipの中身を一覧するコマンド（unzip -l 等）で構造を確認しながら進める。
