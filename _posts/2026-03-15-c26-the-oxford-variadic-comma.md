# C++26: The Oxford Variadic Comma - C++26: オックスフォード可変長コンマ
「...の前にカンマを入れるだけ」でC++の曖昧さが消える理由

## 要約
C++26では、引数リストの省略記号（ellipsis）に先行するカンマを省略する書き方を非推奨にします。これによりC互換性が強まり、パラメータパックとの混同が減り将来の言語拡張の余地が生まれます。

## この記事を読むべき理由
C++を扱う日本の開発現場でも、古い表記やテンプレート周りの曖昧さはバグやレビュー時間の原因になります。今回の変更は既存コードを壊さずツールで自動修正できるため、コード品質向上の小さな一歩としてすぐ取り入れられます。

## 詳細解説
ポイントは「Cスタイルの可変長引数（printf系の...）」と「テンプレートのパラメータパック（Ts...）」が文法的に混同されやすいことです。現在、C++は次の2通りを許容します。

```cpp
// C互換（推奨）
void foo(int, ...);

// カンマ無し（古いC++由来、非推奨予定）
void foo(int ...);
```

テンプレート周りの例で混乱が顕著になります。

```cpp
// パラメータパック（正常）
template<class... Ts>
void f(Ts... args);

// 見た目は似ているが、Tはパックでない場合のellipsis（非推奨対象）
template<class T>
void h(T...);
```

さらに奇妙な表記も可能です（現行規則では成立します）：

```cpp
// 例: 六つのドットに見えるが意味はパックの後にellipsis
void g(auto ......); // = (auto..., ...)
```

C++26の提案（P3176R1）は「カンマ無しのellipsisパラメータ」を非推奨にします。つまり

- 非推奨: void f(int ...);
- 推奨（C互換）: void f(int, ...);

非推奨なので既存コードは直ちに壊れません。多くのケースは単純にカンマを挿入するだけで自動変換可能です。加えて、この整理は将来の機能提案（例: 同種可変長引数など）が新しい構文を使えるようにするための準備にもなります。

## 実践ポイント
- 既存コードベースで該当箇所を検索してカンマを挿入する：正規表現やASTベースのツールを使うのが確実。
  - 例: 単純なsedでの一括修正（注意：文脈解析は行わないのでバックアップ推奨）

```bash
# 単純置換の例（プロジェクト全体で慎重に使うこと）
git grep -nP '\b([A-Za-z_]\w*)\s+\.\.\.' | cut -d: -f1 | xargs -I{} sed -i.bak 's/\([A-Za-z_]\w*\)\s\+\.\.\./\1 , .../g' {}
```

- clang-tidy/clang-formatや自前のASTツールで安全に自動修正を行う。非推奨警告を出すコンパイラフラグが有効ならビルドで検出する。
- 新規コードでは最初からカンマを入れる習慣を付ける（void foo(int, ...);）。テンプレートパックは明示的に class... Ts のように書く。

この変更は小さいですが、可読性と互換性の観点で即効性のある改善です。プロジェクトのスタイルガイドに追記しておくと良いでしょう。
