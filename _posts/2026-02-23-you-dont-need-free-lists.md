# You don't need free lists - フリーリストは要らない（ビットプールで高速・省メモリに）

フリーリストを捨てて、ビットだけでスロット管理する「Bit Pool」アイデア。メモリ効率が良く、挿入・削除・探索が最悪ケースでもO(1)に近い（事前にバッファを確保した場合）。

## 要約
フリーリストの代わりに、1ビット/スロットのビットマップ＋多層ビット要約（L1, L0）を使うことで、低オーバーヘッドかつ高速なスロット割当て・解放を実現する手法の紹介。

## この記事を読むべき理由
フリーリストは簡単だが、スロット配置がランダム化しメモリ局所性を落とす。ゲーム開発やリアルタイム系、組み込みなど日本でも需要の高い用途で、より高速・省メモリな代替を検討できる。

## 詳細解説
- 基本アイデア：各スロットに1ビット（使用中/未使用）を割り当て、64ビット単位の配列（L0）で管理。前提は $N \bmod 64 = 0$。
- ビット検索効率化：x86の tzcnt（trailing zero count）命令を使い、~used[i] の最下位1ビット位置を1サイクル程度で得る。単一u64から空きビットを即座に見つけられる。
- 多層要約（2レベル）：L0の各64ブロックについて「満杯かどうか」を示す1ビット配列（L1）を追加。L1を先に検索すれば、1つの tzcnt で4096スロット（64×64）をスキップ可能。
- 拡張性：さらに上位レイヤーを追加すれば、1命令で $64^k$ スロットを探索可能（例：$$64^3 = 262144$$）。実務上は数万スロット程度で十分なことが多い。
- メリット：およそ1ビット/スロットのオーバーヘッド、低レイテンシ、低分散（低インデックス優先で割当て→局所性向上）。SIMD AoSoA構成と相性が良い。
- 注意点：事前確保（preallocate）前提。複数スレッドでのロックフリー化は可能性があるが正当性確認が必要（原子的操作とメモリ順序を慎重に設計すること）。

簡潔な探索例（概念コード）：

```Odin
find_empty_slot :: proc (pool : Bit_Pool(N)) -> (index:int, ok:bool) {
    // L1で空きブロックを探す
    for i in 0..len(pool.l1) {
        l1slot := int(count_trailing_zeros(~pool.l1[i]))
        if l1slot != 64 {
            l0_index := 64*i + l1slot
            l0slot := int(count_trailing_zeros(~pool.l0[l0_index]))
            if l0slot != 64 { return l0_index*64 + l0slot, true }
        }
    }
    return -1, false
}
```

## 実践ポイント
- 小さく始める：$N$ を64の倍数で確保し、まず1レイヤ（L0）で試す。パフォーマンスを測定してからL1を追加。
- 命令サポート確認：ターゲット環境で tzcnt（もしくは同等のintrinsic）が使えるか確認する。
- メモリ局所性を活かす：割当てが低インデックス側に偏るため、バースト割当ての局所性が改善される場面で効果大。
- SIMD/AoSoAとの併用：L0ビットマスクはSIMDマスクと相性が良い。パーティクルやエンティティ管理に有効。
- 並列化は慎重に：ロックレス化は魅力的だが正当性証明とベンチが必要。まずはミューテックスやスピンで安全に実装して評価する。

元記事のアルゴリズムはシンプルで実用的。フリーリストの欠点に悩んでいるなら、まず試してみる価値が高い。
