# Anthropic blocks third-party tools using Claude Code OAuth tokens - AnthropicがClaude CodeのOAuthトークンでのサードパーティツールをブロック
「あなたのClaude契約が突然使えなくなるかもしれない」――Anthropicの“ループホール封鎖”が開発者コミュニティに波紋を広げる

## 要約
AnthropicがClaude CodeのOAuth経由でサードパーティ製のターミナル型コーディングツール（例：OpenCode）が自社のPro/Maxサブスクリプションを利用する仕組みを事実上遮断した。事前告知なしの変更でユーザー反発と競合サービスへの追い風を招いている。

## この記事を読むべき理由
日本の開発現場でも、ターミナル上でLLMを使うワークフローやサードパーティ製の開発ハーネスが普及しつつある。今回の出来事は「サブスク契約で使えるはずの機能がプロバイダの判断で止められるリスク」を可視化しており、ツール選定や契約交渉、運用設計に直接影響します。

## 詳細解説
- 背景（vibe codingとターミナル型エージェント）  
  ターミナル上でプロンプト→LLM応答→ツール実行を自動でループする“vibe coding”型の開発エージェントが主流化。多くのクライアントはモデル選択と認証（APIキーやOAuth）を提供し、LLMを“知能”として呼び出す役割を担う。

- 仕組みの“抜け道”  
  一部のクライアントは、ユーザーが持つClaude Pro/MaxのサブスクリプションをOAuth経由で使えるようにしていた。内部的にはクライアント側の「システムプロンプト」にClaude Codeを示す特定フレーズが含まれているかで判定され、その条件下でサブスク経由の安価な利用が可能になっていた。

- Anthropicの対応と理由付け  
  2026年1月9日、Anthropicはその判定を変更してサードパーティクライアントからのリクエストを検出・拒否するようにした。公式の事前告知はなく、社内や社員の個人アカウントでのやり取りが事実上の説明となった。Anthropic側は「サードパーティ製ハーネスが特異なトラフィックやサポートの複雑化を招く」との理由を挙げている。

- ビジネス的影響  
  Anthropicはモデル提供に留まらず、サブスクの価値連鎖を自分で握ろうとしている。だが今回の措置は既存ユーザーの反感を買い、OpenAIなど競合がオープンサポートを打ち出す口実にもなった。結果として短期的には自社主導の価値確保を図れても、長期的な評判・顧客流出リスクを高める可能性がある。

## 実践ポイント
- 自社や個人の開発ワークフロー確認  
  Claudeのサブスクを第三者クライアント経由で使っている場合は、突然使えなくなるリスクを想定して代替手段（APIキー経由、別プロバイダ、セルフホストなど）を用意する。

- 契約とSLAの整備  
  ベンダー依存が業務リスクになる場合、契約にサポートや互換性に関する明確な条項を盛り込む。サブスク型サービスの利用規約やAPI変更ポリシーも定期チェックする。

- ツール開発者への注意点  
  認証フローはユーザーに明示し、プロバイダ側の仕様変更に柔軟に対応できる設計（フェールバック、ユーザー通知）を組み込む。

- 競合とエコシステム動向を監視  
  主要プロバイダの戦略（ロックイン志向かオープン連携か）は短期間で変わる。日本のプロダクト開発でもそれが影響するため、技術選定だけでなくビジネス面での判断材料として継続監視を。

短くまとめると、今回の件は「便利さの裏にあるサブスク依存のリスク」と「プロバイダの戦略が開発者体験に直結する現実」を浮き彫りにしました。自分の開発フローと契約を今一度確認しておく価値があります。
