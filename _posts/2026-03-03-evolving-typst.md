# Evolving Typst - Typstの進化
Typstが0.xのままでも「壊せる」ようにする互換性戦略とは？

## 要約
Typstは既に実務で使えるが、将来的な「破壊的変更」をどうやってやさしく導入するかを議論している。提案は「ターゲット互換モード」と自動移行ツールで、エコシステムへの衝撃を緩和するもの。

## この記事を読むべき理由
日本でも技術文書や論文、レポート作成にTypstを使う例が増えている。互換性戦略は、パッケージ依存や社内ドキュメントの保守コストに直結するため、今後の採用判断や運用方針に大きく影響する。

## 詳細解説
- バージョン問題：Typstは0.xでありつつも実用に耐えるが、SemVerを前提に1.0を出すなら「破壊的変更」の定義と保証が必要。明確な線引き（構文変更は明白、レイアウト差は微妙）をどう扱うかが課題。
- 提案の核心：typst.tomlでパッケージごとにコンパイラの最小バージョン(min)と「ターゲット」(target)を指定し、パッケージが開発された当時の挙動を維持してコンパイルできるようにする。これはRustのEditionに似た概念だが、永久保証ではなく「移行期間を与える」目的。
- 互換性の扱い：targetを指定しない既存パッケージには当面すべての互換性振る舞いを適用する案。ただし新規パッケージで古い振る舞いが返ってくると混乱するため、compilerフィールドは警告で必須化する方針。
- 自動移行ツール：構文的変更（例：関数パラメータ表記の再設計、数式の結合規則変更）は大量文書に影響するため自動変換ツールが有効。設計方針は誤検知の低さ、使いやすさ、CLI／Webとの統合性。
- 社会的側面：0.xで自由に壊せる一方、実際にはエコシステムが「壊される痛み」を生むため、互換性メカニズムや移行ツールへの投資は将来の1.0到達を早める投資と位置付けられる。

コード例（提案）
```toml
[package]
name = "fancy-diagrams"
version = "0.4.0"
entrypoint = "lib.typ"
compiler = { min = "0.15", target = "0.17" }
```

## 実践ポイント
- 新規/既存パッケージはtypst.tomlのcompilerフィールドを必ず記載しておく（minと可能ならtargetも）。  
- 依存パッケージを更新する前にtarget互換モードが提供されているか確認する。  
- 大規模ドキュメントは自動移行ツールの提供を待つか、旧コンパイラをロックして段階的に移行する。  
- 日本のプロジェクトでは図表や学術文書で依存が多いため、CIでコンパイラバージョン固定とターゲットチェックを導入すると安全。
