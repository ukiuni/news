# I Built a Desktop App That Commits to GitHub So I Don’t Have To Lie About Consistency - 「一貫性を偽らなくて済むように、GitHubへ自動コミットするデスクトップアプリを作った」
魅せるコミットグラフは作れるか？「忘れた」を消すローカル自動コミットの正体と使いどころ

## 要約
作者は「今日コミットし忘れた」という精神的負担を減らすために、ローカルでスケジュールしてGitHubへコミット／プッシュするデスクトップアプリを作った。目的は作業の代替ではなく、コミットという“雑務”の摩擦を取り除くこと。

## この記事を読むべき理由
- GitHubの貢献グラフを巡る心理的プレッシャーは、世界中のエンジニアに共通の悩み。日本でも面接やポートフォリオ評価で表層的な指標に振り回される場面がある。  
- 自動化で「習慣」を支える設計や認証・配布の実装課題は、実務で役立つ実践ノウハウになる。

## 詳細解説
- 動機：貢献グラフは本質的な生産性指標ではなく「vibe（見た目）」だが、それが採用や自己イメージに影響する現実がある。作者は「規律」ではなく「仕組み」で一貫性を作るアプローチを選んだ。
- 機能概要：アプリはローカルで動き、あらかじめ設定したスケジュールでコミットを作成し、リポジトリへプッシュする。ブラウザや外部cronを使わず、ワンセットで動く点を売りにしている。
- 技術的な焦点：
  - デスクトップ化：ユーザーが1回設定すれば以後自動で動くパッケージ化（例：ElectronやPyInstallerなどの技法を想定）についての学びがあったとされる。
  - Git認証：トークン管理やSSH鍵の扱いを安全に行う実装が重要。ローカルで動く分、トークンの保管・権限は最小化すべき。
  - 自動コミットの設計：空コミットやダミー変更を使うか、実際のファイル更新を伴うかで挙動が変わる。意味のあるメッセージやブランチ運用を決める設計が必要。
- 倫理と実務上の注意点：自動コミットは実作業を代替しない。採用側の浅い評価指標を暴露する手段にもなりうるが、虚偽アピールに使うと信頼を損なう。

## 日本市場との関連
- 日本企業でもオープンな活動を重視する場面が増え、公開コミット数を参照する採用慣行は存在する。だが多くの実務コードは企業のプライベートリポジトリにあり、コミット数で技術力を測るのは誤導になりやすい。  
- セキュリティ意識の高い日本の企業では、CIや社内ポリシーで外部への自動プッシュが制限される場合があるため、導入前に就業規則や社内ガイドラインの確認が必須。

## 実践ポイント
- 利用目的を明確に：習慣化の補助か、虚偽擁護か。自己管理ツールとして使うなら透明に。  
- 認証は最小権限で：PATやSSH鍵は専用に作り、不要権限を与えない。トークンはOSのシークレットストアで管理する。  
- 自動コミットは意味を残す：できれば実際の作業ログや小さな改善をコミットする運用にして、メッセージを自動化でも意味が分かるようにする。  
- 企業利用は要相談：所属先のポリシーやライセンス、機密管理に反しないか事前確認する。  
- 代替案も検討：単純なスケジューリングはgh CLI、GitHub Actions（公開リポ向け）、あるいはローカルのタスクスケジューラで実現可能。自動化のコストとメリットを比較する。

オリジナルのリポジトリと議論（倫理・採用の観点を含めた反響）も興味深いので、試す前に設計と運用ルールを決めてから導入すると良い。
