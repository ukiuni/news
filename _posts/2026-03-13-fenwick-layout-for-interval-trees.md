# Fenwick layout for interval trees - 区間木のFenwickレイアウト
最短で速い区間検索を手に入れる――Fenwick風ビットレイアウトで作る実用的な区間木

## 要約
Fenwick（ビットトリック）風のインデックス付けで区間木のノード配置を最適化すると、ノード数を削りつつビット演算で親子移動ができ、最悪時の定数を改善したシンプルで速い区間木が作れる。

## この記事を読むべき理由
$O(n\log n)$ 構築・$O(\log n + ans)$ 問い合わせという区間木の理論はそのままに、キャッシュ効率と実行定数を改善する実装テクニックを知れば、実務での大データ処理（バイオインフォマティクス、計算幾何、範囲グラフなど）やツール構築で「速くて予測可能な」挙動が得られます。

## 詳細解説
- 区間木の基本：各ノードは中心 $m$ を持ち、区間 $[a,b)$ は中心 $m$ が含まれる最上位ノードに格納される。点 $x$ の照会は $x$ を含む $O(\log n)$ ノードを探索して各ノード内のリストを絞り込む（リストは開始点 $a$ と終了点 $b$ で分けておくと絞り込みが簡単）。
- セグメント木／Eytzinger レイアウト：完全二分木を配列に詰めることでポインタ不要・キャッシュにも強い。Fenwick 木はこれを更に圧縮したバリエーションで、ビット表現がノードの範囲情報を表すので高速なビットトリックが使える。
- 本稿のアイデア：各ノードを「中心 $m$」で一意にインデックス化し、不要なノードを落として Fenwick 的に配置する。点 $x$ の問い合わせは「$m=x$ から上へ辿る」方式に変えることで、再帰や分岐を減らす。親ノードの計算は「最下位の1ビットを消して、その上のビットを立てる」などのビット演算で実現できる。
- 挿入（区間 $[l,r)$ の配置）：最上位で中心が $[l,r)$ に入るように、$r-1$ を基点に下位ビットをリセットして最適な $m$ を求める。ビットの最上位で異なる位置を見つければ定数時間で決定できる。
- 特殊ケース：$0$ 周りの境界やインデックスシフトは「座標を +1 して扱う」などでクリーンに解決でき、最終的にノード 0 を取り除くことでオフバイワンが消える。
- トレードオフ：Fenwick 配置はノード数が半分程度になり計算が単純で速いが、トップレベルのメモリ分散により中サイズでのキャッシュ局所性はセグメント木より悪いことがある。SIMD 最適化されたワイドなセグメント木には敵わない場面もある。

技術的ポイント（要点）
- 問い合わせを下→上の順にし、ビット演算だけで「次の親」を計算する。
- 各ノードに保存する配列は開始 / 終了で分け、出力時は線形に先頭から走査して打ち切る。
- サイズは次の $2^k$ に丸めるが、実際に使う座標数に応じて不要ノードは省メモリ化可能。

## 実践ポイント
- ライブラリ実装：静的な座標集合で、高頻度の点クエリがあるなら Fenwick レイアウトを検討する。GitHub に参考実装あり（元記事参照）。
- 実装改善案：ノード配列を単一確保にまとめる、ソートは必要最小限に、イテレータ走査を効率化する。
- 適用例：範囲から単一頂点へ張る「区間エッジ」を持つグラフの DFS（$O(V\log V + E)$ に収める）、CFG を持たない逆コンパイル補助、バイオデータの大規模区間検索など。
- テスト／ベンチ：実データでキャッシュ動作と分岐予測の影響を見る。小〜中規模では従来のセグメント木が有利な場合があるので比較を怠らない。

参考（概念サンプル）
```rust
// Rust風：m から親へ上る（概念的な例）
let mut m = x;
while m != root {
    process(&nodes[m]); // 中心 m に格納された区間を絞り込み出力
    let low = m & (!m + 1);            // 最下位1ビット
    m = (m & (m - 1)) | (low << 1);    // 最下位1を消して上のビットを立てる（概念）
}
```

短くまとめると、Fenwick のビットレイアウトを区間木に応用すると「ノードを減らし、分岐を減らし、ビット演算で高速化」できる――実用的で実装も短く、試して損はないテクニックです。
