# How Github monopoly is destroying the open source ecosystem - Github独占がオープンソース・エコシステムを破壊している
Githubしか見ない若者たち──大学の授業が暴いた「見えない独占」の恐怖

## 要約
ある大学教員が学生課題を採点中にGithubからIPブロックされ、過去7年分の238件の学生レポートのうちGithubを使っていないのは1件だけだったことから、Github中心主義がオープンソースの発見性・耐障害性を著しく損ねていると指摘する。

## この記事を読むべき理由
日本の企業・教育機関でもGitHub依存は深まっており、サービス停止やアカウントロック、国際的な政治リスク、ベンダーロックインによる技術的・運用的な脆弱性がそのまま業務／教育リスクにつながるため。今すぐ自社やチームの依存度を点検すべき兆候が詰まっている。

## 詳細解説
- きっかけの事例：著者は授業で学生にオープンソースへ貢献させる課題を出しており、採点のためにGitHubを閲覧していたところ短時間でIPブロックされアクセス不能になった。過去7年分238件のうち、Githubを使っていない報告は事実上1件だけだった。
- 発見バイアスと集中化の影響：開発者の多くはプロジェクト検索や貢献先探しをまずGithubで行うため、Githubに存在しないプロジェクトは「見えない」状態になりやすい。結果としてプロジェクト移行や新興フォージの成長が阻害され、エコシステム全体が一つのプラットフォームへ収束する。
- 技術的な危険性：
  - 単一障害点（SPOF）：ホスティング停止やIPブロックでアクセス不能、CIやパッケージ配布に依存していると業務停止につながる。
  - アカウントロックとデータアクセス：個人や組織のアカウント凍結でソース・通知・CIが失われるケースが報告されている。
  - 発見性とコミュニティ効果：スター／フォーク／Issue等のメトリクスがGithub上で完結するため、他のフォージはコミュニティ形成で不利。
  - ロックインの摩擦：リポジトリ移行時にIssue履歴・PR・CI設定・パイプライン等の移行コストが高い。
- 代替案とアーキテクチャ的選択肢：GitLab（セルフホスト可）、Gitea/Forgejo、Codeberg、SourceHut、Radicle（P2P）、Fossilなど。これらは分散性や透過性、自己ホスティングによる管理性を改善するが、UXやエコシステムの成熟度では差がある。
- 教育的気付き：著者は自分の課題設定が「学生にGitHub内を探させる」方法になっていたことを反省し、以降「自分が使う／使いたいプロジェクトへの貢献」を必須にして視野の拡大を図る方針に変えた。

## 実践ポイント
- まず依存度を可視化する：社内・学内で重要なリポジトリ、CI、パッケージが外部サービスにどれだけ依存しているかをリストアップする。
- ミラーとバックアップを用意する：重要リポジトリはセルフホストのGitLabやGiteaにミラーし、Issue/PR履歴のエクスポート方針を決める。
- 認証と権限を分離する：OAuth集中やSAMLプロバイダ一択に頼らず、代替ログインやローカルアカウントを用意する。
- 貢献先の選び方を教育する：学生や新人には「自分が実際に使っているソフト」のメンテナを探す、コマンドラインでgit操作する、複数のフォージを検索する習慣をつけさせる。
- 小さく始める：RadicleやSourceHutのような分散・軽量ツールを試験的に使い、運用コストと利点を比較する。
- 法務・セキュリティ視点で評価：データレジデンシーや国際政治リスクがある場合は、外部クラウド一辺倒を見直す。

この問題は技術的好みの話だけでなく、運用リスクと民主的なソフトウェアエコシステムの観点からも重要です。まずは「どこにコードがあるか」をチームで問い直すことから始めましょう。
