# A sufficiently detailed spec is code - 十分に詳細な仕様はコードである
「仕様を書けばコードになる」は幻想か？AI時代の“仕様＝コード”論を分かりやすく検証する

## 要約
AIエージェントに仕様から実装を丸投げする流行に対し、詳細な仕様は結局「コードに等しい」ため手間も誤りも消えない、という批判を示した記事の要点を解説します。

## この記事を読むべき理由
仕様ベースでの自動生成を検討しているチームや、AIを使った開発プロセスに疑問を持つ日本のエンジニア／マネージャーにとって、現実的な落とし穴と実務的対処法が得られます。

## 詳細解説
- 背景：OpenAIの「Symphony」プロジェクトは「SPEC.md」による仕様から実装を生成する例として紹介されたが、そのSPECは実際には擬似コードや実装に近い細部（DBスキーマ、リトライ／バックオフのアルゴリズム、状態遷移の疑似コードなど）で埋められている。つまり仕様が「コードに見える」。
- 誤解その1（仕様はコードより単純）：実装を正確に生成できるほど仕様を精緻化すると、結局仕様側に実装と同等の精度と労力が要求される。Dijkstraが指摘する「狭いインターフェース（＝コード的正確さ）」の必要性に合致する。
- 誤解その2（仕様はコーディングより熟考的）：短納期や効率重視の文化では、仕様作成が雑になりがちで「仕様がより丁寧」とは限らない。記事はSymphonyのSPECを「スロップ（手抜きの仕様）」と評し、AI生成物に起因する矛盾や曖昧さを挙げる。
- 実例での失敗：著者が実際にAI（Claude Code）へSPEC.mdからHaskell実装を生成させたが、多数のバグと実行不具合が発生。さらに、詳細な既存仕様（例：YAML）でも実装間で不整合が残る現実を指摘。
- 本質的結論：仕様が不十分ならAIは補完できない（GIGO）。仕様を正確にするコストは高く、納期短縮目的なら直接コードを書く方が現実的な場合が多い。

## 日本市場との関連性
- 日本企業で多い「上流で仕様を定めて下流へ渡す」体制は、AIエージェントを中抜きしても同じ問題に直面する可能性が高い。発注側が「書類で仕事を簡単にする」期待は裏切られやすい。
- 人手不足・外注コストの背景で「仕様でアウトソースする」動機は強いが、仕様品質に投資しないままAIへ委ねると保守負担が増す恐れがある。

## 実践ポイント
- 仕様を使うなら「実行可能な検証」を用意する（テストスイート、例示的データ、コントラクトテスト）。
- インターフェースは狭く明確に：曖昧さを減らすため最小限のAPI/契約を定義する。
- 小さく試す：まずプロトタイプやミニマム実装を手で作ってから仕様化する。
- 仕様作成に時間を割くか、短納期ならコードで直接実装する判断を明確にする。
- AIに任せる場合はCIで自動テストを必須化し、人間による仕様・実装レビューを残す。
