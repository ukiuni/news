# On Making - 作ることについて
「AIに作らせる」の先で、いまエンジニアが失いつつある「作る喜び」をどう取り戻すか

## 要約
Beej（Brian Hall）は、ジェネレーティブAIがもたらす効率と引き換えに「作る」という行為の満足感が失われていると指摘し、コンパイラや道具との違いを挙げつつ、自分で手を動かす価値を改めて問い直しています。

## この記事を読むべき理由
日本でもAIツールが普及しつつある今、「プロンプトで作らせる」ことと「自分で作る」ことの境界、責任・帰属、教育や採用への影響を考えるための示唆が得られます。

## 詳細解説
- AI開発分断（AI dev schism）: 手作業でのコーディングや低レイヤーの問題解決が失われる一方で、高レイヤーの設計やプロダクト完成の速度は上がる──双方に利点と欠点があるとする観察。
- 帰属と誠実さ: Beej は「AIに作らせたものを自分が作ったと言えない」と感じる。請負業者に作らせたものを「作った」と言わない感覚に近いと説明。
- プロンプトは技能か？: 有効なプロンプト作成はビジョン、判断、コミュニケーションを要するが、Beej はそれを「誰かに作らせる」行為に近いと見る。
- コンパイラとの比較: C→アセンブリ→機械語の流れはツールの自動化だが、道具（ハンマーやコンパイラ）に自分が直接手を触れて「作った」感覚が残るのに対し、LLMに自然言語で作らせるのは「管理」と捉えがち、という哲学的な差異。
- 実例: 著者はデッキ制作は外注（職人作業）として認め、短いフラッシュカードアプリは手で実装（Google SheetsのCSVエンドポイントを利用）して「自分で作った」と誇れると述べる。CコードをLLMにアセンブルさせる例も示し、グレーな領域を探る。

## 実践ポイント
- 小さくても自分で作る習慣を残す：学習ツールやミニプロダクトを手で実装して「作る感」を保つ。
- AIはアシスタントとして使う：アイデア出し・定型コード生成・ドキュメント作成はAI、設計・核となるロジックは自分で手を動かす。
- プロンプトは「リーダーシップ技能」として磨く：仕様設計やレビュー能力を高め、AIアウトプットの監督者になる。
- ライセンスと帰属のルールを明確に：チームやプロジェクトでAI生成物の扱い（帰属・ライセンス・品質基準）を合意して記録する。
- まずは実験：Google SheetsのCSVエンドポイントなど、手早く作れる連携からAI支援と手作りを両立させる。

コード例：Google Sheets の CSV エクスポート（シートIDと gid を置き換えて使用）
```bash
# bash
curl -L "https://docs.google.com/spreadsheets/d/<SHEET_ID>/export?format=csv&gid=<GID>" -o sheet.csv
```

日本の開発文化では職人性や品質が重視される傾向が強く、Beej の問いは「効率」を取り入れつつ「作る喜び」と「責任の所在」をどう守るか、という形で非常に実務的な示唆を与えます。
