# Extending C with Prolog (1994) - CをPrologで拡張する
魅力的なタイトル: CとPrologを組み合わせて「賢い」ツールを手早く作る方法 — IRQトラブル解決の実例から学ぶ

## 要約
Cの手続き型処理とPrologの探索／ルール表現を組み合わせると、設定・診断・推論系の機能を短期間で高品質に実装できる。記事はIRQ（割り込み）アドバイザのサンプルで、C↔Prolog連携の設計パターンを示す。

## この記事を読むべき理由
日本の現場でも、インストーラ、診断ツール、ルールベースの業務ロジックなど「ケース分岐が多く知識を明示化したい」用途が増えている。C/C++主体の既存資産を活かしつつ、ルール実装をPrologに任せる実装パターンは即戦力になる。

## 詳細解説
- Prologの強み（初心者向け）
  - シンボル（原子的な名前）を扱うことが容易、文字列比較や列挙を明示的に書かずに済む。
  - 統一(unification)とバックトラッキングによるパターンマッチ／探索が組み込み。
  - 動的データベース（事実のassert/retract）で知識をランタイムに追加・変更できるため、ルール追加が楽。

- 典型的な用途
  - 文法解析、ナチュラルランゲージの前処理、エキスパートシステム（診断・設定アドバイザ）、価格・構成ルール、業務ルールの表現。

- CとPrologの役割分担（設計パターン）
  - Prologは「問い合わせを受けて推論するデータベース／知識層」として動く（例: main がクエリを起点）。
  - CはI/O、OS/ハードウェアアクセス、GUIなど副作用を伴う処理を担当する。Prologが直接できない機能はC側で拡張述語（predicate）として実装する。
  - 双方向インターフェースの典型：
    1. C→Prolog: C側でクエリ文字列を組み、Prologへ投げる（データベースAPIのように扱う）。
    2. Prolog→C: Prologの述語をC関数にマッピングして、システム情報やUIを取得／表示する。

- IRQアドバイザ例（設計の要点）
  - CがマシンのIRQ割り当てを取得し、各IRQをPrologの事実 irq(slot, device) としてassertする。
  - Prologはルール群で空きスロット探索や再割当候補の推論を行い、解決策を生成する。
  - Cは最終的なUI表示や実際の操作（スロット変更）手順の提示を担当する。
  - こうして「実機依存の情報収集」はC、「ルールの表現と探索」はPrologに分離され、保守性と拡張性が高まる。

（イメージコード）
```prolog
% Prolog: 呼び出し例
irq_advice(Device) :- get_irqs, find_solution(Device, Solution), write(Solution).
```
```c
/* C: Prologを呼ぶ例（擬似） */
lsCallStr(&t, "irq_advice('%s')", sDevice);
```

## 実践ポイント
- 適材適所で分離する：OS/ハード依存や高速IOはC、ルール／探索はPrologに。
- 連携パターン
  - データを「プッシュ」：Cで事実をassertしてPrologが推論する。
  - クエリを「投げる」：Cからクエリ文字列を組んで呼び出す（戻り値はPrologの解として受け取る）。
  - サービスを「提供」：PrologからCの関数を述語として呼び、副作用を実行する。
- 日本の現場での活用案：インストーラの自動診断、社内のルールエンジン（見積／評価）、組込み機器の設定アシストなどで短期導入が見込める。
- 開発のコツ：まずはテキスト・ファイルやモックで事実供給を実装してProlog側のルールを熟成させ、後で実機連携のC実装を差し替えると安全。

短くまとめると、「Prologで知識を表現し、Cで現実世界へつなぐ」アーキテクチャは、複雑な条件分岐や診断ロジックを扱う日本のプロジェクトに有効です。
