# A distributed queue in a single JSON file on object storage - オブジェクトストレージ上の単一JSONファイルで作る分散キュー
200ms の壁を味方にする──たった1ファイルで作る高可用・低レイテンシ分散キュー

## 要約
オブジェクトストレージ上の単一JSONファイル（queue.json）と比較・交換(CAS)を組み合わせ、グループコミットと無状態ブローカー、ハートビートで可用性と性能を両立した分散ジョブキューの設計を紹介します。

## この記事を読むべき理由
オブジェクトストレージ（S3/GCS等）は安価でスケーラブル。日本のクラウド運用でも「単純で運用しやすい」設計は魅力的です。サーバ数を増やすだけでなく、ストレージの特性を活かした設計で性能や信頼性を改善する実践例が学べます。

## 詳細解説
- 基本アイデア（Step 1）  
  すべてのジョブを小さなJSON配列にし、queue.jsonを丸ごと読み書きする。書き込みはCAS（オブジェクトの世代番号やETagを使った条件付きPUT）で行い、同時更新はリトライで解決。シンプルにFIFO・at-least-onceを満たせます。

- スループット問題とグループコミット（Step 2）  
  オブジェクト置換に時間がかかる（例: ~200ms）ため、単発書き込みだと最大書き込み頻度は $1/(0.2\ \mathrm{s}) = 5\ \mathrm{writes/s}$ 程度に制約されます。これを解決するために、書き込み中は要求をメモリでバッファしてまとめて1回で書く「グループコミット」を適用し、実効スループットを大幅に改善します。

- ブローカー導入で競合回避（Step 3）  
  多数クライアントが直接オブジェクトを更新するとCAS競合が発生するため、無状態のブローカーを置き、すべてのクライアントはブローカー経由でリクエストする。ブローカーがグループコミットを単独で実行するので、オブジェクトへの競合が消え、数百〜数千クライアントを効率的にさばけます。

- 高可用化と再割当（Step 4）  
  ブローカー障害対策として、現在のブローカーのアドレスをqueue.jsonに書き込んでおき、タイムアウト時は新しいブローカーを立ち上げて切替可能にする。ジョブ処理側は定期的にブローカーへハートビート（タイムスタンプ）を書き、一定時間ハートビートが途切れたジョブは次のワーカーが再取得して処理を継続する（＝失敗検知と再割当）。CASがあるので複数ブローカーが重なっても整合性は保たれます。

- トレードオフと期待効果  
  - メリット: 実装が単純で可観測性に優れる、オブジェクトストレージの耐久性・スケールを活かせる、tail latency改善。  
  - 注意点: queue.jsonのサイズはメモリに乗る範囲に限定すること（著者は <1 GiB を想定）、オブジェクトストレージのAPIレートや置換レイテンシに依存する設計であること。

## 実践ポイント
- queue.jsonは小さく保つ（メモリに収まる設計）。  
- オブジェクトのCASは If-Match / generation 等で実装する（GCS/S3の条件付きPUTを活用）。  
- グループコミット：書き込み中はリクエストをメモリでバッファして一括書き込み。  
- 無状態ブローカーを一つ置き、ブローカーが書き込みとリーダーシップを担当。  
- ブローカーのアドレスをqueue.jsonに記録し、フェイルオーバ時は新ブローカーを検出可能にする。  
- ワーカーは定期的にハートビートを書き、タイムアウトでジョブを再取得する。  
- 実運用ではオブジェクトストレージの書き込みレイテンシ／RPS制限を監視し、必要ならキューのシャーディングを検討する。

以上を踏まえると、既存のオブジェクトストレージの「弱点」だった低速置換やAPIレートを、バッファリングとブローカーでうまく吸収してしまう設計は、日本のクラウド運用でも即戦力になる考え方です。
