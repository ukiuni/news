# Cursorless: Voice coding at the speed of thought - 思考の速さでコードを書く「Cursorless」
声だけでコードを直感的に編集する——Cursorlessが切り開く次世代のコーディング体験

## 要約
Cursorlessは「構造を意識した音声コマンド」でコード編集を行うオープンなプロジェクトで、キーボード操作を補完・代替しながら高速な編集を目指す。公式ドキュメントとエディタ拡張で試せる。

## この記事を読むべき理由
- コード編集の方法が変わりつつある今、音声で“意味単位”（関数・引数など）を扱える利便性は生産性とアクセシビリティに直結する。  
- 日本のリモートワークや多様なワーカー環境で、入力負荷軽減や障害を持つ開発者支援につながる可能性が高い。

## 詳細解説
- 基本アイデア：普通の音声入力（文字起こし）とは違い、Cursorlessはソースコードの「構造」を対象にする。関数・クラス・引数・行といった単位を指定して操作（選択・削除・置換・ラップなど）できるため、誤認識で長い修正を繰り返す必要が少ない。  
- 操作イメージ：自然言語に近い短い命令で「関数 foo を選択」「引数 2 を削除」「選択範囲を if で囲む」などを実行できる（具体的なコマンドはドキュメント参照）。  
- エディタ統合：VS Code などのエディタ拡張と組み合わせて使えるため、既存ワークフローに導入しやすい。公式ドキュメントに設定やデモがまとまっている。  
- 音声認識との関係：Cursorless自体は「音声→コマンド」処理の上位レイヤーを提供するため、利用者は好みの音声認識エンジン（OS標準や専用ソフト）と組み合わせて利用する。  
- コミュニティとライセンス：オープンな開発でドキュメントや寄付ページが公開されており、改善やカスタムコマンドの共有が進んでいる点も採用の追い風。

## 日本市場との関連性
- 日本語で話しかける環境や日本語の音響特性は認識精度に影響するため、英語コマンドとの組合せやカスタムマッピングで安定化を図るのが現実的。  
- コードは英語の識別子が多いため、キー操作を声で補う形は日本の開発現場でも価値が高い（ペアプロやレビュー時のハンズフリー操作など）。  
- 企業のアクセシビリティ方針や多様性推進の施策と親和性がある。

## 実践ポイント
1. 公式ドキュメントを一読する（まずは操作概念を把握）。公式: https://www.cursorless.org/  
2. VS Code 拡張などエディタ統合を導入して、キーボード併用で少しずつ慣れる。  
3. 高品質なマイクを用意し、背景ノイズ対策を行う。  
4. よく使う操作をショートリスト化してから音声コマンドに慣れる（例：選択・削除・移動・ラップ）。  
5. 日本語環境では英語コマンドやカスタムマッピングを試し、認識誤差を減らす。  
6. チーム導入を考える場合は、短いトレーニングとテンプレート共有で学習コストを下げる。

まずはドキュメントのチュートリアルを動かしてみるのが早道。構造を意識した声操作に慣れれば、キーボード中心の編集では得られない「直感的な高速編集体験」が期待できる。
