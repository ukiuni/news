# Trying Out Claude Code Teams - Claude Code チームを試す
AIエージェント同士に開発を任せてみたら本当に動く？Claude Code Teamsで作るKafkaベースのPub/Subルーター実験レポ

## 要約
Claude Codeの「チーム」機能を使い、Kafkaバックエンド＋Admin UIを持つPub/Subルーターを段階的に開発。大規模一括より「小さく役割を限定したチーム」を順に回す方が高速で安定した、という実験結果。

## この記事を読むべき理由
AIエージェントによる複数並列ワークの現実的な限界と有効な運用ルール（キックオフ、役割分離、プラン優先）を具体例で理解できる。日本の開発現場でAIを導入する際の実践的な指針になる。

## 詳細解説
- 機能の違い：Claude Codeはサブエージェント（単一セッション内で報告）とチーム（個別セッション・共有タスクリスト・直接メッセージ可）を提供。本検証は後者を利用。  
- テストプロジェクト：HTTPで受けたイベントをKafkaでルーティングし、gRPCで購読者へ配信（リトライ、DLQ、フル可観測性）。複数サービス・ストレージ・運用要件があるため実戦的。  
- 運用パターン比較：
  - ワンビッグチーム（失敗）：フロント〜SREまで一括で走らせると役割混同、過剰な調整コスト、矛盾が多発。
  - 小チームフェーズ制（成功）：各フェーズを2〜3エージェントに限定。キックオフで共通コンテキストを与え、各エージェントには「担当ファイル」「禁止領域」「作成するタスク数」を明確に指定。  
- フェーズ例：
  - アーキテクト + デビルズアドボケイト：設計責任者と設計批判者で早期リスク摘出。
  - バックエンド分割：Admin API、Ingest、Deliveryで並列化。
  - フロント、SDET、パフォーマンス、レビューはそれぞれ専任チームで実施。  
- 収穫：プラン先行の厳格ルール（コード禁止モード、固定出力ファイル、最後は「READY FOR APPROVAL」）が品質とレビュー効率を大幅に向上させ。インフラ周り（Docker等）はエージェントが苦手で人手介入が必要。

## 実践ポイント
- 小さく分ける：2エージェント前後の狭い責務で回す。  
- キックオフ必須：共通コンテキスト＋ルールを最初に与える。  
- プラン優先：コード前に「承認可能なプラン」を必須化するテンプレートを用意する。  
- 出力を一箇所に：各エージェントに「編集対象ファイル」を固定して競合を防ぐ。  
- インフラは人の目を入れる：Dockerやプラットフォーム設定は最初から人間がレビューする工程を残す。  
- リスク管理：デビルズアドボケイト役を入れて設計リスクを早期に洗い出す。

この実験は「AIで全部任せる」ではなく「AIを小さく責任限定して効率化する」現実的な活用法を示す。日本のチーム構成やSRE文化にも適用しやすい手法なので、まずは小さなモジュールで試してみるのが有効。
