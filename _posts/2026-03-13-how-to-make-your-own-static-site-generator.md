# How to make your own static site generator - 自分だけの静的サイトジェネレータを作る方法
120msでビルドする秘密：シンプルに速く、拡張しやすい静的サイト生成の実践ガイド

## 要約
筆者は数年かけて自家製の静的サイトジェネレータを磨き、クリーンビルドで約120ms、差分ビルドで約50msを達成した。この記事はその設計上の判断と実装上の工夫を分かりやすくまとめたもの。

## この記事を読むべき理由
- 日本でもブログ、ドキュメント、プロダクトサイトの多くは静的構成で十分。高速で管理しやすい自前のジェネレータは運用コストと表示速度を両立する実務的な選択肢になる。  
- GitHub Pagesや社内ドキュメント生成にそのまま応用できる実践的テクニックが得られる。

## 詳細解説
主な処理パイプラインは単純：記事一覧取得 → 各記事のパース（AST化） → リント → 目次生成 → HTML生成 → RSS/インデックス生成。重要なポイントは以下。

- 記事一覧とメタ情報の取得  
  ファイル列挙だけでなく、「作成日時／更新日時」を正確に得るために git の履歴を一度に取得する方が速い（N+1 プロセスを避ける）。例：  
  ```bash
  # Shell
  git log --format='%aI' --name-status --no-merges --diff-filter=AMDR --reverse '*.md'
  ```
  大量処理では定期的な `git gc --aggressive --prune=now` で高速化する。

- パースとASTの利点  
  Markdown を AST に変換してから処理することで、構造的に目次抽出、リント、検索インデックス作成、ビルド時シンタックスハイライトなどが容易になる。テキストとコードを正確に分離できるのが強み。

- リント（ASTベース）  
  無効なリンク、言語指定のないコードブロック、アクセシビリティ（alt属性）不足、見出しレベルの飛び（h2→h4）など、構造を走査して検出する。開発ワークフローに組み込むと品質が保てる。

- 目次生成（TOC）  
  見出しを平坦な配列で収集し、深さを追跡しながらネストしたリストHTMLを線形スキャンで生成するのがシンプルで高速。ツリー構築は不要なことが多い。

- HTML生成と付随機能  
  ASTからHTMLを機械的に生成すると自由度が高い（クラス、ID、data属性、フットノート収集、ビルド時ハイライト等）。特殊文字エスケープやフットノートの後出しなどに注意。

- RSSとID管理  
  記事を安定したIDで管理するために UUIDv5（ブログ名前空間＋記事パスのハッシュ）を使う設計は再現性がある。

- 行番号の実装（CSSのみ）  
  コード行ごとに空の span を置き、疑似要素でカウンタを表示。`user-select: none`でコピー時に行番号を含めない設計が有効。

- 検索戦略（クライアントサイド）  
  小規模サイトならトライグラムや複雑な索引より、記事HTMLを必要時に並列フェッチしてインデックス化し、`indexOf()` 等で線形検索する方が実装も運用も楽。初回だけ転送コストが発生する工夫（必要時に遅延ロード）も重要。

- ドラフト運用  
  ローカルのブランチ運用でドラフト管理するというシンプルな運用はビルド系を複雑化させない。

- ライブリロード（SSE実装の例）  
  サーバー側でファイル監視→条件変数で通知→SSEでブラウザに伝える構成は、低CPUでリアルタイム更新が可能。簡易なEventSourceで十分。例（JS）:
  ```javascript
  // Javascript
  const es = new EventSource('/live-reload');
  es.onmessage = () => location.reload();
  ```

## 実践ポイント
- まずは最小のMakefileやスクリプトで開始し、機能は必要に応じて追加する（RSS、TOC、タグ、検索は後から追加可）。  
- メタ情報取得はgitコマンド1回でまとめて取得してプロセス数を減らす。  
- Markdownは既存のパーサでASTを取り、リントや目次生成はASTを直接操作する。  
- 小規模サイトならクライアント側線形検索＋遅延フェッチで十分高速。  
- ローカル執筆はブランチ運用、編集体験はSSEベースのライブリロードで改善。  
- ビルド時間の計測を CI/ローカルで取り入れ、まずは「短い」ビルドを目標に最適化する（N+1回の外部コマンド起動を避けること）。

以上を押さえれば、拡張しやすく速い自作静的サイトジェネレータが作れる。日本の個人ブロガーやOSSドキュメント、社内ワIKIにもすぐ応用できるアプローチ。
