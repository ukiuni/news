# The Missing GitHub Status Page - 欠けていたGitHubステータスページ
GitHubの「見えないダウンタイム」を可視化するミラーサイト — 本当の稼働率を自分の目で確認しよう

## 要約
GitHub公式が最近は集計された稼働率を公開しなくなったため、有志が過去のステータスフィードを再構築してコンポーネント別・分単位のダウンタイムを復元したミラーを作った。オープンソースで履歴検証やPR参加も可能。

## この記事を読むべき理由
日本の開発チームやSREにとって、GitHubの停止や遅延は開発・CI/CD・リリースに直結する問題であり、公式の「曖昧になった」稼働率だけで判断するとリスクが見落とされるため、詳細な履歴があることは実運用で役立つ。

## 詳細解説
- 背景: GitHubは以前はステータスページで集計稼働率を出していたが、近年それが止まっているため可視性が落ちた。ミラーはその「欠け」を埋める目的で作られた。
- データ復元の仕組み（要点）:
  - 公開されているステータスの Atom フィードのスナップショットを Flat Data として git に保存した履歴を再生して構築している。
  - 各インシデントのタイムラインを再構築し、日単位ではなく分単位でダウンタイムウィンドウをマージして稼働率を算出する（より精密な集計）。
  - インシデントテキストからサービスへのマッピングを行い、コンポーネント別の稼働率を出す。
  - タグ付けは GLiNER2 による推定を使うが、テキストに明確に裏付けられるラベルのみ採用している。
- 付帯情報: 2017〜2019などのアーカイブ履歴が公開され、プラットフォーム全体・個別サービス単位で過去90日などの稼働率確認が可能。全工程はオープンソースで、PR歓迎。

## 実践ポイント
- 障害切り分けに使う: 自分のCIやデプロイが失敗したとき、まずローカル要因かGitHub側か判断する材料になる。  
- SRE/運用レポートで活用: SLAや事後分析に分単位のダウンタイムデータを使うと精度が上がる。  
- 自社ミラーの参考に: 同様の手法（フィードのスナップショット保存→再生→分単位集計）は自社サービスの監査ログ復元にも応用可能。  
- 参加・検証: ソースが公開されているので、疑問点はレポジトリにPRやIssueを出して改善に貢献できる。
