# The PERFECT Code Review: How to Reduce Cognitive Load While Improving Quality - 完璧なコードレビュー：認知負荷を減らし品質を向上させる方法
魅了タイトル：レビュー嫌いが変わる。認知負荷を下げて品質を上げる「PERFECT」レビューの処方箋

## 要約
認知的負荷を減らしつつ本質に集中できるコードレビューの原則「PERFECT」を紹介。目的志向・境界ケース・信頼性・設計・証拠・明瞭さ・趣味の分離で、レビューを効率化し品質を向上させる実践ガイド。

## この記事を読むべき理由
日本の開発現場ではレビューが形式化したり、議論が長引いて納期に影響するケースが多い。PERFECTは「何を最優先するか」を明確にして無駄な議論を減らすため、アジャイルやメンテナンス重視のプロジェクト、金融・組み込みなど検証が重要な分野で即効性がある。

## 詳細解説
PERFECTは各文字が重要チェックポイントを表す覚えやすい枠組み。順に解説する。

- Purpose（目的）  
  - まず「この変更は何を解決するのか」を確認。チケットやPR説明と照合し、実装が目的を満たしているかを最初に判断する。機能を満たさないコードは見た目が良くても価値がない。

- Edge cases（境界・例外）  
  - 入力の端、null/型不整合、外部依存の不具合、想定外の状態（将来の変更で破綻し得る“到達不能”ケース）を探す。プロジェクト固有のチェックリストを持つと効果的。

- Reliability（信頼性）  
  - パフォーマンス、セキュリティ、リソース使用、外部APIの失敗耐性などを簡単に評価。商用・ミッションクリティカルな部分はここで重点的に見る。

- Form（設計・形式）  
  - 高凝集・低結合の原則でコード構造を評価。SOLIDなどは参考だが、チーム合意に基づいた設計基準を優先することで主観的議論を抑える。

- Evidence（証拠）  
  - テスト・CIが通っていること。自動化されたチェックはレビュー者の認知負荷を下げる。壊れたパイプラインは放置せず修復または削除する運用ルールを設ける。

- Clarity（明瞭さ）  
  - 意図が読み取れるか。命名、ファイル構造、コメントやドキュメントで「斜め読み」で理解できることを目標に。明確な合意がない限り、軽微なスタイルは著者の裁量に任せるのが現場の摩擦を減らす。

- Taste（好み）  
  - 個人的な好みはコメントに残しておくが、根拠がない限りマージの阻害要因にしない。良い意見はチームルールへ昇華させる。

認知負荷を下げるための設計上の工夫：
- レビュー対象を小さく（1PRは概ね数百行未満）し、1回のレビューで扱う責務を限定する。
- 自動チェックで機械的な項目を捨てる（リンタ、型チェック、簡易セキュリティスキャン）。
- レビュー合意（誰が見るか、何をコメントするか、SLA）を文書化して標準化する。

文化的・日本市場との関連：
- 日本では合意形成や対面での和を重視する傾向があり、曖昧なレビュー基準が心理的負担を強めがち。明文化されたルールは「意見対立」を減らし、レビュー速度と満足度を改善する。
- 外注や海外チームとの協業が増える中で「目的→境界→信頼性」の優先順は共通言語になりやすい。特に金融・医療・組み込みなどの規制・品質要件が厳しい業界で有効。

## 実践ポイント
- チームで「PERFECT」チェックリストを作り、PRテンプレートに組み込む。  
- PRは小さく（例：< 400 LOC）・短時間でレビュー（目安：30–60分以内）。  
- 必須項目として「目的」と「境界ケースの検証」をPR説明に書かせる。  
- 自己レビューを義務化し、セルフチェック項目を用意する（テスト、ドキュメント、セキュリティ確認）。  
- CI／自動検査は「レビュー前」に通す。壊れたCIは即修復か除去。  
- レビューSLA（レビュー応答時間）とレビュー役割を明確にする。  
- 定期的にレビュー文化を見直し、繰り返し発生するコメントはチームルールに昇格させる。

短い手順を導入するだけで、レビューの認知負荷は劇的に下がり、本質的な欠陥検出に集中できる。まずは「目的」と「境界」をPRテンプレートに必須化することから始めると効果が早く現れる。
