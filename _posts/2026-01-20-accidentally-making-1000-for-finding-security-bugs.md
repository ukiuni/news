# Accidentally making $1000 for finding Security Bugs as a Backend Developer - バックエンド開発者が偶然見つけた脆弱性で1000ドルを稼いだ話
「DevToolsを開いただけで判明した“全部丸見え”の落とし穴 — Supabase / JWT が招いた実録」

## 要約
バックエンド開発者がブラウザの開発者ツールでAPIの挙動を追ったところ、Supabaseの公開テーブルや不適切な認可実装（IDORや公開API）により大量のユーザーデータが第三者に見える状態になっていた。報告の結果、合計で約1000ドルのバグ報奨を受けたという実話。

## この記事を読むべき理由
近年、日本のスタートアップや個人開発でもSupabaseやFirebaseのようなBaaSが急速に普及している。短期間でプロダクトを出す流れの中で、今回のような設定ミスや認可の欠如は日本のサービスにも十分起こり得るため、初級者〜中堅エンジニアが知っておくべき教訓が含まれている。

## 詳細解説
- 構成と発端  
  フロントはReact、バックエンドはFastAPI、認証やデータはSupabaseを利用。開発者がSSE（サーバー送信イベント）やログの挙動を調べるためにDevToolsでHTTPリクエストを観察したのが出発点。

- 具体的な問題点（高レベル）  
  1. Supabaseの「REST API」と認証トークンを利用して、スキーマやテーブルに対する問い合わせが可能になっていた。  
  2. 複数のテーブルがRLS（Row Level Security）や適切なACLなしに公開されており、ユーザー情報・メッセージ・添付ファイルへのアクセスが可能だった。  
  3. OpenAPI（Swagger）仕様が外部に公開されており、内部向けのテストAPIや権限をバイパスするエンドポイントの存在が明らかになっていた。  
  4. IDOR（Insecure Direct Object Reference）：有効なトークンがあれば、UUIDや数値IDを指定して他ユーザーのデータを読み書きできてしまう実装ミスがあった。  
  5. オブジェクトストレージのファイルが公開URLで直接参照でき、テーブルを介してファイルURLが漏れると認証不要で閲覧できてしまう状態だった。

- 結果と対応  
  報告後、運営はテーブルを非公開にし、問題箇所を修正。報奨金は合計で約1000ドル。開発者は「バックエンド視点だとこの手の脆弱性は比較的見つけやすい」と振り返っている。

## 実践ポイント
すぐに実行できる防止策（開発者・運営向け）
- Supabase/Firebase等を使う際はRLSやセキュリティルールを必ず有効化する。デフォルトで公開になっていないか確認。  
- 認可は「認証されたユーザーか？」だけでなく「そのユーザーがそのリソースの所有者か？」をチェックする（オーナーチェック）。  
- 公開APIドキュメント（OpenAPI）や内部テスト用エンドポイントを本番環境でそのまま公開しない。ビルド時に除外するか認証を必須に。  
- オブジェクトストレージは署名付きURLや限定公開設定を使い、推測できる公開パスにしない。  
- トークン管理：長期的に有効なトークンを安易に配布しない。最小権限・短寿命トークンを推奨。  
- 監査ログと検出ルールを整備し、異常なAPI列挙や大量リクエストを検知する。  
- バグ報告窓口と報奨ポリシーを明確にしておく（対応の早さと透明性が重要）。  
- 開発者はProdでDevToolsを開いて挙動を見る習慣を持ち、セキュリティ観点でのAPI観察を定期的に行う。

日本の事業者へ一言
- APPI（個人情報保護法）や事業者責任の観点からも、こうした露出は重大リスク。開発スピード優先でも、最低限のRLSや認可チェックは設計段階で組み込もう。

以上が今回の出来事から学べるポイント。短期でサービスを出す裏には「見落としやすい設定ミス」が潜んでいる――という警告として受け止めてほしい。
